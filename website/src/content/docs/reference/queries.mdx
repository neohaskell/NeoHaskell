## The `Query` typeclass

```haskell
class Query query where
  canAccessImpl :: Maybe UserClaims -> Maybe QueryAuthError
  canViewImpl :: Maybe UserClaims -> query -> Maybe QueryAuthError
  maxResultsImpl :: Int
  maxResultsImpl = absoluteMaxLimit  -- default: 1000
```

- **`canAccessImpl`** — checked before fetching data. "Can this user access this query type?"
- **`canViewImpl`** — checked after fetching data. "Can this user view this specific instance?"
- **`maxResultsImpl`** — maximum page size for this query type.

You typically don't write this instance by hand. The `deriveQuery` helper generates it from top-level `canAccess` and `canView` functions in your query module.

## The `QueryOf` typeclass

```haskell
class (Entity entity, Query query) => QueryOf entity query where
  queryId :: entity -> Uuid
  combine :: entity -> Maybe query -> QueryAction query
```

One instance per (entity, query) pair. Implements:

- **`queryId`** — what's the key of the query instance this entity contributes to?
- **`combine`** — given the current entity state and the existing query (if any), what's the new query state?

## `QueryAction`

```haskell
data QueryAction query
  = Update query  -- store or replace
  | Delete        -- remove
  | NoOp          -- unchanged
```

What `combine` returns. Three possibilities:

- **`Update newQuery`** — store this as the new value.
- **`Delete`** — remove the query instance.
- **`NoOp`** — take no action.

## The `deriveQuery` helper

```haskell
{-# LANGUAGE TemplateHaskell #-}

import Service.Query.TH (deriveQuery)

deriveQuery ''MyQuery [''Entity1, ''Entity2]
```

`deriveQuery` generates:

- The `Query` instance (wiring `canAccess` and `canView` from your top-level functions to the typeclass methods).
- The `NameOf` type instance (derived from the type name, converted to kebab-case).
- The `EntitiesOf` type instance (telling the query subscriber which entities to listen to).
- The `KnownHash` instance for the query name.

You still write the `QueryOf` instances yourself — one for each entity in the list.

## Query auth helpers

```haskell
import Service.Query.Auth
```

| Function | Use case |
|---|---|
| `publicAccess` | Anyone can access (no auth required). |
| `publicView` | Anyone can view any instance. |
| `authenticatedAccess` | Any logged-in user can access. |
| `adminOnlyAccess` | Requires admin claim. |
| `ownerOnly (.ownerField)` | Only the owner (matched via a field accessor) can view. |

These are functions you plug into `canAccess` and `canView`:

```haskell
canAccess :: Maybe UserClaims -> Maybe QueryAuthError
canAccess = authenticatedAccess

canView :: Maybe UserClaims -> MyQuery -> Maybe QueryAuthError
canView = ownerOnly (.ownerId)
```

Return `Nothing` to allow, `Just error` to deny. The helpers handle common cases; for custom logic, write your own function with the same signature.

## Type family

```haskell
type family EntitiesOf (query :: Type) :: [Type]
```

Declares the list of entity types a query depends on. Generated by `deriveQuery` from its second argument.

## A complete example

```haskell
{-# LANGUAGE TemplateHaskell #-}

module MyApp.User.Queries.UserDashboard (
  UserDashboard (..),
  canAccess,
  canView,
) where

import Array qualified
import Core
import Json qualified
import Maybe qualified
import Service.Query.Auth (QueryAuthError, UserClaims, authenticatedAccess, ownerOnly)
import Service.Query.TH (deriveQuery)
import MyApp.User.Core (User (..))
import MyApp.Post.Core (Post (..))


data UserDashboard = UserDashboard
  { userId :: Uuid
  , userName :: Text
  , email :: Text
  , postCount :: Int
  , mostRecentPostTitle :: Maybe Text
  }
  deriving (Eq, Show, Generic)


instance Json.FromJSON UserDashboard
instance Json.ToJSON UserDashboard
instance ToSchema UserDashboard


canAccess :: Maybe UserClaims -> Maybe QueryAuthError
canAccess = authenticatedAccess


canView :: Maybe UserClaims -> UserDashboard -> Maybe QueryAuthError
canView = ownerOnly (.userId |> Uuid.toText)


deriveQuery ''UserDashboard [''User, ''Post]


instance QueryOf User UserDashboard where
  queryId user = user.userId

  combine user existing =
    Update UserDashboard
      { userId = user.userId
      , userName = user.name
      , email = user.email
      , postCount = existing |> Maybe.map (.postCount) |> Maybe.withDefault 0
      , mostRecentPostTitle =
          existing |> Maybe.andThen (.mostRecentPostTitle)
      }


instance QueryOf Post UserDashboard where
  queryId post = post.authorId

  combine post existing = case existing of
    Nothing ->
      -- User dashboard hasn't been created yet; skip this event.
      NoOp
    Just dashboard ->
      Update dashboard
        { postCount = dashboard.postCount + 1
        , mostRecentPostTitle = Just post.title
        }
```

## HTTP endpoints

Registering a query with `Application.withQuery @MyQuery` generates an HTTP endpoint at:

```text
GET /queries/{query-name}/{id}
```

Where `{query-name}` is the kebab-case form of the query type name (e.g., `UserDashboard` → `user-dashboard`) and `{id}` is the query instance ID (what `queryId` returns).

Pagination is supported via query parameters:

```text
GET /queries/user-dashboard?limit=20&offset=40
```

## Registering queries with the application

```haskell
app :: Application
app =
  Application.new
    |> ...
    |> Application.withQuery @UserDashboard
```

`withQuery` does three things:

1. Registers the query definition with the query registry.
2. Creates the query object store (in-memory by default; use `withQueryObjectStore` to override).
3. Adds an HTTP endpoint.

## Rebuilding queries

On application startup, all queries are rebuilt from the complete event history. If you change a query's shape or logic, the old read models are discarded and replaced with fresh ones. This happens automatically.

For large event stores this rebuild can take a noticeable amount of time. In production, you typically snapshot query state and only rebuild when detecting a version change. See the [PostgreSQL event store guide](/guides/postgresql-event-store/) for details.
