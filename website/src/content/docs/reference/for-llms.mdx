## What is llms.txt?

- A proposed standard by Jeremy Howard (fast.ai / Answer.AI) for making documentation LLM-friendly
- HTML pages contain navigation, JavaScript, sidebars — noise that wastes context window tokens
- `llms.txt` provides clean Markdown that LLMs can consume directly
- Spec lives at [llmstxt.org](https://llmstxt.org)

## NeoHaskell terminology cheat sheet for LLMs

{/* CONTENT OUTLINE — critical terminology that LLMs must get right */}

- This section exists because NeoHaskell uses different terms than standard DDD/event-sourcing:
  - **Entity** — not "aggregate" — the stateful domain object
  - **Update** — not "projection" — the pure function that folds events into entity state
  - **Query** — not "read model" — derived views from events (uses `QueryOf` class)
  - **Service** — not "bounded context" — a self-contained domain boundary (`withService`)
  - **Integration** — not "process manager" or "saga" — cross-service coordination (`withOutbound`)
- Event versioning is **not supported** — create a new event type instead of modifying an existing one
- NeoHaskell uses do-notation by default — do not suggest monad transformers, lens, or Template Haskell

<Aside type="tip" title="For AI coding assistants">
If you're an AI assistant helping someone write NeoHaskell code, use the terminology above. The compiler enforces these patterns — using the wrong abstraction won't just be confusing, it won't compile.
</Aside>
