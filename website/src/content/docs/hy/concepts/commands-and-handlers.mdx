Հրամանները այն վայրն են, որտեղ արտաքին աշխարհը հանդիպում է ձեր տիրույթի հետ։ Օգտատուն սեղմում է կոճակ, ինտեգրումը հրաձգում է, հրամանային գիծը գործիք է կատարում — և ինչ-որ տեղ ձեր `decide` ֆունկցիայի մոտ հայտնվում է հրամանի օբյեկտ։ Այս էջը անցկացնում է ՆեոՀասկել հրամանի ամբողջ կյանքի ցիկլը, որպեսզի դուք հասկանաք ճշգրտորեն, թե ինչ է տեղի ունենում ամեն քայլում և ինչու։

## Չորս շարժվող մասեր

Ուղեցույցից առաջ, սա պատմության մեջ հերոսներն են.

1. **Հրաման** — տիպային ռեկորդ, որը նկարագրում է կանչողի ցանկությունը։
2. **`decide` ֆունկցիա** — մաքուր ֆունկցիա, որը հրամանը փոխարկում է `Decision`-ի (ընդունում կամ մերժում)։
3. **Հրամանի կատարող** — կատարման ապարատ, որը անվտանգ կերպով կատարում է `decide` և պահում է արդյունքային իրադարձությունները։
4. **Իրադարձության պահոց** — ավելացված միայն տվյալների պահոց, որտեղ իրադարձությունները ապրում են ընդմիշտ։

Մնացած ամեն ինչ (ՀՏՏՊ տրանսպորտ, ՋՍՕՆ մեկնաբան, հաստատում, կրկնություն) խողովակաշար է։ Վերը նշված չորս հասկացություններն են կարևորը։

## Լրիվ կյանքի ցիկլ

Եկեք հետևենք մի միակ խնդրանքի վերջից սկիզբ։ Պատկերացրեք, որ կանչողը կատարում է այս ՀՏՏՊ խնդրանքը.

```http
POST /commands/Deposit HTTP/1.1
Content-Type: application/json

{ "accountId": "abc-123", "amount": 50 }
```

### Քայլ 1 — Տրանսպորտ վերծանում է խնդրանք

ՀՏՏՊ տրանսպորտը ստանում է խնդրանքը, մեկնաբանում է ՋՍՕՆ մարմինը `Deposit` տիպի և կառուցում է `RequestContext` (պարունակում է հաստատում, օգտատիրոջ հայտարարություն, հետքի իդ, և այլն)։

Սա մաքուր խողովակաշար է։ Դուք չեք գրել այն։ `command ''Deposit` տողը ձեր հրամանի ֆայլի վերևում այս կոդը գեներացրել է ձեզ համար։

### Քայլ 2 — Կատարող է վերցնում հաշիվը

Կատարողը հիմա պետք է հաշվի ներկա վիճակը։ Այն կանչում է `EntityFetcher`-ը, որը.

1. Որոնում է սնիպետ պահոցը այս էնտիտի իդ-ի համար։ Եթե պահված սնիպետ գոյություն ունի, այն օգտագործում է այն որպես սկզբնական կետ։
2. Կարդում է այն իրադարձությունները, որոնք եկել են սնիպետի ընդունումից հետո։
3. Այդ իրադարձությունները միավորում է սնիպետի հետ էնտիտի `update` ֆունկցիայի օգնությամբ։
4. Վերադարձնում է հաշվի ներկա `Account` վիճակը (կամ `Nothing` եթե այս իդ-ի համար դեռ ոչ մի իրադարձություն չկա)։

Սա է ձևը, թե ինչպես `decide`-ը ստանում է էնտիտի թարմ և ճիշտ տեսք առանց ձեր կողմից տվյալների բազայի հետ աշխատելու։

### Քայլ 3 — Decide է կանչվում

Ձեր մաքուր `decide` ֆունկցիան է կատարվում.

```haskell
decide :: Deposit -> Maybe Account -> RequestContext -> Decision AccountEvent
decide cmd account _ctx = case account of
  Nothing ->
    Decider.reject "Հաշիվ գոյություն չունի"
  Just _ ->
    if cmd.amount <= 0
      then Decider.reject "Վճարման գումարը պետք է դրական լինի"
      else
        Decider.acceptExisting
          [ MoneyDeposited
              { entityId = cmd.accountId
              , amount = cmd.amount
              }
          ]
```

Այն ստանում է հրամանը, էնտիտի վիճակը և խնդրանքի համատեքստը։ Այն վերադարձնում է `Decision`։ Դա այն է։ Այս ֆունկցիան մաքուր է — ոչ մի Ի/Ե, ոչ մի տվյալների բազայի հասանելիություն, ոչ մի կողմնակի ազդեցություն։ Դուք
