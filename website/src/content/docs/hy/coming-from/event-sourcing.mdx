## Դուք արդեն գիտեք դժվար մասերը

- Դուք գիտեք, թե ինչու է իրադարձությունների աղբյուրը հզոր. աուդիտի հետքեր, ժամանակի ճամփորդություն, վերախաղարկ, CQRS
- Դուք նաև զգացել եք շփոթությունը. սխեմայի գրանցամատյան, սպառող խմբի կառավարում, իրադարձության տարբերակ, թարմացման վերակառուցում
- NeoHaskell-ի պնդումն է, որ մի քանի այդ շփոթությունները պատահական են. առաջացել են ES-ը ենթակառուցվածքային խնդիր համարելուց, այլ ոչ լեզվային խնդիր
- Մի քանիսը այդպես են։ Մի քանիսը ոչ։ Այս էջը ճշմարտասեր է այն մասին, թե որն է որը։

<Aside type="caution" title="Նախաթողարկում">
NeoHaskell-ը նախաթողարկում է (v0.1-alpha)։ Այս էջում ներկայացված պնդումները հիմնված են դիզայնի հատկությունների վրա, այլ ոչ արտադրական դեպքերի ուսումնասիրությունների վրա։ Չկան Fortune 500 տեղադրումներ։ Չկան խոշոր մասշտաբային մարտական պատմություններ։ Եթե ձեր գնահատումը պահանջում է արտադրական ապացույց, սպասեք։
</Aside>

## Ինչ է իրապես լավ

Այս բարելավումները իրական են, այլ ոչ գովազդային պնդումներ.

- **Ոչ մի սխեմայի գրանցամատյան**: Կաֆկա + Avro/Protobuf-ում սխեմայի գրանցամատյանը առանձին համակարգ է, որը դուք պահպանում եք, տարբերակում եք և աշխատացնում եք։ Նեո Հասկելում տիպերը ՍԽԵՄԱՆ են։ `data AccountEvent = MoneyDeposited { amount :: Int } | AccountClosed` սխեման է։ Կոմպիլյատորը ստուգում է այն։ Ոչ մի առանձին գրանցամատյան, ոչ մի սխեմայի շեղում, ոչ մի "գրանցամատյանը խափանված է"։

- **Ոչ մի սպառող խումբ**: Կաֆկայում թարմացումները պահանջում են սպառող խմբի կառավարում. օֆսեթի հետևում, վերահավաք, ուշացումը հետևել։ Նեո Հասկելում թարմացումները մաքուր ֆունկցիաներ են. `fold :: [AccountEvent] -> AccountState`։ Ոչ մի սպառող խումբ, ոչ մի օֆսեթի կառավարում, ոչ մի վերահավաք։ Թարմացումը պարզապես ֆունկցիա է։

- **Իրադարձության տարբերակ ալգեբրական տիպերի միջոցով**: նոր իրադարձության տիպ ավելացնելը նշանակում է գումարային տիպի կառուցիչ ավելացնել։ Կոմպիլյատորը հետո ստիպում է ձեզ մշակել նոր դեպքը ամեն տեղ, որտեղ այն
