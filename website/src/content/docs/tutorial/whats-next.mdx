## Common next questions

### "Where do I put validation?"

In the `decide` function of the relevant command. That is the only layer with the context to validate correctly — it sees the command, the current entity, and the request context. Validation libraries that live in a different layer are validating in the wrong place.

### "What about authorization?"

Two places. `canAccess` on queries decides "can this user access this query type at all?". `canView` decides "can this user see this specific instance of a query?". For commands, you check the `RequestContext` inside `decide`. A full write-up is in the [Auth and authorization guide](/guides/error-handling/) (and the `Service.Auth` module has all the primitives you need).

### "How do I do long-running background work?"

Inbound integrations with a timer. `Integration.Timer.every` lets you fire a command on a schedule. Outbound integrations let you react to events by calling external services. Both are typed and compile-checked.

### "What about deployment?"

NeoHaskell compiles to a single static binary via GHC. You can drop it on any Linux box, in any container, or behind a reverse proxy. See [Guides → Deployment](/guides/deployment/) for specifics.

### "How do I query across entities?"

A query can list multiple entity types in its `deriveQuery` call. You write a `QueryOf` instance for each entity that contributes. That's how a read model like "customer dashboard" can combine data from user entities, order entities, and subscription entities into one document.

### "What about schema evolution — what if my event shape has to change?"

Create a new event type. Keep the old one. Events are immutable facts — you don't rewrite history. The [Schema evolution page](/concepts/schema-evolution/) has the full pattern, but the short version is: new events alongside old events, and your `update` function handles both.

## A closing thought

You started this tutorial not knowing how to write NeoHaskell. A few hours later, you have:

- A working event-sourced banking service.
- A mental model for commands, events, entities, queries, and integrations.
- Enough muscle memory to start modeling your own domain.
- The confidence that the compiler will tell you when you make a mistake.

That is a substantial amount of progress for one afternoon. The rest — the ecosystem, the tooling, the advanced patterns — comes as you need it. You don't have to know everything. You just have to know enough to start.
