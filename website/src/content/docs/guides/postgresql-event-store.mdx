The in-memory event store is perfect for development and tests but loses everything on restart. For production, you use PostgreSQL. This guide covers how to set it up, how to size it, how to back it up, and what to watch in production.

## Why PostgreSQL

NeoHaskell's event store is PostgreSQL-backed by design. The reasoning:

- **PostgreSQL is everywhere.** Your ops team already knows it.
- **ACID at the stream level.** Optimistic concurrency on append is atomic.
- **Great tooling.** Backup, replication, monitoring, disaster recovery — all well-established.
- **Scales further than you think.** With proper schema design and partitioning, a single Postgres instance handles tens of thousands of events per second.
- **Flexible.** If you outgrow Postgres, the event store interface is pluggable — you can swap in a custom backend.

## Switching from InMemory to Postgres

You already saw this in [Tutorial Part 6](/tutorial/06-audit-everything/). The switch is a one-line change:

```haskell
-- Before
Application.withEventStore @() (\_ -> InMemory.new)

-- After
Application.withEventStore makePostgresConfig


makePostgresConfig :: NeoBankConfig -> PostgresEventStore
makePostgresConfig config =
  PostgresEventStore
    { user = config.dbUser
    , password = config.dbPassword
    , host = config.dbHost
    , databaseName = config.dbName
    , port = config.dbPort
    }
```

On startup, the event store:

1. Connects to PostgreSQL.
2. Checks that the expected schema exists, creating it if not.
3. Starts serving reads and writes.

The schema it creates is a few tables — one for events, one for stream metadata, a couple for snapshots. You don't manage them by hand; the store manages its own schema.

## Configuration

Use the `Config` system to pull credentials from environment variables:

```haskell
{-# LANGUAGE TemplateHaskell #-}

module MyApp.Config (NeoBankConfig (..), HasNeoBankConfig) where

import Config (defineConfig)
import Config qualified
import Core

defineConfig
  "NeoBankConfig"
  [ Config.field @Text "dbHost"
      |> Config.doc "PostgreSQL host"
      |> Config.defaultsTo ("localhost" :: Text)
      |> Config.envVar "DB_HOST"
  , Config.field @Int "dbPort"
      |> Config.doc "PostgreSQL port"
      |> Config.defaultsTo (5432 :: Int)
      |> Config.envVar "DB_PORT"
  , Config.field @Text "dbUser"
      |> Config.doc "PostgreSQL username"
      |> Config.defaultsTo ("neohaskell" :: Text)
      |> Config.envVar "DB_USER"
  , Config.field @Text "dbPassword"
      |> Config.doc "PostgreSQL password"
      |> Config.defaultsTo ("neohaskell" :: Text)
      |> Config.envVar "DB_PASSWORD"
      |> Config.secret
  , Config.field @Text "dbName"
      |> Config.doc "PostgreSQL database name"
      |> Config.defaultsTo ("neohaskell" :: Text)
      |> Config.envVar "DB_NAME"
  ]
```

`Config.secret` flags the password as sensitive, which means it won't be printed by `--help` or included in config dumps.

In production, pass the credentials via environment variables:

```bash
DB_HOST=postgres.prod.internal \
DB_PORT=5432 \
DB_USER=neobank_app \
DB_PASSWORD=$(vault read -field=password kv/neobank/db) \
DB_NAME=neobank \
./neobank
```

## Sizing a production instance

Rules of thumb for a single PostgreSQL instance:

| Load | Instance size | Notes |
|---|---|---|
| < 1k events/sec | db.t3.medium equivalent, 100 GB SSD | Comfortable headroom |
| 1k–5k events/sec | db.m6i.large / 4 vCPU, 16 GB RAM | Watch WAL generation |
| 5k–20k events/sec | db.m6i.2xlarge / 8 vCPU, 32 GB RAM | Dedicate IOPS |
| 20k+ events/sec | Tune carefully; consider partitioning | Usually need custom setup |

The bottlenecks you'll hit, in order:

1. **Write IOPS.** Events are small but append-heavy. Provision enough IOPS for your peak write rate.
2. **WAL bandwidth.** Every append writes to WAL. Replication lag becomes noticeable before throughput does.
3. **Memory for indexes.** Queries against the event store need the main index in RAM for good performance.

For most applications, you never hit these limits. A comfortable default for a production system with moderate traffic is a 4 vCPU / 16 GB RAM instance with 500 GB of SSD. You can monitor and scale up from there.

## Connection pooling

NeoHaskell uses `hasql-pool` internally, with **three** pools per app, each at an explicit, deploy-tier-safe default:

| Pool | Default size | What flows through it |
|---|---|---|
| EventStore | `6` | Command appends, entity fetches, query catch-up reads |
| QueryObjectStore | `4` | Read-model writes during rebuild, read-model reads |
| FileUpload | `6` | File metadata writes (shares the EventStore pool size) |

On top of the pools, the app keeps a persistent `LISTEN/NOTIFY` listener pair (2 connections) and opens **one connection per active stream subscription**. So the steady-state demand is `6 + 4 + 6 + 2 + 1 (init) = 19` connections, plus one for each active stream subscription.

The EventStore pool size is tunable from the environment with `DB_POOL_SIZE` (default `6`), which also sizes the FileUpload pool since it shares that config. Raise it on a larger Postgres tier; the EventStore pool is the one to grow first.

```haskell
PostgresEventStore
  { ... connection fields ...
  , poolSize = 12  -- omit to use the B1ms-safe default of 6
  }
```

Postgres itself has a `max_connections` setting. Make sure it's higher than the sum of every pool plus the listener pair, per-stream subscriptions, and headroom for admin connections and background workers. On a small tier this budget is tight — see the [Flexible Server connection budget](/guides/azure-postgresql-flexible-server/#the-connection-budget-on-b1ms) for the full arithmetic on Azure `B1ms`.

## Running on Azure Flexible Server

If your managed Postgres is **Azure Database for PostgreSQL — Flexible Server** (our standard cloud target), two rules carry the deployment:

- **Connect on the direct port `5432`, never the PgBouncer pooler on `6432`.** PgBouncer's transaction mode silently breaks `LISTEN/NOTIFY`, so live projections and integrations stop firing with no error. On the Burstable `B1ms` tier there's no pooler, so `5432` is automatic; on larger tiers, keep `DB_PORT=5432`.
- **Optionally harden TLS** with `DB_SSL_MODE=require` (forbid silent plaintext downgrade) or `verify-full` with a root-CA bundle (`DB_SSL_ROOT_CERT`). It's off by default, so localhost and dev are unaffected, and it applies uniformly to all three pools and the `LISTEN/NOTIFY` connections.

The full golden path — connection budget, scale-to-zero pairing with Container Apps, and storage cost — lives in the dedicated **[Azure PostgreSQL Flexible Server](/guides/azure-postgresql-flexible-server/)** guide.

## Backup and recovery

The event log is the source of truth. If you lose it, you lose your application's memory. Treat backups accordingly.

### Continuous backup (recommended)

Use PostgreSQL's continuous archiving (PITR) so you can restore to any point in time. On AWS RDS, this is "automated backups with point-in-time recovery." On self-hosted Postgres, use `pg_basebackup` + WAL archiving to S3 (or equivalent).

### Nightly snapshots

In addition to continuous backup, take a full snapshot every night and store it in a separate region. A ransomware attack or a bad migration can wipe continuous backups if they live next to the live database.

### Verify restores

A backup you haven't tested restoring is not a backup. Once a month, restore your latest backup to a scratch environment and verify that NeoHaskell can start and serve queries against it. Automate this if possible.

## Read replicas

For read-heavy workloads, you can run PostgreSQL read replicas and point your projection rebuild or query subscriber at them. This keeps the primary free for event appends.

<Aside type="caution" title="Replicas have lag">
Read replicas are eventually consistent with the primary. If your projections read from a replica, they can fall behind. This is usually fine — the projections themselves are eventually consistent with events — but make sure your SLOs account for the added lag.
</Aside>

## Partitioning

If your event volume grows large enough that a single `events` table becomes a problem (typically > 100 million rows), you can partition the table by stream or by date. PostgreSQL native partitioning works well for this.

The NeoHaskell event store does not partition automatically — you'd do this as a custom operational step. The internal layout of the events table is documented; reach out on Discord if you need help setting this up.

## Monitoring

Watch these metrics on your production event store:

- **Append latency** — p50, p95, p99. Watch for regressions.
- **Events per second** — to catch traffic spikes before they hit capacity.
- **WAL generation rate** — helps you plan replication bandwidth.
- **Connection pool utilization** — if you see 100% utilization, bump the pool size or investigate slow queries.
- **Replication lag** (if using replicas) — keep under 1s for read-your-writes safety.
- **Disk usage** — events accumulate. Plan for growth.
- **Cache hit ratio** — PostgreSQL's internal stats. A healthy database has > 99% cache hit on indexes.

NeoHaskell exposes structured logs for each command. Feed them into your observability stack (DataDog, Honeycomb, Loki, whatever) and set alerts on error rate, p99 latency, and append failures.

## Schema evolution at the database level

The event store manages its own schema. Upgrades to NeoHaskell may include schema migrations — they run on startup and are idempotent. In a zero-downtime deployment, you'd:

1. Deploy the new version to a canary instance and let it run the migration.
2. Verify the migration succeeded.
3. Roll the rest of the fleet once the schema is confirmed good.

Migrations in the NeoHaskell event store are designed to be backwards-compatible: a new version reads old events correctly without needing to rewrite them.

## Common production issues

### High WAL generation

**Symptom:** Replication is lagging or backup storage is growing fast.

**Cause:** Either many commands are being processed (legitimate) or something is causing spurious writes (bug).

**Fix:** Check command rates via your observability stack. If writes are legitimate, you may need a bigger instance or faster disks. If they're not, investigate what's generating the writes.

### Slow entity reconstruction

**Symptom:** Commands that target a single entity take longer than expected.

**Cause:** Long event streams with no recent snapshot cache. Every command reconstructs the entity from the full stream.

**Fix:** NeoHaskell takes snapshots automatically. If reconstruction is still slow, check that snapshot caching is enabled in your config and that the snapshot cache is not full.

### Connection pool exhausted

**Symptom:** Latency spikes; logs show "failed to acquire connection."

**Cause:** Too many concurrent commands for your pool size.

**Fix:** Increase pool size, ensure `max_connections` on Postgres is higher than the sum of all pool sizes, investigate whether commands are blocking longer than expected.

### Replica lag growing

**Symptom:** Projections reading from a read replica are out of date.

**Cause:** Replica hardware is slower than primary, or network bandwidth to replica is saturated.

**Fix:** Point projections at the primary temporarily, fix the replica, then switch back.
