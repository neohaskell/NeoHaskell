A NeoHaskell application compiles to a single static binary. Deploying one is mechanically simple — you build, you copy, you run. This guide covers the practical concerns: how to structure your build pipeline, how to package the binary, how to manage configuration across environments, and what to watch during rollouts.

## The short version

1. Build your binary inside the Nix dev shell with `cabal build`.
2. Copy the binary to a container image or a target host.
3. Provide configuration via environment variables.
4. Run the binary as a long-running process.
5. Watch the `/health` endpoint for readiness and liveness.

That's it. Everything else is refinement.

## Build pipeline

### Building locally

Inside the Nix dev shell:

```bash
nix develop
cabal build
```

The binary ends up somewhere under `dist-newstyle/build/`. The exact path depends on your architecture and version. To find it:

```bash
cabal list-bin exe:your-app-name
```

Copy that path to a more convenient location:

```bash
cp "$(cabal list-bin exe:your-app-name)" ./dist/your-app
```

### Building in CI

The cleanest CI approach is to use `nix develop` inside your CI runner. On GitHub Actions with Determinate Nix:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: DeterminateSystems/determinate-nix-action@v17
      - uses: cachix/cachix-action@v17
        with:
          name: neohaskell
      - run: nix develop --command cabal build
      - run: nix develop --command cabal test
      - name: Extract binary
        run: |
          mkdir -p dist
          cp "$(nix develop --command cabal list-bin exe:your-app)" dist/your-app
      - uses: actions/upload-artifact@v4
        with:
          name: binary
          path: dist/your-app
```

The Cachix action reuses the NeoHaskell binary cache, which makes `nix develop` finish in a minute or two instead of half an hour.

### Reproducible builds with Nix

For maximum reproducibility, build via `nix build` instead of `cabal build`. This pins every dependency to exact versions from the flake lockfile. It's slower on first run but guarantees bit-for-bit reproducibility across machines.

```bash
nix build .#your-app
./result/bin/your-app --version
```

## Containerization

### The simple approach: a multi-stage Dockerfile

```dockerfile
# syntax=docker/dockerfile:1.6

FROM nixos/nix:latest AS builder
WORKDIR /build
COPY . .
RUN nix --experimental-features "nix-command flakes" \
    develop --command cabal build
RUN mkdir -p /out && \
    cp "$(nix --experimental-features "nix-command flakes" \
           develop --command cabal list-bin exe:your-app)" /out/your-app

FROM debian:bookworm-slim
RUN apt-get update && apt-get install -y --no-install-recommends \
      ca-certificates \
      libgmp10 \
      libnuma1 \
      && rm -rf /var/lib/apt/lists/*
COPY --from=builder /out/your-app /usr/local/bin/your-app
EXPOSE 8080
ENTRYPOINT ["/usr/local/bin/your-app"]
```

This produces an image around 100 MB — most of it is the debian base + a handful of shared libraries needed by the GHC runtime.

### The smaller approach: static linking

For the smallest possible image, build with static linking:

```bash
cabal build --ghc-options="-static -optl-static -optl-pthread"
```

You can then use a `scratch` or `distroless` base image:

```dockerfile
FROM scratch
COPY --from=builder /out/your-app /your-app
EXPOSE 8080
ENTRYPOINT ["/your-app"]
```

Static linking has some GHC quirks (issues around `libc` and name resolution), so test carefully in your target environment before committing to it.

### The Nix-native approach

If you're comfortable with Nix, build the container image with `nix build .#container` and pull the result into your registry without ever touching Docker:

```nix
# flake.nix
outputs = { self, nixpkgs, ... }: {
  packages.x86_64-linux.container = pkgs.dockerTools.buildImage {
    name = "your-app";
    tag = "latest";
    config = {
      Cmd = [ "${self.packages.x86_64-linux.default}/bin/your-app" ];
      ExposedPorts."8080/tcp" = {};
    };
  };
};
```

This produces a minimal image with exactly the runtime dependencies your app needs. It's also fully reproducible — same inputs, same image hash.

## Running the binary

NeoHaskell applications read configuration from environment variables (via the `Config` system). You start them like any other long-running process:

```bash
DB_HOST=postgres.internal \
DB_PORT=5432 \
DB_USER=myapp \
DB_PASSWORD=$(vault read -field=password secrets/myapp/db) \
DB_NAME=myapp \
HTTP_PORT=8080 \
LOG_LEVEL=info \
./your-app
```

The app prints structured logs to stdout. In a container, these are picked up automatically by your log collector.

## Health checks

NeoHaskell exposes a health endpoint at `/health` by default:

```bash
curl http://localhost:8080/health
# {"status":"ok"}
```

For Kubernetes:

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
```

The health check verifies the HTTP server is responding. If you want to include "can I reach the database?" in the check, customize it with `withHealthCheck`:

```haskell
Application.new
  |> Application.withHealthCheck @() (\_ ->
       Web.HealthCheckConfig { healthPath = "health" })
```

For more sophisticated health checks (e.g., "can I reach the database *and* the message broker"), you'd customize the path and the handler. The base health check is fine for most services.

## Configuration management

Follow the [Twelve-Factor App](https://12factor.net/config) principle: configuration lives in environment variables, not in the binary.

The `Config` system makes this ergonomic — every field you declare with `defineConfig` becomes an environment variable by default. For secrets (database passwords, API keys, JWT signing keys), use `Config.secret` and load them from a real secrets manager (Vault, AWS Secrets Manager, Kubernetes Secret).

For local development, the template supports `.env` files:

```env
DB_HOST=localhost
DB_PORT=5432
DB_USER=neohaskell
DB_PASSWORD=neohaskell
DB_NAME=neohaskell
```

`Application.run` automatically loads `.env` if present, so you can just `cabal run` and have the right config.

## Rollouts

Because NeoHaskell apps are stateless (state lives in the event store), rolling deployments work the same as any other web service.

### Blue/green or canary

1. Deploy the new version alongside the old one.
2. Route a small percentage of traffic to the new version.
3. Watch error rates, latency, and custom metrics.
4. Gradually shift traffic.
5. Retire the old version.

The event store handles schema migrations idempotently, so running two versions against the same event store at the same time is safe as long as both versions are ABI-compatible with the schema. (Usually they are; NeoHaskell never makes breaking schema changes within a minor version.)

### Zero-downtime deploys

For zero-downtime deploys, use a rolling update that takes instances out of the load balancer before terminating them. `/health` reports the app as healthy once it's finished loading; the load balancer stops sending traffic once the process is told to drain.

Integrations are a wrinkle: inbound integrations with long-running loops (like timers) should be drained gracefully before the process exits. NeoHaskell doesn't currently have a built-in "graceful shutdown" hook — you'd need to trap SIGTERM and give the integrations a few seconds to finish. This is on the roadmap.

## Observability

The minimum viable observability setup:

- **Structured logs to stdout.** NeoHaskell already does this.
- **Error alerting.** Wire your log collector to alert on any `ERROR`-level log.
- **Latency percentiles.** Measure p50, p95, p99 for each command type at the transport layer.
- **Event append rate.** Useful for capacity planning.
- **Database connection pool metrics.** Watch for exhaustion.

For Prometheus-style metrics export, NeoHaskell doesn't currently have a first-class integration — you'd add one yourself with a small custom integration. Expect this to be built-in soon.

## Rollbacks

A rollback is redeploying the previous binary. Because state is in the event store (which is backwards-compatible across versions), rolling back is safe as long as the schema hasn't changed in a breaking way.

Two things to watch:

1. **Projections rebuilt under the new version** may have a shape that old code doesn't understand. If you introduced a new field in a projection, the old code won't know about it. This is usually harmless (the field is just ignored), but verify.
2. **Events emitted under the new version** are still in the store after rollback. Make sure the old code handles them gracefully (usually by ignoring unknown variants — NeoHaskell does this via the `AnyStreamState` insertion type and the exhaustiveness checks in `update`).
