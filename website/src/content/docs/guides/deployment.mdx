## Building for Production

{/*
  The build step. neo build vs nix build. What the output looks like.
*/}


- `neo build` compiles your application for production — optimized, no debug symbols
- The output is a single statically-linked binary — no runtime dependencies, no JVM, no interpreter
- Show: the `neo build` command and its output
- Show: where the binary is placed (`dist/` or similar)
- Build with Nix for reproducible builds: `nix build` produces the same binary on any machine
  - The Nix build is hermetic — it doesn't depend on anything installed on the build machine
  - Useful for CI: the build is deterministic, so you can cache it
- Build flags:
  - `--optimize` — enables additional optimizations (slower build, faster runtime)
  - `--target` — cross-compile for a different architecture (e.g., build on macOS, deploy to Linux)
- Note: the binary includes everything — no separate config files required at build time

## Container Images (Docker/OCI)

{/*
  How to package the binary in a container image.
  NeoHaskell's static binary makes this very simple.
*/}


- Because the binary is statically linked, the container image can be minimal
- Show: a minimal Dockerfile using a scratch or distroless base image
  - `FROM scratch` — the smallest possible image (just the binary)
  - `FROM gcr.io/distroless/static` — slightly larger, but includes CA certificates for HTTPS
- Show: building the container image with `docker build`
- Show: running the container locally with `docker run`
- Multi-stage build: use Nix to build the binary, then copy it into a minimal image
  - Show: a multi-stage Dockerfile with a Nix build stage
- Image size: a NeoHaskell service image is typically 10-50MB — much smaller than JVM or Node.js services
- Note: the binary needs to be compiled for the target architecture — use `--target linux/amd64` if building on macOS

{/*
  How NeoHaskell's defineConfig system works in production.

- Every config field becomes an environment variable: `DATABASE_URL`, `PORT`, `LOG_LEVEL`

## Basic Monitoring Setup

{/*
  What to monitor in an event-sourced service.
  Metrics, logs, traces.
*/}


- What to monitor in an event-sourced service:
  - **Command throughput**: commands per second, by command type
  - **Command latency**: p50, p95, p99 latency for command processing
  - **Error rate**: domain rejections vs. infrastructure errors (different alerts)
  - **Event store lag**: how far behind is the query processor?
  - **Event store size**: total events, events per entity type
- NeoHaskell emits structured logs by default — compatible with any log aggregation system
- Show: the log format (JSON, with correlation IDs)
- Metrics: NeoHaskell exposes a `/metrics` endpoint in Prometheus format
- Show: a basic Prometheus scrape config
- Tracing: NeoHaskell supports OpenTelemetry — show how to configure the OTLP exporter
- Alerting recommendations:
  - Alert on infrastructure errors (event store connection failures)
  - Don't alert on domain rejections (those are expected business behavior)
  - Alert on command latency p99 > threshold

## Event Store Persistence

- The in-memory event store is for development only — it loses all data on restart
- For production, use PostgreSQL as the event store
- Switching is a single configuration change — no code changes required
- See the full guide: [PostgreSQL Event Store](/guides/postgresql-event-store/)

## Scaling Considerations

{/*
  How to think about scaling a NeoHaskell service.
  Start simple, scale when you need to.
*/}


- Start with a single instance — NeoHaskell services are stateless (state is in the event store)
- Horizontal scaling: run multiple instances behind a load balancer
  - All instances read from the same event store
  - Commands are idempotent — duplicate delivery is safe
- Vertical scaling: NeoHaskell services are efficient — a single instance handles significant load
- When to scale:
  - Scale when command latency increases under load
  - Scale when the event store becomes the bottleneck (see PostgreSQL guide)
- What NOT to do: don't run multiple instances with the in-memory event store — they won't share state

<Aside type="note" title="Start simple">
Most NeoHaskell services don't need horizontal scaling at launch. A single well-configured instance handles thousands of commands per second. Scale when you have evidence you need to, not preemptively.
</Aside>
