NeoHaskell gives you three ways to represent something going wrong: `Decider.reject`, `Task.throw`, and `Result`. They're not interchangeable. Picking the right one matters for both correctness and operational clarity.

This guide explains when to use each, how they compose, and the specific patterns you'll see in real code.

## The three error types

| Mechanism | Lives in | Used for | Carries |
|---|---|---|---|
| **`Decider.reject reason`** | `Decision` (pure) | Business rule violations inside a command | A `Text` reason |
| **`Task.throw err`** | `Task err val` (effectful) | System failures — I/O, network, etc. | A typed error of your choice |
| **`Result err val`** | Any (pure) | Pure computations that can fail | A typed error |

## When to use `Decider.reject`

Use rejection for **expected business outcomes** inside a command's `decide` function.

- "Account doesn't exist."
- "Insufficient balance."
- "Email already registered."
- "Permission denied."

A rejection is not a bug. It's a normal, expected outcome — the domain told the caller no. The caller gets a structured `Rejected` response with the reason; your logs stay quiet.

```haskell
decide :: Deposit -> Maybe Account -> RequestContext -> Decision AccountEvent
decide cmd account _ctx = case account of
  Nothing -> Decider.reject "Account does not exist"
  Just _  -> if cmd.amount <= 0
    then Decider.reject "Amount must be positive"
    else Decider.acceptExisting [ MoneyDeposited { ... } ]
```

Rejection reasons are free text. Keep them user-friendly — they might appear in an API response, an error toast, or a metric label. Use full sentences, not codes.

<Aside type="note" title="Why not use exceptions?">
If you threw an exception from `decide`, the command executor would catch it as an unexpected failure and log it as an error. That's wrong — the command simply wasn't accepted. Rejection is the semantic fit.
</Aside>

## When to use `Task.throw`

Use `Task.throw` for **unexpected system failures** in effectful code (anywhere you're inside a `Task`).

- "Database connection lost."
- "External API returned 503."
- "Failed to parse configuration file."
- "Deserialization failed."

`Task.throw` puts a typed error into the failure channel of the `Task`. It propagates up through `<-` bindings until someone catches it with `Task.recover` or until it reaches the top of the call stack.

```haskell
fetchConfig :: Path -> Task Text Config
fetchConfig path = do
  contents <- readFile path
  case parseConfig contents of
    Ok config -> Task.yield config
    Err parseError -> Task.throw [fmt|Invalid config at #{path}: #{parseError}|]
```

Notice that the error channel is `Text` here — a human-readable message. For richer errors, use a sum type:

```haskell
data ConfigError
  = FileNotFound Path
  | ParseError Text
  | MissingRequiredField Text

fetchConfig :: Path -> Task ConfigError Config
fetchConfig path = do
  contents <- readFile path |> Task.mapError (\_ -> FileNotFound path)
  case parseConfig contents of
    Ok config -> Task.yield config
    Err parseError -> Task.throw (ParseError parseError)
```

Typed errors let callers switch on the failure type and react differently to different causes.

## When to use `Result`

Use `Result err val` in **pure code** that can fail but isn't inside a `Task`.

```haskell
parseAge :: Text -> Result Text Int
parseAge text = case Int.fromText text of
  Nothing -> Err "not a valid integer"
  Just n -> if n < 0 then Err "age must be non-negative" else Ok n
```

`Result` is the pure cousin of `Task`. It's what you use inside `decide` functions (which are pure), or in helper functions that don't need side effects.

Converting a `Result` to a `Task`:

```haskell
fromResult :: Result err val -> Task err val
fromResult r = case r of
  Ok v -> Task.yield v
  Err e -> Task.throw e
```

## The decision tree

Use this to pick the right tool:

```
Is the failure a business rule violation inside a decide function?
├── YES → Decider.reject
└── NO  → Is the code inside a Task?
         ├── YES → Task.throw
         └── NO  → Result
```

## Propagation

Errors propagate automatically in `do` blocks. If any `<-` step fails, the rest of the block is skipped:

```haskell
loadDashboard :: Uuid -> Task Text Dashboard
loadDashboard userId = do
  user <- fetchUser userId       -- if this fails, posts/stats are never fetched
  posts <- fetchPosts userId     -- if this fails, stats is never fetched
  stats <- fetchStats userId
  Task.yield Dashboard { user = user, posts = posts, stats = stats }
```

If `fetchUser` fails, the whole task fails with whatever error `fetchUser` threw. The caller sees the error; you didn't have to write try/catch.

## Recovering from errors

Three ways to recover:

### `Task.recover`

Run a recovery function when the task fails:

```haskell
fetchUserOrDefault :: Uuid -> Task err User
fetchUserOrDefault userId =
  fetchUser userId
    |> Task.recover (\_err -> Task.yield defaultUser)
```

### `Task.asResult`

Convert a failing task into one that always succeeds with a `Result`:

```haskell
handleRequest :: Uuid -> Task err Text
handleRequest userId = do
  result <- Task.asResult (fetchUser userId)
  case result of
    Ok user -> Task.yield [fmt|Hello, #{user.name}|]
    Err reason -> Task.yield [fmt|Could not load user: #{reason}|]
```

### `Task.mapError`

Transform one error type into another as the task propagates:

```haskell
readConfigTyped :: Path -> Task AppError Config
readConfigTyped path =
  readConfig path
    |> Task.mapError (\textErr -> AppConfigError textErr)
```

This is how you compose errors across layers: the lower layer uses its own error type, the upper layer converts as it wraps.

## Composing errors across services

Different parts of your system may use different error types. The pattern is:

1. **Each service layer uses its native error type.** The event store uses `EventStoreError`. The HTTP transport uses `Text`. Your integration uses `IntegrationError`. Etc.
2. **Adapters map between layers.** When you call the event store from an integration, you `Task.mapError` to convert.
3. **At the top of the call stack, you have one type.** Usually `Text`, often wrapped in a richer type that includes the layer the error came from.

A typical wrapper:

```haskell
data AppError
  = EventStoreFailure Text
  | IntegrationFailure IntegrationError
  | ValidationFailure Text

runApp :: Application -> Task AppError Unit
runApp app = do
  Application.run app |> Task.mapError EventStoreFailure
```

## Logging errors

When you catch an error, log it before recovering:

```haskell
fetchWithFallback :: Uuid -> Task err User
fetchWithFallback userId =
  fetchUser userId
    |> Task.recover (\err -> do
        Log.warn [fmt|fetchUser failed for #{userId}: #{err}. Using fallback.|]
        Task.ignoreError
        fetchUserFromReplica userId)
```

A rule of thumb: **every caught error should be logged somewhere**. Silently swallowed errors are the source of half of all production mysteries.

## Rejection reasons vs. error messages

Rejection reasons (from `Decider.reject`) are **user-facing**. They should be actionable by the caller:

- ✅ "Insufficient funds"
- ✅ "Email must be a valid address"
- ✅ "This account is closed and cannot accept deposits"
- ❌ "Balance check failed at line 47 of Withdraw.hs"
- ❌ "null pointer at AccountService.java:213"

Error messages (from `Task.throw`) are **developer-facing**. They go to logs and alerting, not to users:

- ✅ "Failed to connect to PostgreSQL: connection refused (127.0.0.1:5432)"
- ✅ "JSON decode failed at offset 142: expected integer, got string"
- ✅ "OAuth2 token refresh failed: invalid_grant"

Never show a raw `Task.throw` error to an end user. Wrap it and replace the message with something helpful.

## Pattern: authorization inside decide

A common pattern you'll see in real codebases:

```haskell
decide cmd entity ctx = case ctx.user of
  Nothing ->
    Decider.reject "Authentication required"
  Just user ->
    case entity of
      Nothing ->
        Decider.reject "Resource does not exist"
      Just e ->
        if e.ownerId /= user.sub
          then Decider.reject "You do not own this resource"
          else -- ... proceed
            Decider.acceptExisting [ ... ]
```

Authorization failures are rejections, not exceptions. They're expected outcomes — the caller asked for something they're not allowed to do. Log them if you want (for audit), but don't treat them as bugs.

## Pattern: graceful degradation

When a secondary feature fails, let the primary feature succeed:

```haskell
serveHomepage :: Task err HomepageHtml
serveHomepage = do
  posts <- fetchPosts  -- primary data
  sidebar <- fetchSidebar
    |> Task.recover (\_err -> Task.yield emptySidebar)  -- graceful fallback
  Task.yield (render posts sidebar)
```

If the sidebar service is down, the user still sees the posts. If the main data fails, the whole request fails.

## Pattern: retry with backoff

For transient failures, retry with exponential backoff:

```haskell
import Timer qualified

retryWithBackoff :: Int -> Task err val -> Task err val
retryWithBackoff maxAttempts action = go 1
  where
    go attempt
      | attempt >= maxAttempts = action
      | otherwise =
          action |> Task.recover (\_err -> do
            Timer.sleep (Timer.seconds (2 ^ attempt))
            go (attempt + 1))
```

<Aside type="caution" title="Let the executor handle command retries">
The command executor already retries on concurrency conflicts automatically. Don't wrap commands in your own retry logic — you'll double up.
</Aside>
