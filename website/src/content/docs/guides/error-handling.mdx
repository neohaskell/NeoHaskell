## The Result Type: Ok and Err

{/*
  The foundation. Result is the synchronous error type.
  Show the type, show the constructors, show basic usage.
*/}


- `Result err val` represents a computation that either succeeds (`Ok val`) or fails (`Err err`)
- This is not an exception — it's a value. You can pass it around, store it, pattern match on it.
- The type signature documents every possible failure: `Result AccountError Account` tells you exactly what can go wrong
- Pattern matching on Result is exhaustive — the compiler forces you to handle both cases
- Show: creating an `Ok` value
- Show: creating an `Err` value
- Show: pattern matching on a Result
- Contrast with exceptions: exceptions are invisible in type signatures; Result errors are explicit
- Note: `Result` is for synchronous operations. For async operations, use `Task`.

## The Task Type: Async Results

{/*
  Task is the async version of Result. It's the primary type for effectful operations.
*/}


- `Task err val` is like `Result err val` but for async operations (database calls, HTTP requests, file I/O)
- A `Task` is a description of a computation — it doesn't run until you execute it
- The error type `err` is explicit in the signature — no hidden exceptions

## Error Propagation with Task.andThen

{/*
  The key composition operator. This is how you chain operations without nesting.
*/}


- `Task.andThen` chains two Tasks: if the first succeeds, run the second with the result; if the first fails, short-circuit
- This replaces nested try/catch blocks — errors propagate automatically

## Custom Error Types for Your Domain

{/*
  How to define error types that are meaningful to your domain.
*/}


- Define error types as sum types (union types) — one constructor per failure mode
- Example: `AccountError = AccountNotFound | InsufficientFunds | AccountAlreadyClosed`
- Each constructor can carry data: `InsufficientFunds { requested: Amount, available: Amount }`
- The error type becomes documentation — anyone reading the function signature knows every way it can fail

- Convention: name error types after the domain concept, not the operation (`AccountError`, not `DepositError`)
- Keep error types in the same module as the entity they describe

## Composing Error Types Across Services

{/*
  The hard part: when you call code from another service, the error types don't match.
  This section shows the standard patterns for handling this.
*/}


- When a Service calls an Integration, the error types are different — you need to compose them
- Option 1: `Task.mapError` — transform the inner error into the outer error type
  - Use when: the outer context knows about the inner error and wants to handle it specifically
  - Show: mapping a `DatabaseError` to an `AccountError`
- Option 2: Wrap in a sum type — add a constructor for "integration failed"
  - Use when: the outer context doesn't care about the details of the inner failure
  - Show: `AccountError = ... | IntegrationFailed Text`
- Option 3: Use a shared error type — define a common error type for cross-context failures
  - Use when: multiple services share the same infrastructure errors
  - Show: a `ServiceError` type that wraps infrastructure errors
- Guideline: keep domain errors domain-specific; wrap infrastructure errors at the boundary

## Error Recovery Strategies

{/*
  What to do when an error occurs. Not all errors are fatal.
*/}


- `Task.orElse` — try an alternative if the first Task fails
  - Use when: you have a fallback (e.g., cache miss → fetch from database)
  - Show: `Task.orElse` with a fallback Task
- `Task.recover` — handle a specific error and continue
  - Use when: a specific error is expected and recoverable
  - Show: recovering from `AccountNotFound` by creating a new account
- `Task.attempt` — convert a failing Task into a `Result` — never fails, always returns `Ok (Result err val)`
  - Use when: you want to inspect the error without propagating it
  - Show: using `Task.attempt` to log an error and continue
- Guideline: recover at the boundary, not deep in the call stack
- Guideline: don't recover from errors you don't understand — let them propagate

<Aside type="note" title="Errors vs. Domain Rejections">
Not every "failure" is an error. A command that's rejected because the business rules say no (e.g., insufficient funds) is a domain rejection — it's expected, it's correct behavior, and it should be modeled as a `Result.err` with a domain error type. Reserve `Task` failures for unexpected infrastructure problems (database down, network timeout). The distinction matters for monitoring: domain rejections are business metrics; infrastructure failures are alerts.
</Aside>
