Reading compiler errors is a skill. It's not obvious, it's not intuitive the first time, and nobody is born knowing how to do it. This page teaches it directly.

## The General Skill

{/*
  Before showing specific errors, teach the meta-skill.
  What do you look for first? Where is the useful information?
*/}

When you see a NeoHaskell error, read it in this order:

1. **The file and line number** — top of the error, tells you where to look
2. **The error category** — "type mismatch", "unknown variable", "non-exhaustive patterns" — tells you what kind of problem it is
3. **The "expected" vs "actual" section** — for type errors, this is the most useful part
4. **The suggestion** — NeoHaskell often tells you what to do. Read it.
5. **The context** — the surrounding code shown in the error. This confirms you're looking at the right place.

- Most errors are in the first 3 lines — don't scroll past them looking for more information
- The error message is not blaming you — it's describing a mismatch between what you wrote and what the compiler expected
- If the error is confusing, look at the line number first, then read the code at that line — often the error makes sense once you see the code

## Anatomy of a NeoHaskell Error Message

  This is the "map" that makes all subsequent examples easier to read.

- Show a complete error message (a type mismatch is a good example)
- Label each part with an annotation:
  - **File path and line:column** — where the error is
  - **Error code** — a short identifier (useful for searching)
  - **Error category** — the type of problem
  - **The problematic expression** — the code that caused the error, highlighted
  - **Expected type** — what the compiler expected
  - **Actual type** — what it found instead
  - **Suggestion** — what to do (not always present, but often helpful)
  - **Context** — surrounding lines for orientation
- Note: NeoHaskell errors are designed to be readable — they're more verbose than GHC errors by design

## Error 1: Type Mismatch

{/*
  The most common error. Show a real example, annotate it, explain the fix.
*/}

**What it looks like**: `Expected type X, but got type Y`

- When you see this: you passed the wrong type to a function, or a function returned the wrong type
- Show the full error message
- Annotate: which part is "expected", which part is "actual"
- Explain: the compiler is telling you the types don't match — one of them is wrong
- How to fix:
  - Look at the function's type signature — what does it expect?
  - Look at what you're passing — what type is it?
  - One of them needs to change
- Common cause: forgetting to unwrap a `Maybe` or `Result` before passing it to a function
- Common cause: using a `Text` where an `Int` is expected (or vice versa)
- Common cause: a function returns `Task err val` but you're treating it as `val`

## Error 2: Missing Pattern Match

{/*
  Non-exhaustive patterns. Common when adding a new event or command variant.
*/}

**What it looks like**: `Non-exhaustive patterns in case expression`

- When you see this: you added a new constructor to a type but didn't handle it everywhere
- This is the compiler enforcing completeness — it's a feature, not a bug
- Show the full error message
- Annotate: which case expression is incomplete, which constructor is missing
- How to fix:
  - Add a case for the missing constructor
  - If you don't know what to do with it yet, use `_ -> Task.fail (NotImplemented "...")`
- Common cause: adding a new event type and forgetting to update the query fold
- Common cause: adding a new command type and forgetting to update the command handler
- Note: this is exactly the compiler catching the "missing event handler" problem described in [Why NeoHaskell](/concepts/why-neohaskell/)

## Error 3: Unknown Function (Forgot Module Prefix)

{/*
  Very common for beginners. The function exists, but you forgot to qualify it.
*/}

**What it looks like**: `Variable not in scope: functionName`

- When you see this: you used a function without its module prefix, or you forgot to import the module
- Show the full error message
- Annotate: the function name that's not found, the suggestion (if any)
- How to fix:
  - Check if the function is in a module you haven't imported
  - Add the module prefix: `List.map` instead of `map`
  - Or add the import: `import Data.List (map)`
- Common cause: using `map` instead of `List.map`
- Common cause: using `andThen` instead of `Task.andThen`
- Common cause: a typo in the function name (the error message often suggests the closest match)
- Tip: NeoHaskell uses qualified imports by convention — always prefix with the module name

## Error 4: Wrong Import

{/*
  Importing something that doesn't exist, or importing from the wrong module.
*/}

**What it looks like**: `Module 'X' does not export 'Y'`

- When you see this: you're trying to import something that doesn't exist in that module
- Show the full error message
- Annotate: the module name, the thing you tried to import
- How to fix:
  - Check the module's documentation — is the name spelled correctly?
  - Check if it moved to a different module in a recent version
  - Use `neo docs` to look up what a module exports
- Common cause: importing `Task.andThen` as a standalone function instead of using it qualified
- Common cause: a function was renamed between versions

## Error 5: Linter Warning (Used `$` Instead of `|>`)

{/*
  NeoHaskell's linter enforces the pipe operator style.
  This is a warning, not an error, but it's confusing the first time.
*/}

**What it looks like**: `Warning: Use |> instead of $`

- When you see this: you used Haskell's `$` operator instead of NeoHaskell's `|>` pipe operator
- This is a linter warning, not a compiler error — your code still compiles
- Show the full warning message
- Annotate: the line with `$`, the suggested replacement with `|>`
- How to fix: replace `f $ g x` with `g x |> f`
- Why NeoHaskell prefers `|>`: it reads left-to-right, matching the direction of data flow
- Note: `$` and `|>` are equivalent in behavior — this is purely a style preference enforced by the linter

## When You're Still Stuck

- Copy the error message and search for it in the [NeoHaskell Discord](https://discord.gg/neohaskell) `#help` channel
- The [Common Errors](/guides/common-errors/) guide is organized by what you were trying to do — it may have your exact error
- If the error is in a file you didn't write (a library), the problem is usually in how you're calling it, not in the library itself
- `neo check` runs the type checker without building — faster feedback loop when debugging type errors

<Aside type="tip" title="The error is almost always on the line before">
Type errors often point to the line where the mismatch is detected, not where it was introduced. If the error points to line 42 but line 42 looks fine, look at line 41 — that's often where the wrong type was created.
</Aside>
