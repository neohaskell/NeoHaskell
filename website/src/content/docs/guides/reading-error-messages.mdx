NeoHaskell compiler errors can look intimidating the first time. This page teaches you to read them fast. The secret is that they follow a predictable structure, and once you know the structure, you can skim most errors in two seconds and zero in on the fix.

## The anatomy of a compiler error

Every GHC error has the same rough shape:

```text
src/MyApp/Foo.hs:42:15: error:
    <summary of the problem in one or two lines>

    <detailed expansion — types involved, expected vs. actual>

    |
 42 |   result = someFunction wrongArg
    |                         ^^^^^^^^
```

Four parts:

1. **Location** (`file:line:col`)
2. **Summary** (a one-sentence description)
3. **Detail** (the specific types and what's wrong with them)
4. **Source pointer** (the line of code with a caret showing the exact position)

**Read the summary first, then the detail.** The source pointer is last. Most errors can be understood from the summary alone.

## Error catalog — what you'll actually see

### "Couldn't match expected type ... with actual type ..."

This is the most common error. You passed one type where another was expected.

```text
src/MyApp/Deposit.hs:15:20: error:
    • Couldn't match expected type 'Int' with actual type 'Text'
    • In the second argument of 'Deposit', namely 'amount'
      In the expression: Deposit { accountId = ..., amount = "50" }
```

**What it means:** `Deposit.amount` is `Int`, but you passed `"50"` which is `Text`.

**The fix:** Write `50` (without quotes), or use `Int.fromText "50"` if you really have a string.

### "Non-exhaustive patterns in function ..."

You forgot to handle a case in a `case` expression or a function.

```text
src/MyApp/Account.hs:88:1: warning:
    Non-exhaustive patterns in function update

    In an equation for 'update':
        Patterns of type 'AccountEvent' not matched:
            MoneyWithdrawn{}
```

**What it means:** Your `update` function handles some `AccountEvent` cases but not `MoneyWithdrawn`.

**The fix:** Add a `MoneyWithdrawn` branch to the `case` expression.

<Aside type="note" title="This is a warning, but it should be treated as an error">
By default GHC emits non-exhaustive patterns as a warning, not an error. The NeoHaskell build pipeline treats warnings as errors, so it *will* fail the build. Fix it.
</Aside>

### "Variable not in scope: ..."

You used a name that isn't imported.

```text
src/MyApp/Deposit.hs:12:10: error:
    Variable not in scope: Text.trim
```

**What it means:** You called `Text.trim` but didn't import the `Text` module.

**The fix:** Add `import Text qualified` to the top of the file.

### "No instance for (Entity Foo)"

You tried to use a type as an entity but didn't give it the `Entity` instance.

```text
src/MyApp/Account.hs:14:1: error:
    • No instance for (Entity Account) arising from a use of 'withService'
```

**The fix:** Add the `instance Entity Account where ...` declaration to your `Core.hs`.

### "Event type mismatch in service definition!"

Two commands in the same service use different event types.

```text
error:
    Event type mismatch in service definition!

    All commands in a service must use the same event type.

    Expected event type: CartEvent
    But command 'ProcessPayment' uses event type: PaymentEvent

    To fix: Ensure all commands in this service share a common event type.
```

**The fix:** Move `ProcessPayment` to a separate service that handles `PaymentEvent`. Or merge the two event types if they really belong together.

### "Couldn't match expected type 'Task Text ...' with actual type ..."

You returned a raw value where a `Task` was expected.

```text
error:
    • Couldn't match expected type 'Task Text User'
                  with actual type 'User'
    • In the expression: someUser
```

**The fix:** Wrap the value in `Task.yield`:

```haskell
fetchUser :: Task Text User
fetchUser = Task.yield someUser
```

### "Variable not in scope: pure"

You tried to use `pure` but NeoHaskell uses `Task.yield` instead.

```text
error:
    Variable not in scope: pure :: User -> Task err User
```

**The fix:** Replace `pure` with `Task.yield`.

### "Ambiguous type variable"

The compiler can't figure out which type you meant.

```text
error:
    • Ambiguous type variable 'a0' arising from a use of 'Json.decode'
    • In the expression: Json.decode someBytes
```

**The fix:** Add a type annotation to tell the compiler what you want:

```haskell
result :: User
result = Json.decode @User someBytes
```

### "Orphan instance"

You defined a typeclass instance in a module that doesn't own either the typeclass or the type.

```text
warning:
    Orphan instance: instance Json.ToJSON MyType
```

**The fix:** Move the instance to the module that defines `MyType` (usually the same file). Orphan instances can cause confusing behavior and GHC warns about them by default.

### "Record update has multiple constructors"

You tried to update a field on a record, but the type has multiple constructors and not all of them have that field.

```text
error:
    • Record update has multiple constructors
```

**The fix:** Use a `case` expression to handle each constructor explicitly:

```haskell
case event of
  AccountOpened { ... } -> event { ... }
  MoneyDeposited { ... } -> event { ... }
```

### "Could not find module 'Core'"

Your `.cabal` file isn't set up to include the NeoHaskell core library.

**The fix:** Check that the `build-depends:` section of your `.cabal` file includes `nhcore`. If you're starting from the template, this is already configured.

## The "type error wall of text" strategy

Sometimes you get a multi-screen error. Here's how to handle it:

1. **Read the first line of the summary.** This tells you the category of problem.
2. **Look at the first `•` (bullet).** That's the main complaint.
3. **Look at the source pointer.** The `^^^^^^` shows you exactly where.
4. **Go to your code.** Fix the thing at that location.
5. **Rebuild.** Often the "wall of text" collapses into one small error once you fix the first mistake.

Do NOT try to parse the entire error message on first read. The first problem is almost always the real one; the rest are consequences.

## Tooling tips

### Use HLS (the language server)

If you have HLS set up in your editor, you'll see errors inline as you type. This catches most issues before you ever run `cabal build`. The NeoHaskell template ships with HLS config, so it should "just work" when your editor is pointed at the right project.

### Use `cabal build --ghc-options="-fdefer-type-errors"` for exploration

If you're refactoring and want to keep iterating without fixing every error immediately, you can defer type errors to runtime:

```bash
cabal build --ghc-options="-fdefer-type-errors"
```

This lets broken code run (and fail at runtime when it hits the error). Use sparingly — it hides problems.

### Use `:reload` in GHCi

If you're exploring in GHCi and you change a file, `:reload` picks up the new version and re-type-checks. Faster than exiting and restarting.

## When an error makes no sense

Rare but real: sometimes an error points to a line that looks obviously correct. The usual causes:

- **A hidden typeclass mismatch.** Some function expects a typeclass you forgot to derive.
- **A type family that doesn't match.** `type instance EntityOf Foo = Bar` but you wrote `Baz` somewhere.
- **An implicit import.** The function you're using comes from a module you didn't import, and there's a same-named local function.

When this happens, try:

1. Look for any `type instance` declarations related to the types in the error.
2. Check that you've imported all the modules you think you imported.
3. Try adding an explicit type annotation to the problematic expression. Often the annotation will either fix the error or produce a better one.
