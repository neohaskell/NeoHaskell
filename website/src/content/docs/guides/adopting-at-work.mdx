This is the guide you want if you just finished the tutorial, you're convinced, and now you have to explain this decision to the rest of your team. Enthusiasm is not a plan. This page is the plan.

Adopting a new language is rarely a technical problem — it's an organizational problem. The code works. The real questions are: will the team learn it? Will we be able to hire? What if the champion leaves? What if it turns out to be the wrong call? Addressing those questions honestly is what makes an adoption stick.

## Step 1 — Be honest with yourself about fit

Before you pitch NeoHaskell to anyone, check the [trade-offs](/concepts/trade-offs/) page and the [architecture evaluation](/concepts/architecture-evaluation/). Answer these questions for yourself first:

- Does our domain genuinely benefit from event sourcing? If not, NeoHaskell is over-engineering.
- Do we have at least one engineer (ideally two) willing to learn it deeply?
- Do we have the organizational appetite for a slower first month in exchange for a faster long-term pace?
- What will we do if the maintainers of NeoHaskell disappear in a year?

If you can't answer these confidently, don't pitch it yet. Do a weekend project first, validate your own conviction, then come back.

## Step 2 — Pick a pilot project

The single biggest factor in a successful adoption is the pilot project. Pick badly and you'll spend months fighting. Pick well and you'll have a reference everyone can point to.

### Ideal pilot characteristics

- **New feature, not a rewrite.** Rewriting an existing system means you're fighting the old one and the new language at the same time. A new feature lets NeoHaskell succeed on its own merits.
- **Clearly bounded.** The smaller and more independent the better. "Add a new microservice" is easier than "rewrite the checkout flow."
- **Domain-rich.** NeoHaskell shines when there are interesting state transitions. A settings panel is a bad pilot. An order management service is a good one.
- **Non-critical, but visible.** If the pilot fails, the business doesn't suffer. If it succeeds, people notice.
- **Has a clear success metric.** "Fewer bugs in production", "faster feature delivery", "better audit capability" — pick one and measure it.

### Anti-patterns

- **The "rewrite everything" pilot.** Never. Always additive.
- **The "tiny CRUD endpoint" pilot.** Doesn't show the benefits of NeoHaskell; people come away thinking "that was a lot of work for a simple feature."
- **The "mission-critical" pilot.** If it fails, the damage is catastrophic. Start lower stakes.
- **The "nobody needs this" pilot.** If nobody cares about the feature, success doesn't buy you political capital.

## Step 3 — Write a proposal

You need something written down. A proposal for engineering leadership should cover:

1. **The problem.** What's the specific pain point NeoHaskell solves for us? Be concrete — "we've had three incidents in the last year where state was inconsistent across services and we couldn't reconstruct the history" is much better than "we want better types."
2. **The proposal.** "We pilot NeoHaskell for [specific feature] with [specific engineer(s)] over [specific timeline]."
3. **The investment.** Engineering time, training, tooling setup.
4. **The success criteria.** What outcomes would justify broader adoption? What would trigger a rollback?
5. **The rollback plan.** If the pilot fails, how do we unwind it without wasting the work?
6. **The FAQ.** Preempt the objections. The next section is the material for this.

Keep the proposal to two pages. Executives read the summary and the FAQ. Detail goes in appendices.

## Step 4 — Preempt the objections

Here are the objections you will hear, and honest responses.

### "Who will maintain this after you leave?"

**Response:** "Any engineer who can read Go or Python can learn NeoHaskell in a few weeks. The [tutorial](/tutorial/) gets them productive, and the [concept pages](/concepts/) teach the mental model. If I leave, the next engineer will ramp up faster than on any legacy codebase because the code is smaller and the compiler catches most mistakes."

### "We can't hire NeoHaskell developers."

**Response:** "We don't need to. We hire generalists and train them — the same way we'd train someone on our internal frameworks, except the training materials are better. The relevant skill is thinking in events, and that transfers from any language."

### "What if NeoHaskell is abandoned?"

**Response:** "It compiles to native code via GHC, which is a production-grade compiler with decades of history. If the NeoHaskell project stagnates, we can keep running the code we have indefinitely. If we need to add new features, we can fork it. It's open-source and the code is approachable."

### "Functional programming is too hard for our team."

**Response:** "The code we'll write looks like this [show a `decide` function from the tutorial]. It has `case`, `if`, records, and pipes. No monads, no category theory, no academic jargon in our application code. Any developer who has written a Redux reducer has the skills."

### "We already have patterns for event sourcing in [X]. Why switch?"

**Response:** "We have patterns, but the compiler doesn't enforce them. In NeoHaskell, breaking the event-sourcing contract is a compile error, not a runtime one. That's the difference between 'we have a convention' and 'the convention is a type.'"

### "This will slow us down in the short term."

**Response:** "Yes, for about a month. After that, we'll be faster than before because the compiler catches mistakes that currently become production incidents. The pilot project's metrics will tell us whether the tradeoff holds."

### "We can't afford to experiment."

**Response:** "We also can't afford to keep fighting the same class of bugs. The pilot is bounded — one feature, one timeline, clear success criteria. If it doesn't work, we spent a month; if it does, we learned how to build more reliable systems."

## Step 5 — Run the pilot

During the pilot, some habits that help:

### Over-communicate

Send a weekly update to your engineering org. Include:

- What shipped.
- What surprised you (good and bad).
- What you learned.
- What's next.

Short, specific, regular. People trust work they can see. Missing updates breed suspicion.

### Write down the lessons

Keep a living document of "things we learned about NeoHaskell that surprised us." This becomes your internal onboarding doc when you expand.

### Benchmark against the alternative

If the pilot replaces something that would have been built in another language, keep track of the "alternate history" — how many lines of code, how many bugs, how much test coverage. When you pitch broader adoption, these numbers will matter.

### Pair with skeptics

Invite your most skeptical teammate to pair-program on the pilot. Either they'll become your second champion, or you'll learn about objections you hadn't considered. Both outcomes are good.

### Don't sell it internally until you're ready

Early in the pilot, focus on making the pilot work. Resist the urge to evangelize until you have something compelling to show. Early evangelism with unfinished work creates headwinds.

## Step 6 — Evaluate honestly

At the end of the pilot:

- Did it meet the success criteria?
- Did it cost more or less than you expected?
- What surprised you?
- What would you do differently next time?
- Would the team be comfortable continuing in NeoHaskell?

Write it up honestly. If the pilot failed, say so. A failed pilot that you can explain is better than a "success" that nobody believes.

## Step 7 — Expand or unwind

If the pilot succeeded:

1. **Present the results** to engineering leadership.
2. **Propose the next project.** Don't try to convert everything at once; pick another bounded feature.
3. **Train a second champion.** Someone who's not you.
4. **Document conventions.** As you adopt more of the language, write down the patterns that work and the ones that don't.
5. **Grow incrementally.** Six months of gradual adoption beats one big-bang switch.

If the pilot failed:

1. **Document why.** Was it the language, the problem, the team, the timing? Be specific.
2. **Unwind carefully.** The feature you built needs to keep running (or be rewritten in your usual stack). Don't let the "pilot" code become orphaned tech debt.
3. **Share the lessons.** A failed pilot is a learning opportunity for the org. Treat it that way.

## A realistic timeline

For a team of 2–3 engineers working on the pilot part-time (not full-time, not AI-assisted, writing everything by hand):

- **Week 1:** Installation, template setup, tutorial completion for the pilot team.
- **Week 2:** First few commands and queries written. Productive but slow.
- **Week 3:** Feature feels "small enough" that the team is confident shipping.
- **Week 4:** First feature deployed to a canary environment.
- **Weeks 5–8:** Iterate, fix bugs, learn the edge cases. Feature running in production.
- **Weeks 9–12:** Second feature, often 2x faster than the first.
- **Month 4+:** Consider expanding. The team should feel comfortable picking up NeoHaskell work without friction.

If someone on the team is working full-time with an AI coding assistant (Claude Code, Cursor, etc.), expect roughly 3–5x compression on those timelines. The compiler validates the assistant's output, so the ramp is less about "learn the syntax" and more about "learn the domain modeling." A full-time, AI-assisted engineer typically ships the first feature in week 1 and hits full stride by week 3. The tutorial takes an afternoon instead of a week because the assistant handles the boilerplate while the engineer focuses on understanding the patterns.

## When to stop trying

Some adoptions don't stick. Signs that it's time to step back:

- After the pilot, the team is still not comfortable working in NeoHaskell. This is OK — not every team has the interest.
- Leadership won't commit to broader adoption even after a successful pilot. This is usually organizational, not technical.
- You, the champion, have left the company (or plan to). Adoption that depends on one person rarely survives.

If any of these happen, the honest answer is to wind down the NeoHaskell work and continue in your usual stack. Keep the experience — it's valuable for the next team, the next company, or the next problem.

## The bottom line

NeoHaskell adoption is about **earning trust incrementally**. A great pilot earns the right to a second project. A second success earns the right to a standard. You don't get that trust by evangelizing — you get it by shipping something that demonstrably works better than the alternative.

Pick a pilot you can win. Ship it. Measure. Expand.
