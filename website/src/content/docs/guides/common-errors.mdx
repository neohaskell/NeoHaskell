Rather than a generic "here are the error categories" page, this is a catalog of specific errors keyed to the situation that produces them. When you see one of these, jump to the section and apply the fix.

## When adding a new event type

### "Non-exhaustive patterns in function update"

You added a new variant to an event type and didn't handle it everywhere.

**Fix:** Add a branch for the new variant in:

- The `update` function
- The `getEventEntityIdImpl` function (if the new event has an `entityId` field)
- Any `combine` function in queries that cares about the new event

The compiler will walk you through each one.

### "Non-exhaustive patterns in function getEventEntityIdImpl"

Same cause as above — same fix.

## When adding a new command

### "Could not deduce (Record.KnownSymbol ...) arising from a use of 'command'"

You forgot to provide a `NameOf` instance for your command type.

**Fix:** The `command ''Foo` Template Haskell line at the bottom of your command file generates this automatically — make sure you haven't removed it. If you still get the error after the TH line, check that the command type is at the top level of the module and is exported.

### "No instance for (Schema.ToSchema ...) arising from a use of 'command'"

Your command type needs a `ToSchema` instance so NeoHaskell can generate the OpenAPI spec.

**Fix:** Add `deriving (Generic)` to the command type and import `import Schema qualified`. The `command ''Foo` line will derive the rest.

### "No instance for (Json.FromJSON ...)"

Same fix — ensure `deriving (Generic)` and add `instance Json.FromJSON YourCommand`.

### "Couldn't match type 'EventOf EntityOf SomeCommand'"

You declared `EntityOf SomeCommand = Foo` but `Foo`'s `EventOf` isn't what `decide` returns.

**Fix:** Verify the chain:
- `EntityOf YourCommand = YourEntity`
- `EventOf YourEntity = YourEvent`
- `decide ... -> Decision YourEvent`

All three must line up. The compiler error usually names exactly which link is broken.

## When adding a new service

### "Event type mismatch in service definition!"

You tried to register two commands in the same service, but they have different event types.

**Fix:** Put each command in a service matching its entity/event type. If you have `CartCommand`s and `OrderCommand`s, you need `cartService` and `orderService` as two separate things. The compiler error names the specific command that caused the mismatch.

### "No instance for (ServiceEventType cmds)"

Your service has no commands registered yet.

**Fix:** Add at least one `|> Service.command @SomeCommand`.

## When adding a new query

### "No instance for (QueryOf Foo Bar)"

You declared a query that depends on entity `Foo` but didn't write the `QueryOf Foo Bar` instance.

**Fix:** Add:

```haskell
instance QueryOf Foo Bar where
  queryId foo = foo.id
  combine foo existing = Update Bar { ... }
```

### "deriveQuery: symbol already exists"

Two different query types were derived with the same name.

**Fix:** Rename one of them. Each query type gets a `NameOf` instance based on its Haskell name — the kebab-case form — so `MyQuery` and `MyOtherQuery` both get unique names, but `MyQuery` in two modules collides.

## When wiring the application

### "withEventStore requires withConfig when the factory uses the config parameter"

You called `withEventStore makePostgresConfig` where `makePostgresConfig` takes your config type, but you never called `withConfig @MyConfig`.

**Fix:** Add `|> Application.withConfig @MyConfig` before the `withEventStore` call. Or, if you don't need config, use the `@()` form: `withEventStore @() (\_ -> someConfig)`.

### "No EventStore configured. Use withEventStore."

You called `Application.run` without ever calling `withEventStore`.

**Fix:** Add `|> Application.withEventStore ...` to the builder chain.

### "Application.withConfig called twice. Only one config per Application is supported."

You chained `withConfig` twice with different types.

**Fix:** Consolidate into a single config type that contains everything you need.

### "Transport not registered: &lt;name&gt;"

Your command declares a transport via `TransportsOf`, but the application hasn't registered that transport with `withTransport`.

**Fix:** Add `|> Application.withTransport WebTransport.server` (or whichever transport the command expects) to the builder chain.

## When running tests

### "cabal: No such target component: nhcore-test-core"

You're running tests from the wrong directory, or the test suite name doesn't match.

**Fix:** Run `cabal test` (no arguments) to run all suites. If you want a specific suite, check the `.cabal` file for the exact name.

### "Test suite failed to build"

Your test file has a compile error. Look at the output above the "test suite failed to build" line for the actual error.

### "Connection refused (127.0.0.1:5432)"

You're running a test suite that requires PostgreSQL, but PostgreSQL isn't running.

**Fix:** Start PostgreSQL:

```bash
docker run -d --name neohaskell-postgres \
  -e POSTGRES_USER=neohaskell \
  -e POSTGRES_PASSWORD=neohaskell \
  -e POSTGRES_DB=neohaskell \
  -p 5432:5432 \
  postgres:16-alpine
```

Or run only the suites that don't need Postgres: `cabal test nhcore-test-core`.

## When running the app

### "No EventStore configured"

You called `Application.run` without `withEventStore`.

**Fix:** Add `withEventStore` to the builder chain.

### "Connection refused" on startup

PostgreSQL isn't running, or the credentials in your config don't match.

**Fix:** Verify PostgreSQL is up and the `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME` environment variables match. For local dev, the template uses `localhost`, `5432`, `neohaskell`, `neohaskell`, `neohaskell`.

### "Event store schema migration failed"

The database user doesn't have permission to create tables.

**Fix:** Grant the necessary privileges, or use a superuser account for local development.

### "Command dispatched to unknown transport"

An integration tried to dispatch a command, but the command's `TransportsOf` does not include a transport that's registered with the app.

**Fix:** Ensure the command has `InternalTransport` listed in `TransportsOf`, or register the public transport it expects.

## When deploying

### "binary not found" in production

You forgot to `cabal build` before deploying, or the deploy artifact doesn't include the binary.

**Fix:** Your deploy script should run `cabal build` inside the Nix dev shell and copy the resulting binary from `dist-newstyle/build/...` to the deploy target.

### "Glibc version mismatch"

You built the binary on one Linux version and deployed to another with an incompatible libc.

**Fix:** Build inside a container that matches your deploy environment, or use static linking via `ghc-options: -static`.

### "FATAL: no pg_hba.conf entry"

PostgreSQL is rejecting the connection because of access control rules.

**Fix:** Update `pg_hba.conf` on the Postgres server to allow connections from your application host.

## General debugging habits

When you're stuck:

1. **Rebuild from scratch.** `cabal clean && cabal build` clears any stale build cache.
2. **Read the error summary, not the whole wall of text.** The first line of the error is usually the real problem; everything else is consequences.
3. **Add a type annotation.** If a type error looks confusing, add an explicit type signature to the problematic expression. Often the annotation either fixes the problem or produces a better error.
4. **Check the imports.** Many "variable not in scope" errors come from forgetting to import a module.
5. **Ask in Discord.** The [NeoHaskell Discord](https://discord.gg/wDj3UYzec8) has people who have seen every error on this page and many that aren't.
