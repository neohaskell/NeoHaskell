This guide is the practical companion to [Testing event-sourced systems](/concepts/testing-event-sourced-systems/). The concept page explained *why* testing is easy; this page is *how* to set it up, the specific recipes you'll use day-to-day, and how to wire tests into CI.

## Setup

The NeoHaskell template ships with a `tests/` directory pre-wired for `hspec-discover`. Every file named `*Spec.hs` anywhere under `tests/` is automatically picked up. You don't register tests manually.

A minimal test file looks like this:

```haskell
module NeoBank.Account.Commands.DepositSpec where

import Core
import Test

spec :: Spec Unit
spec = describe "Deposit" do
  it "does something" \_ -> do
    Task.yield ()
```

Run all tests:

```bash
cabal test
```

Run only a specific suite:

```bash
cabal test nhcore-test-core
```

## The test helpers you'll use

The `Test` module (imported as `import Test`) re-exports the common helpers:

| Helper | Purpose |
|---|---|
| `describe "..."` | Group related tests |
| `it "..."` | A single test |
| `shouldBe expected` | Assert equality |
| `shouldSatisfy predicate` | Assert a predicate holds |
| `Test.fail "message"` | Fail with a custom message |
| `pending "reason"` | Mark a test as pending (not yet implemented) |

## Recipe 1 — Testing a command's decide function

The most common test. You want to verify that `decide` produces the right result for a given input. The pattern:

1. Set up a test entity state.
2. Build the command.
3. Run `decide` via `runDecision`.
4. Assert on the result.

```haskell
module NeoBank.Account.Commands.WithdrawSpec where

import Array qualified
import Core
import Decider (runDecision, DecisionContext (..), CommandResult (..))
import NeoBank.Account.Commands.Withdraw (Withdraw (..), decide)
import NeoBank.Account.Core
import Service.Auth (RequestContext (..))
import Service.Event (InsertionType (..))
import Task qualified
import Test
import Uuid qualified


-- A helper that runs decide with a default context.
runDecide ::
  Withdraw ->
  Maybe Account ->
  Task Text (CommandResult AccountEvent)
runDecide cmd entity = do
  let ctx = DecisionContext { genUuid = Task.yield Uuid.nil }
  let reqCtx = RequestContext { user = Nothing }
  runDecision ctx (decide cmd entity reqCtx)


-- A helper that builds a test account.
anAccount :: Int -> Account
anAccount balance =
  Account
    { accountId = Uuid.nil
    , ownerName = "Alice"
    , accountType = Checking
    , balance = balance
    , history = Array.empty
    }


spec :: Spec Unit
spec = describe "Withdraw" do
  it "rejects when account does not exist" \_ -> do
    let cmd = Withdraw { accountId = Uuid.nil, amount = 50 }
    result <- runDecide cmd Nothing
    result |> shouldBe (RejectCommand "Account does not exist")

  it "rejects negative amounts" \_ -> do
    let cmd = Withdraw { accountId = Uuid.nil, amount = -10 }
    result <- runDecide cmd (Just (anAccount 100))
    result |> shouldBe (RejectCommand "Withdrawal amount must be positive")

  it "rejects insufficient balance" \_ -> do
    let cmd = Withdraw { accountId = Uuid.nil, amount = 200 }
    result <- runDecide cmd (Just (anAccount 100))
    result |> shouldBe (RejectCommand "Insufficient funds")

  it "accepts valid withdrawal and emits MoneyWithdrawn" \_ -> do
    let cmd = Withdraw { accountId = Uuid.nil, amount = 40 }
    result <- runDecide cmd (Just (anAccount 100))
    let expected =
          AcceptCommand
            ExistingStream
            (Array.fromLinkedList
              [ MoneyWithdrawn { entityId = Uuid.nil, amount = 40 } ])
    result |> shouldBe expected
```

The key insight: `decide` is a pure function. Everything else in the test is boilerplate that you write once (`runDecide`, `anAccount`) and reuse across every test for that command.

## Recipe 2 — Testing the update function

Testing `update` is even simpler. It's a pure function from event and state to new state:

```haskell
module NeoBank.Account.UpdateSpec where

import Core
import NeoBank.Account.Core
import Test

spec :: Spec Unit
spec = describe "Account.update" do
  it "adds balance on deposit" \_ -> do
    let account = initialState { balance = 100 }
    let event = MoneyDeposited { entityId = Uuid.nil, amount = 50 }
    let result = update event account
    result.balance |> shouldBe 150

  it "subtracts balance on withdrawal" \_ -> do
    let account = initialState { balance = 100 }
    let event = MoneyWithdrawn { entityId = Uuid.nil, amount = 30 }
    let result = update event account
    result.balance |> shouldBe 70

  it "can replay a sequence and produce the expected final state" \_ -> do
    let events =
          [ AccountOpened { entityId = Uuid.nil, ownerName = "Alice", accountType = Checking }
          , MoneyDeposited { entityId = Uuid.nil, amount = 100 }
          , MoneyWithdrawn { entityId = Uuid.nil, amount = 30 }
          , MoneyDeposited { entityId = Uuid.nil, amount = 20 }
          ]
    let final = events |> LinkedList.foldl update initialState
    final.balance |> shouldBe 90
```

The last test is a *replay* test: you build a sequence of events, fold them through `update`, and assert on the final state. This is how you test that your entity state machine is correct.

## Recipe 3 — Testing a query's combine function

Queries are pure functions. Test them the same way:

```haskell
module NeoBank.Account.Queries.StatementSpec where

import Array qualified
import Core
import NeoBank.Account.Core
import NeoBank.Account.Queries.Statement
import Service.Query.Core (QueryAction (..))
import Test

spec :: Spec Unit
spec = describe "Statement" do
  it "builds a statement with lines from history" \_ -> do
    let account = Account
          { accountId = Uuid.nil
          , ownerName = "Alice"
          , accountType = Checking
          , balance = 70
          , history = Array.fromLinkedList
              [ Transaction { kind = "deposit", amount = 100 }
              , Transaction { kind = "withdrawal", amount = 30 }
              ]
          }
    case combine account Nothing of
      Update statement -> do
        statement.currentBalance |> shouldBe 70
        Array.length statement.lines |> shouldBe 2
      _ -> Test.fail "Expected Update"
```

## Recipe 4 — Testing the full command flow end-to-end

Unit tests verify `decide`, `update`, and `combine` in isolation. Integration tests verify the whole command flow: decide → append → rebuild entity → update projection. For these, use the in-memory event store.

```haskell
module NeoBank.EndToEndSpec where

import Core
import Service.EventStore.InMemory qualified as InMemory
import Test
import -- ... your domain imports

spec :: Spec Unit
spec = describe "End-to-end flow" do
  it "persists events and rebuilds correct state after a command sequence" \_ -> do
    store <- InMemory.new |> Task.mapError toText
    -- ... dispatch commands via the executor
    -- ... read the resulting state
    -- ... assert
    Task.yield ()
```

The template repository has example integration tests you can copy. The full pattern is more verbose than the unit-test recipes above, but it's the right call for verifying wiring.

## Recipe 5 — Testing an integration

Integrations are pure functions from events to actions. Test the `reactTo` function directly:

```haskell
module NeoBank.Transfer.Integrations.OrchestratorSpec where

import Array qualified
import Core
import Integration qualified
import NeoBank.Transfer.Core
import NeoBank.Transfer.Integrations.Orchestrator
import Test

spec :: Spec Unit
spec = describe "Transfer Orchestrator" do
  it "emits a Withdraw action when a transfer is requested" \_ -> do
    let transfer = initialState { transferId = Uuid.nil }
    let event = TransferRequested
          { entityId = Uuid.nil
          , sourceAccountId = sourceId
          , destinationAccountId = destId
          , amount = 50
          }
    let outbound = reactTo Orchestrator transfer event
    Array.length (Integration.getActions outbound) |> shouldBe 1

  it "emits nothing for unrelated events" \_ -> do
    let transfer = initialState { transferId = Uuid.nil }
    let event = TransferFailed { entityId = Uuid.nil, reason = "test" }
    let outbound = reactTo Orchestrator transfer event
    Array.length (Integration.getActions outbound) |> shouldBe 0
```

## Recipe 6 — Property-based testing

Property-based tests check that a *property* holds for any input. Great for invariants like "no sequence of valid commands produces a negative balance."

```haskell
module NeoBank.Account.PropertiesSpec where

import Core
import NeoBank.Account.Core
import Test

spec :: Spec Unit
spec = describe "Account invariants" do
  it "balance after replay equals sum of deposits minus sum of withdrawals" \_ -> do
    -- forAll is the property-test runner
    -- genValidEvents is a generator you write for your domain
    Test.forAll genValidEvents \events -> do
      let final = events |> LinkedList.foldl update initialState
      let expectedBalance =
            events
              |> LinkedList.foldr deltaFrom 0
      final.balance |> shouldBe expectedBalance

deltaFrom :: AccountEvent -> Int -> Int
deltaFrom event acc = case event of
  MoneyDeposited { amount } -> acc + amount
  MoneyWithdrawn { amount } -> acc - amount
  _ -> acc
```

Property tests shine when you have invariants that are hard to express with specific examples — "always", "never", or "for any input". The [Testing event-sourced systems](/concepts/testing-event-sourced-systems/) concept page has more examples.

## Running tests in CI

The NeoHaskell template ships with a GitHub Actions workflow that runs `cabal test` on every push. The important gotcha: tests that use PostgreSQL (for the real event store) need a running PostgreSQL instance in CI. The template includes a PostgreSQL service container configuration — use it as a starting point.

For the unit and projection tests, no external services are needed. They run in parallel and finish in seconds.

## What to test

- **Every `decide` function**, with at least one rejection and one acceptance case.
- **Every `update` function**, with a replay test for representative sequences.
- **Every `combine` function** in queries, covering each event type that affects it.
- **Every integration's `reactTo`**, for each event it cares about.
- **One or two end-to-end tests per service** to verify the wiring.
- **Property tests** for your most important invariants (usually balance-like constraints).

Don't test:

- The framework itself. Trust `withService`, `withQuery`, etc. — they're tested in the nhcore test suite.
- Pure serialization. `Json.FromJSON` and `Json.ToJSON` are covered by the compiler; if the types line up, encoding works.
- Trivial `getEntityId` and `getEventEntityIdImpl` functions. They're one-liners.
