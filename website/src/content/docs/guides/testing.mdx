## Unit Testing Command Handlers

{/*
  The core of NeoHaskell testing. Given-When-Then maps directly to the event sourcing model.
  Show the pattern, then show what each part means.
*/}


- The Given-When-Then pattern is not a convention — it's the natural shape of event sourcing
  - **Given**: a list of past events that establish the entity's current state
  - **When**: a command issued against that state
  - **Then**: the events that should be produced (or the error that should be returned)
- No database setup — the event store is just a list in memory during tests
- No mocking — command handlers are pure functions; there's nothing to mock
- Show the test helper functions: `given`, `when`, `then`, `thenError`
- Show a complete test for a happy path (command succeeds, events produced)
- Show a complete test for a rejection path (command fails, error returned)
- Show a test for a state-dependent decision (command succeeds only if entity is in a specific state)
- Explain: the `Given` list is the entire history — you can test any state by constructing the right event sequence
- Note: tests are deterministic — same inputs always produce same outputs

## Testing Queries

## Property-Based Testing

{/*
  The most powerful testing technique for event-sourced systems.
  The invariant "no sequence of valid commands produces invalid state" is testable.
*/}


- Property-based testing generates random inputs and checks that invariants hold
- The key invariant for event-sourced systems: "no sequence of valid commands should produce invalid state"
- Examples of invariants to test:
  - Account balance never goes negative (if overdraft is not allowed)
  - An entity can't be in two mutually exclusive states simultaneously
  - Event count only increases, never decreases
  - Replaying all events always produces the same final state
- NeoHaskell integrates with QuickCheck-style property testing
- Show how to define a generator for valid commands
- Show how to write an invariant as a property
- Note: property tests are slower than unit tests — run them in CI, not on every save

## Test Organization and Naming

{/*
  Conventions that make test suites readable and maintainable.
*/}


- File structure: one test file per command handler, one per query
  - `test/Bank/Account/DepositCommandTest.hs`
  - `test/Bank/Account/BalanceQueryTest.hs`
- Naming convention for test cases: `[context]_[action]_[expected outcome]`
  - Example: `activeAccount_deposit_producesDepositedEvent`
  - Example: `closedAccount_deposit_rejectsWithAccountClosedError`
- Group tests by entity state (Given), not by command type
- Use descriptive event lists in Given — name the helper functions after the state they create
  - `accountWithBalance 100` instead of `[AccountOpened, MoneyDeposited 100]`
- Keep test files short — if a test file is over 100 lines, split it

## Running Tests

{/*
  The practical mechanics of running tests.
*/}


- Run all tests: `neo test`
- Run tests for a specific module: `neo test Bank.Account`
- Run tests matching a pattern: `neo test --match "deposit"`
- Watch mode (re-run on file change): `neo test --watch`
- Test output format: shows Given/When/Then for each failing test — easy to read
- CI integration: `neo test` exits with code 1 on failure — works with any CI system
- Test speed: unit tests run in milliseconds; property tests take a few seconds
