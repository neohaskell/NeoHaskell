The world does not consist of NeoHaskell applications. Your system needs to talk to existing databases, third-party APIs, message brokers, payment processors, email services, and the occasional shell script. This guide shows the patterns.

NeoHaskell calls external connections **integrations**. There are two kinds:

- **Outbound integrations** react to domain events by doing things in the outside world (or emitting follow-up commands). Examples: send an email when a user registers, call a payment processor when an order is placed, publish to Kafka when an invoice is paid.
- **Inbound integrations** take input from the outside world (webhooks, polling, timers) and convert it into commands. Examples: accept Stripe webhooks and emit `PaymentReceived` commands, poll an SFTP drop every hour and emit `FileDelivered` commands, fire a `DailyReport` command on a schedule.

Both are first-class, typed, and compile-checked. You don't use a message queue client; you don't write a webhook controller; you don't set up a cron job. You write an integration and plug it into the application.

## Outbound integrations

An outbound integration listens for events on a specific entity type and produces actions in response. The shape is:

```haskell
instance OutboundIntegration MyIntegration where
  reactTo _config entity event = case event of
    SomeEvent { ... } -> Integration.batch [ ... actions ... ]
    _ -> Integration.none
```

`reactTo` is a pure function. It takes the current entity state, the new event, and returns a collection of actions to execute. Actions might be:

- Emitting a follow-up command (`Integration.emitCommand`).
- Making an HTTP call (via a custom `Action` implementation).
- Writing to a shared database.
- Enqueueing a job in a traditional queue.

### Example: email on signup

Let's build an integration that sends a welcome email when a user registers. We'll assume you have a `Sendgrid` module somewhere that knows how to POST to Sendgrid's API.

```haskell
module MyApp.User.Integrations.WelcomeEmail (
  WelcomeEmail (..),
) where

import Core
import Integration qualified
import MyApp.User.Core (User, UserEvent (..))
import Service.OutboundIntegration.Core (OutboundIntegration (..))
import Sendgrid qualified


data WelcomeEmail = WelcomeEmail


type instance EntityOf WelcomeEmail = User


instance OutboundIntegration WelcomeEmail where
  reactTo _config user event = case event of
    UserRegistered { email } ->
      Integration.batch
        [ Integration.outbound
            Sendgrid.Email
              { to = email
              , templateId = "welcome-template"
              , variables = Map.fromList [("name", user.name)]
              }
        ]

    _ -> Integration.none
```

The `Sendgrid.Email` record is a **config record** — it describes what email to send. `Sendgrid` provides a `ToAction` instance that turns the config into an actual HTTP call. You don't write HTTP-handling code inside the integration; you describe what you want, and the library knows how to do it.

Register the integration with the application:

```haskell
app :: Application
app = Application.new
  |> Application.withEventStore ...
  |> Application.withService userService
  |> Application.withOutbound @WelcomeEmail
```

Done. Every `UserRegistered` event now fires a welcome email, with retry logic and error handling managed by the runtime.

### Example: forwarding to Kafka

If you have a Kafka-based event backbone and want NeoHaskell events to flow into it, you build an outbound integration that publishes to Kafka:

```haskell
instance OutboundIntegration KafkaForwarder where
  reactTo _config _entity event =
    Integration.batch
      [ Integration.outbound
          Kafka.Publish
            { topic = "user-events"
            , key = Json.encodeText event
            , payload = Json.encodeText event
            }
      ]
```

Now every event on the `User` entity is mirrored to Kafka. Downstream services (Spark jobs, analytics, legacy systems) can subscribe to the Kafka topic without knowing anything about NeoHaskell.

### Example: emitting a follow-up command

Integrations can also emit commands back into the system — this is how sagas work. When event A happens, run command B:

```haskell
instance OutboundIntegration OrderToInventory where
  reactTo _config _order event = case event of
    OrderPlaced { items, orderId } ->
      Integration.batch
        [ Integration.outbound
            ReserveItems
              { orderId = orderId
              , items = items
              }
        ]

    _ -> Integration.none
```

`ReserveItems` is a command on the inventory service (marked `InternalTransport` so it's not exposed via HTTP). The integration emits it; the runtime dispatches it to the inventory service. The two services remain decoupled — neither calls the other directly.

## Outbound integrations with lifecycle

Some integrations have state that spans multiple events. For example, a rate limiter that tracks API calls per user, or a batch emitter that buffers events and flushes periodically. For these, use `withOutboundLifecycle`:

```haskell
-- A stateful integration that batches events
batchingIntegration :: OutboundLifecycleIntegration User UserEvent
batchingIntegration = do
  buffer <- ConcurrentVar.new Array.empty
  -- ... construct the integration with access to `buffer` ...
```

Lifecycle integrations are more advanced; see the testbed for a full example.

## Inbound integrations

Inbound integrations take input from the outside world and feed it into NeoHaskell as commands.

### Example: a periodic timer

```haskell
import Integration.Timer qualified as Timer
import MyApp.Report.Commands.GenerateDailyReport (GenerateDailyReport (..))


dailyReportTimer :: Integration.Inbound
dailyReportTimer =
  Timer.every
    Timer.Every
      { interval = Timer.hours 24
      , toCommand = \_ -> GenerateDailyReport {}
      }
```

Every 24 hours, this fires a `GenerateDailyReport` command. The command is dispatched through the same command executor as external commands — so it goes through `decide`, produces events, updates projections, etc. The timer is just an event producer.

Register it:

```haskell
Application.new
  |> ...
  |> Application.withInbound @() (\_ -> dailyReportTimer)
```

### Example: a webhook handler

For webhooks from services like Stripe or Slack, the pattern is:

1. Add a public HTTP route that accepts the webhook payload.
2. Validate the webhook signature.
3. Convert the payload to a command.
4. Dispatch the command.

You can do this inside a custom inbound integration:

```haskell
stripeWebhook :: Integration.Inbound
stripeWebhook = Integration.inbound
  InboundConfig
    { run = \emit -> do
        Http.serve "/webhooks/stripe" \request -> do
          verified <- Stripe.verifySignature request
          case verified of
            Ok event ->
              emit (PaymentReceived { paymentId = event.id, ... })
            Err reason -> do
              Log.warn [fmt|Invalid Stripe signature: #{reason}|]
              Task.yield ()
    }
```

The `emit` callback dispatches the command into the NeoHaskell system.

## Integrations are typed

Every integration has a specific entity type and event type it reacts to. The compiler enforces that:

- The event type matches the entity's event type.
- Commands you emit are registered with the application.
- The transports required by emitted commands are registered.

This is a big deal in practice. In a Kafka-based system, you might have a producer publishing "UserCreated" and a consumer listening for "UserCreated" — but if someone renames the event, the two are silently desynced until a test fails. In NeoHaskell, the sync is a type constraint; the compiler rejects the rename unless both sides are updated.

## Handling failure in integrations

Integrations can fail. An HTTP call might return 503. An email service might be rate-limited. A downstream command might be rejected.

The integration runtime handles transient failures with retries (with exponential backoff). For permanent failures, you handle them inside the integration's `reactTo`:

```haskell
instance OutboundIntegration EmailOrLog where
  reactTo _config user event = case event of
    UserRegistered { email } ->
      Integration.batch
        [ Integration.outbound (Sendgrid.Email { to = email, ... })
            -- If this fails, we also log
        , Integration.outbound
            (LogWarning { message = [fmt|Welcome email pending for #{email}|] })
        ]
    _ -> Integration.none
```

The two actions run in order. If the first fails, the second still runs (unless you explicitly decide otherwise in the action's implementation).

For persistent coordination — "if this fails, retry in 5 minutes" — you use an integration with lifecycle state, or you emit a `RetryLater` command that schedules the retry.

## Integrating with a legacy database

Sometimes you need to read from or write to a database that an old system owns. The cleanest pattern:

1. **Don't write to it from NeoHaskell directly.** Instead, emit a domain event and have an outbound integration call the legacy system's API (even if the "API" is just a SQL stored procedure).
2. **Read from it via a custom data source**, wrapped as an integration. This keeps the database access out of the domain logic.

If you truly must share a database between NeoHaskell and a legacy system, the usual pattern is to use the event store as the source of truth and expose a read replica of projections into the legacy system's schema via a separate integration.

## Integrating with message brokers

If you already use RabbitMQ, NATS, or Kafka:

- **Outbound:** Write an outbound integration that publishes events to the broker.
- **Inbound:** Write an inbound integration that subscribes to topics and emits commands.

Both are a few dozen lines of code. The hardest part is usually handling broker-specific features like consumer groups, offsets, or dead-letter queues — and those are features of the broker, not NeoHaskell.

## Strangler fig pattern

Introducing NeoHaskell into an existing system is usually a strangler fig: the old system keeps running, NeoHaskell takes over one feature at a time.

1. **Pick a bounded feature** — ideally one with a clear input/output boundary.
2. **Write the NeoHaskell service** for that feature.
3. **Route traffic through NeoHaskell** for the new feature.
4. **Use integrations** to keep old and new systems in sync where they overlap.
5. **Expand** as confidence grows.

The typical failure mode is trying to migrate everything at once. Don't. Start with one service, prove the value, expand.
