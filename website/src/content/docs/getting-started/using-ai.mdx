Most software is written with AI assistance now. That is reality, not a problem. NeoHaskell was designed with this reality in mind, and the fit is unusually tight: the language's rules happen to be exactly the rules AI assistants are bad at enforcing on their own.

This page covers what that means in practice — the workflow, the ready-to-paste system prompt, and the reason you can get by with cheaper models in NeoHaskell than you can anywhere else.

## Why NeoHaskell is good for AI-assisted development

<CardGrid>
  <Card title="Wrong code does not compile" icon="approve-check">
    The compiler enforces every event-sourcing contract. If the assistant forgets a case, uses the wrong type, mismatches an event, or misconfigures a service, the build fails with a specific error. You learn within seconds, not in production.
  </Card>
  <Card title="Less code per feature" icon="document">
    Commands, events, entities, and queries are language primitives — there is no glue layer for the assistant to hallucinate. A command handler is one pure function. A query is one pure function. The assistant cannot invent a fictional middleware layer because there is no middleware layer to invent.
  </Card>
  <Card title="Context windows go further" icon="puzzle">
    Small files, a curated standard library, and zero boilerplate mean the assistant can hold entire modules in its working memory. Fewer hallucinations happen when the model can see the types it needs to match.
  </Card>
  <Card title="Cheaper models work" icon="seti:shell">
    You do not need a frontier model to write correct NeoHaskell. The compiler is the validator — a smaller, faster, cheaper model produces output that either compiles (and is structurally correct) or fails a check the assistant itself can fix. Token spend drops accordingly.
  </Card>
</CardGrid>

The important claim is the last one, and it is worth sitting with for a moment. In a loose language, the quality of your AI-assisted code is roughly proportional to the quality of your model — because the model is the only thing enforcing correctness. In NeoHaskell, the compiler is the enforcer, and the model's job shrinks to "propose something plausible." Plausible code that does not satisfy the contract does not ship; it fails the build. The assistant reads the error and tries again. You stop paying for a giant model to catch mistakes a smaller one would miss.

## The AI loop that actually works

Here is the workflow. This is not hypothetical — it is the shape you'll settle into within a day of using an assistant with NeoHaskell:

1. **You describe what you want** in plain language. One command, one query, one feature at a time.
2. **The assistant writes the code** following the NeoHaskell conventions (from the system prompt below).
3. **You run `cabal build`**. If it compiles, the assistant's code is structurally correct — every event handled, every type matched, every service consistent.
4. **If it does not compile**, you paste the error back to the assistant verbatim. Errors in NeoHaskell are specific enough that the assistant usually fixes them in one round.
5. **You run `cabal test`**. Pure function tests are fast and trivial for the assistant to write. They catch the logic bugs the compiler can't.

The loop is tight because the feedback is immediate. The assistant is not guessing about whether its output is correct — the build tells it. You are not reviewing output line by line — the compiler already did.

<Aside type="note" title="The 'regenerate until it works' anti-pattern">
In looser languages, the default AI workflow is "regenerate until it compiles, ship it, hope the tests pass." That only works when the compiler is weak enough that "compiles" doesn't mean much. In NeoHaskell, "compiles" is a meaningful guarantee — it means the structural contracts hold. You do fewer regenerate rounds, you write fewer throwaway tests, and you ship with more confidence.
</Aside>

## Ready-to-paste system prompt

Paste this into your assistant's rules file — Cursor's `.cursorrules`, Claude Code's `CLAUDE.md`, Copilot instructions, OpenCode's `AGENTS.md`, or whatever your tool uses. It contains everything the assistant needs to write idiomatic NeoHaskell on the first try.

````markdown
# NeoHaskell coding rules

You are writing NeoHaskell. NeoHaskell is a strongly-typed, purely functional
language where event sourcing, commands, queries, entities, services, and
integrations are first-class primitives. The compiler enforces every contract.
Follow these rules exactly — they are not style preferences, they are how the
language is used.

## Core rules

- Use `|>` for function application. Never `$`, never nested parentheses.
- Use `do` blocks for anything inside a `Task`. `<-` is `await`,
  `Task.yield` is `return`, `Task.throw` is `throw`.
- Use `case ... of` for pattern matching. Never pattern-match in function
  definitions.
- Use `let` inside `do` blocks for local bindings. Never `let ... in`.
  Never `where`.
- Use `[fmt|Hello #{name}|]` for string interpolation. Never `++` for strings.
- Commas go at the START of each line in records and lists.
- Use descriptive type variable names (`forall input output.`), not single
  letters.
- Use qualified imports (`import Text qualified`). Only `import Core` is
  unqualified.

## Types

- `Task err val` — async/effectful computation with a typed error channel.
  Use this instead of `IO a`.
- `Result err ok` — pure computation that can fail. Use instead of `Either`.
- `Text` — strict UTF-8 text. Use instead of `String`.
- `Array a` — efficient persistent sequence. Use for user-facing collections.
- `Map k v` — immutable key-value map.
- `Maybe a` — optional value (`Nothing` or `Just x`).
- `Uuid` — universal identifier.

## Return values

- `Task.yield value` — return inside a `Task`. Never `pure`, never `return`.
- `Task.throw err` — fail inside a `Task`.
- `Ok value` / `Err reason` — for `Result`.

## Event sourcing primitives (the most important rules)

- Commands have a pure `decide` function:
  `decide :: Command -> Maybe Entity -> RequestContext -> Decision Event`
- `decide` returns one of:
  - `Decider.acceptNew [events]` (stream must not exist)
  - `Decider.acceptExisting [events]` (stream must exist)
  - `Decider.acceptAfter position [events]` (optimistic concurrency)
  - `Decider.reject "reason"` (business rule violation)
- Entities implement `class Entity` with `initialStateImpl` and `updateImpl`.
- Events implement `class Event` with `getEventEntityIdImpl`.
- The `update` function must handle every event variant. This is enforced
  by the compiler — missing cases are compile errors.
- Services are built with `Service.new |> Service.command @Foo |> ...`.
  All commands in a service MUST share the same entity and event type.
- Applications are built with `Application.new |> Application.withEventStore
  ... |> Application.withService ... |> Application.withQuery @Foo |> ...`.

## Forbidden patterns

- Never use `IO a`. Use `Task err val`.
- Never use `Either`. Use `Result`.
- Never use `pure` or `return`. Use `Task.yield`.
- Never use `let ... in`. Use `let` inside `do`, or a top-level helper.
- Never use `where` clauses. Use top-level helpers or `let` inside `do`.
- Never use `++` for strings. Use `[fmt|...|]`.
- Never use point-free style.
- Never use monad transformers (`StateT`, `ReaderT`, etc.).
- Never use code-generation macros except the four provided helpers:
  `command ''Foo`, `deriveQuery ''Q [''E]`, `defineConfig "Name" [...]`,
  and the `[fmt|...|]` string interpolation quoter.
- Never pattern-match in a function head. Use `case`.
- Never import from `base` without prefixing with `Ghc`
  (`import Data.List qualified as GhcList`).

## File layout

- `src/MyApp/Feature/Core.hs` — entity and events.
- `src/MyApp/Feature/Commands/DoThing.hs` — one command per file.
- `src/MyApp/Feature/Queries/FeatureSummary.hs` — one query per file.
- `src/MyApp/Feature/Integrations/Reactor.hs` — one integration per file.
- `src/MyApp/Feature/Service.hs` — assembles commands into a Service.

## Verification

After generating any code:

1. State which compile-time checks will run on it
   (pattern exhaustiveness, event-type consistency, transport registration).
2. State what error messages to expect if something is wrong.
3. If asked to write a command, also write a Given-When-Then test for it.

## The compiler is the validator

If your output compiles with `cabal build`, it is structurally correct —
every event is handled, every type matches, every service is consistent.
If the user pastes an error back to you, read the first line of the error
(the summary) and fix exactly what it names. Do not rewrite unrelated code.
````

This prompt is long but specific. It covers nearly every drift that assistants fall into. Save it, paste it, forget about it.

## Common drift, and how to catch it

Even with the system prompt above, assistants sometimes slip. Here are the most common drifts and the quickest way to spot each one.

### 1. Using `IO` instead of `Task`

Other strongly-typed functional languages use `IO a` for effectful computations. NeoHaskell uses `Task err val`. If the assistant writes:

```haskell
-- Wrong
fetchUser :: Uuid -> IO User
```

Correct it to:

```haskell
-- Right
fetchUser :: Uuid -> Task Text User
```

**How to spot it:** Search for `:: IO ` in any generated file. Replace with `Task <error> <value>`.

### 2. Using `pure` or `return`

```haskell
-- Wrong
fetchUser userId = do
  user <- lookupUser userId
  pure user

-- Right
fetchUser userId = do
  user <- lookupUser userId
  Task.yield user
```

**How to spot it:** Search for `pure ` or `return ` on their own line. Replace with `Task.yield`.

### 3. Using `++` for string concatenation

```haskell
-- Wrong
greet name = "Hello, " ++ name ++ "!"

-- Right
greet name = [fmt|Hello, #{name}!|]
```

**How to spot it:** Search for `++`. For lists, `++` is fine — for text, use `fmt`.

### 4. Using `let ... in`

```haskell
-- Wrong
total = let subtotal = price * qty
            tax = subtotal * 0.1
        in subtotal + tax

-- Right (inside a do block)
total = do
  let subtotal = price * qty
  let tax = subtotal * 0.1
  Task.yield (subtotal + tax)
```

**How to spot it:** Search for `let` followed by `in` on the same or adjacent lines.

### 5. Pattern matching in the function head

```haskell
-- Wrong
handle (Just user) = greet user
handle Nothing = Task.throw "no user"

-- Right
handle maybeUser = case maybeUser of
  Just user -> greet user
  Nothing -> Task.throw "no user"
```

**How to spot it:** Look for multiple lines with the same function name but different argument shapes. Rewrite as a single function with a `case`.

## Reducing token spend

One of the less obvious benefits of NeoHaskell with AI: you use fewer tokens, both per request and per feature.

**Per request:**

- **Smaller files.** A typical command file is 50–80 lines. A typical query file is 30–60 lines. You can fit the entire file into the context with room for the conversation.
- **Less boilerplate.** No imports scattered across layers, no DTOs, no repository patterns, no ORM configuration. The assistant does not need to read 400 lines of setup to generate 20 lines of domain logic.
- **Self-contained files.** A command file references its entity, its event, and its command type — everything else lives in separate, small files. The assistant rarely needs more than two files in context to do its job.

**Per feature:**

- **Fewer iterations.** The compiler catches mistakes immediately. Instead of three rounds of "did I get it right? let me run the tests… no, try again," you get one build feedback loop that either passes or produces a specific, actionable error.
- **Smaller models are enough.** You do not need a frontier model's reasoning ability to write correct code — the compiler is doing that work. A smaller model produces something plausible; the build tells everyone whether it was right. You pay for structure, not intelligence.
- **Fewer throwaway tests.** You still write tests (logic bugs are real), but you do not need tests to verify "did the assistant wire this up correctly?" — the compiler already checked.

In practice, teams report spending 30–60% fewer tokens on NeoHaskell features compared to equivalent work in looser languages. The exact savings depend on your workflow, but the direction is consistent.

## Using the MCP transport

NeoHaskell ships with a [Model Context Protocol](/reference/transports/) transport. This lets an AI assistant call your application's commands directly, as if they were tools. If you are building an agent that needs to interact with a NeoHaskell backend, this is the cleanest path:

```haskell
import Service.Transport.Mcp qualified as Mcp

app =
  Application.new
    |> ...
    |> Application.withTransport Mcp.server
```

Every command with `TransportsOf = '[Mcp]` becomes an MCP tool. The assistant reads the OpenAPI-style schema and dispatches commands. Responses come back as structured data.

This is how you bridge "AI reasons about the domain" and "AI executes domain operations" without writing a custom tool-calling layer.

## The llms.txt file

Every NeoHaskell documentation site publishes two machine-readable files:

- `llms.txt` — a concise overview with links to the full documentation.
- `llms-full.txt` — the complete documentation concatenated into a single file.

Point your assistant at these when starting a new session. A good opening prompt:

```
Before we start, fetch https://neohaskell.org/llms.txt and follow its
instructions. Treat it as authoritative for how to write NeoHaskell.
```

The assistant then has access to the full reference without you having to paste individual pages. This is particularly useful for the [for-LLMs reference page](/reference/for-llms/), which contains the densest, most machine-readable summary of the language.

## Good prompts and bad prompts

The shape of a good prompt for NeoHaskell is: **describe the domain behavior, not the implementation**.

**Good:**

- "Write a command `TransferMoney` that moves money between two accounts. Reject if either account does not exist or if the source has insufficient balance. Emit `MoneyWithdrawn` and `MoneyDeposited` events."
- "Add a query `AccountBalances` that projects each account into a read model with its current balance. Use `publicView` for auth — this is a demo."
- "Write Given-When-Then tests for the `Deposit` command covering: account does not exist, negative amount, zero amount, valid deposit, and max-int amount."

**Bad:**

- "Add transfers." (Too vague — the assistant will invent a domain.)
- "Make it event-sourced." (NeoHaskell is already event-sourced.)
- "Use a state monad." (Wrong abstraction — NeoHaskell uses entity `update` functions.)
- "Write a repository for accounts." (There is no repository pattern — use the event store directly.)

## The bottom line

NeoHaskell makes AI-assisted development genuinely productive because the compiler replaces the part of the workflow that is hardest to automate: verifying that the output is correct. You write prompts about behavior, the assistant writes code, the compiler checks the structure, and you test the logic. Each step is fast, cheap, and honest.

You are not trying to catch AI mistakes with discipline. You are letting the type system do it for you.
