Most developers use AI coding assistants. That's a reality — not a problem to solve. NeoHaskell is specifically designed so that AI-generated code is either correct or it does not compile, which makes it one of the safest languages to let an assistant help you with.

This page gives you:

1. A ready-to-paste system prompt for your assistant.
2. A short list of mistakes AI assistants tend to make with NeoHaskell, and how to prevent them.
3. A workflow for collaborating with an assistant on a NeoHaskell feature.

## Why NeoHaskell works well with AI

Three properties make NeoHaskell unusually compatible with AI coding assistants:

- **Wrong code does not compile.** In loose languages, an assistant can write code that runs for a while and then fails in production. In NeoHaskell, the compiler catches missing pattern matches, wrong types, forgotten event handlers, and misconfigured services — every single time. Incorrect suggestions surface immediately.
- **Less code per feature.** Commands, events, and queries are first-class primitives, so the assistant has less glue to generate and less room to hallucinate. A command handler is a single pure function. A query projection is a pure function. The assistant cannot invent a fictional middleware layer because there is no middleware layer to invent.
- **Context windows go further.** Because NeoHaskell files are small and the standard library is carefully curated, an assistant can hold entire modules in its working context. Fewer hallucinations happen when the model can see the types it needs to match.

## The ready-to-paste system prompt

Paste this into your AI assistant's system prompt / rules file (Cursor `.cursorrules`, Claude Code `CLAUDE.md`, Copilot instructions, etc.):

````markdown
# NeoHaskell coding rules

You are writing NeoHaskell, a Haskell dialect with a curated standard library. Follow these rules exactly. Do not invent patterns from standard Haskell — NeoHaskell deliberately avoids many of them.

## Style

- Use `|>` for function application, never `$` or nested parentheses.
- Use `do` blocks for anything inside a `Task`. Treat `do` like async/await.
- Use `case ... of` for pattern matching. Never pattern-match in function definitions.
- Use `let` inside `do` blocks for local bindings. Never use `let ... in` or `where`.
- Use `[fmt|Hello #{name}|]` for string interpolation. Never use `++` or `<>` for strings.
- Commas go at the START of each line in records and lists.
- Use explicit, descriptive type variable names (`forall input output.`), not single letters.
- Use qualified imports for most modules (`import Text qualified`). Only `import Core` is unqualified by default.

## Types

- `Task err val` instead of `IO a`.
- `Result error value` instead of `Either a b`.
- `Text` instead of `String` or `Data.Text`.
- `Array` instead of `[a]` for collections in user-facing code.
- `Map k v` from `Map`, not `Data.Map`.
- `Maybe a` is the same as in standard Haskell.

## Return values

- `Task.yield value` instead of `pure value` or `return value`.
- `Task.throw error` to fail inside a `Task`.
- `Ok value` / `Err reason` for `Result`.

## Event sourcing primitives

- Commands have a `decide :: Command -> Maybe Entity -> RequestContext -> Decision Event` function.
- `decide` returns `Decider.acceptNew [...]`, `Decider.acceptExisting [...]`, or `Decider.reject "reason"`.
- Entities implement `class Entity` with `initialStateImpl` and `updateImpl`.
- Events implement `class Event` with `getEventEntityIdImpl`.
- Services are assembled with `Service.new |> Service.command @Foo`.
- Applications are assembled with `Application.new |> Application.withEventStore ... |> Application.withService ...`.

## What NOT to do

- Never use `IO a` directly — use `Task err val`.
- Never use `let ... in` or `where` clauses.
- Never use `++` for string concatenation.
- Never use point-free style.
- Never use monad transformers (`StateT`, `ReaderT`, etc.).
- Never use Template Haskell except for the provided helpers (`command ''Foo`, `deriveQuery`, `defineConfig`).
- Never use `pure` or `return` — use `Task.yield`.
- Never pattern-match in a function head.
- Never import from `base` directly without prefixing with `Ghc` (`import Data.List qualified as GhcList`).

## File layout

- `src/MyApp/Feature/Core.hs` — entity and events.
- `src/MyApp/Feature/Commands/DoThing.hs` — one command per file.
- `src/MyApp/Feature/Queries/FeatureSummary.hs` — one query per file.
- `src/MyApp/Feature/Service.hs` — assembles commands into a Service.

## Verification

After generating any code, state which compile-time checks will run on it (pattern exhaustiveness, event-type consistency, transport registration, etc.) and what error messages to expect if something is wrong.
````

This prompt removes most of the common pitfalls. The assistant will still occasionally slip — see the next section.

## Common AI mistakes and how to spot them

Even with a good system prompt, AI assistants drift. Here are the top five mistakes to watch for.

### 1. Generating `IO` instead of `Task`

Standard Haskell uses `IO a` for effectful computations. NeoHaskell uses `Task err val`. If your assistant writes:

```haskell
-- Wrong
fetchUser :: Uuid -> IO User
fetchUser userId = ...
```

Correct it to:

```haskell
-- Right
fetchUser :: Uuid -> Task Text User
fetchUser userId = ...
```

**How to spot it:** Search for `:: IO ` in any generated file. Replace with `Task <error> <value>`.

### 2. Using `pure` or `return`

Standard Haskell uses `pure` or `return` to lift a value into a monad. NeoHaskell uses `Task.yield`:

```haskell
-- Wrong
fetchUser userId = do
  user <- lookupUser userId
  pure user

-- Right
fetchUser userId = do
  user <- lookupUser userId
  Task.yield user
```

**How to spot it:** Search for `pure ` or `return ` on their own line. Replace with `Task.yield`.

### 3. Using `++` for string concatenation

```haskell
-- Wrong
greet name = "Hello, " ++ name ++ "!"

-- Right
greet name = [fmt|Hello, #{name}!|]
```

**How to spot it:** Search for `++`. For lists, `++` is fine — for text, use `fmt`.

### 4. Using `let ... in`

```haskell
-- Wrong
total = let subtotal = price * qty
            tax = subtotal * 0.1
        in subtotal + tax

-- Right (inside a do block)
total = do
  let subtotal = price * qty
  let tax = subtotal * 0.1
  Task.yield (subtotal + tax)
```

**How to spot it:** Search for `let` followed by `in` on the same or adjacent lines.

### 5. Pattern matching in the function head

```haskell
-- Wrong
handle (Just user) = greet user
handle Nothing = Task.throw "no user"

-- Right
handle maybeUser = case maybeUser of
  Just user -> greet user
  Nothing -> Task.throw "no user"
```

**How to spot it:** Look for multiple lines with the same function name but different argument shapes. Rewrite as a single function with a `case`.

## A workflow for collaborating with an assistant

The best workflow is this:

1. **Give the assistant one file at a time.** NeoHaskell files are small and self-contained. Ask for a command, a query, or an entity — not a whole feature.
2. **Let `cabal build` be the arbiter.** If the assistant's code compiles, it is almost certainly structurally correct. If it does not compile, paste the error back to the assistant verbatim. The errors are specific enough that the assistant can usually fix them in one round.
3. **Run the tests.** Even compiled code can be logically wrong. Given-When-Then tests are trivial to write in NeoHaskell — ask the assistant to generate them alongside the command.
4. **Trust the type signature, not the body.** When reviewing assistant-generated code, read the type signature first. If the signature is right, the body is constrained. If the signature is wrong, the whole thing is suspicious.

## What to ask the assistant for

Good prompts for NeoHaskell assistants are specific about the shape of the thing you want. Examples:

- "Write a command `TransferMoney` that moves money between two accounts. It should reject if either account doesn't exist or if the source has insufficient balance. It should emit `MoneyWithdrawn` and `MoneyDeposited` events."
- "Write a query `AccountBalances` that projects every account entity into a read model with its current balance. Use `publicView` for auth since this is a demo."
- "Write a test for the `Deposit` command. Cover: account does not exist, negative amount, zero amount, positive amount, maximum int amount."

Bad prompts tend to produce bad code:

- "Add transfers." (Too vague — the assistant will guess at the domain.)
- "Make it event-sourced." (Meaningless — NeoHaskell is already event-sourced.)
- "Use a state monad." (Wrong abstraction — NeoHaskell uses entity `update` functions.)

## The bottom line

AI assistants make NeoHaskell a joy to work with because the language was designed around a property that happens to match what assistants are bad at: silently wrong code. When the assistant is wrong in NeoHaskell, the compiler tells you. When the assistant is right, you ship.
