You have NeoHaskell installed. Now let's feel what it is like to use. By the end of this page, you will have written a command, emitted an event, replayed it to derive state, and watched the compiler catch a bug you would have shipped in any loose language.

This whole thing takes about ten minutes. No prior Haskell experience required — we will explain every line.

## The setup

We are going to model a tiny piggy bank. It has one feature: **deposit money**. No withdrawals yet, no balance checks, no overdraft. Just the simplest possible version of the single most important idea in NeoHaskell — **state comes from events, not the other way around.**

In CRUD code, depositing money means running an `UPDATE` on a row:

```sql
UPDATE accounts SET balance = balance + 50 WHERE id = '...';
```

The balance changed. The reason is gone. If someone later asks "why is the balance what it is?" the database cannot tell you. The history was never recorded.

In NeoHaskell, you never touch a balance field. You record a fact — *"$50 was deposited"* — and the balance becomes a number you compute from all the facts so far. When someone asks "why?", the facts are the answer.

## Step 1 — Open the template

Make sure you are inside a freshly-cloned NeoHaskell template and in its dev shell:

```bash
cd my-first-app
nix develop
```

The template ships with an empty `PiggyBank` module you can fill in. Create these four files under `src/MyApp/PiggyBank/`:

```text
src/MyApp/PiggyBank/
├── Core.hs              # Entity and events
├── Commands/
│   └── Deposit.hs       # The Deposit command
└── Service.hs           # Wires commands into a service
```

## Step 2 — Define the entity and the event

Open `src/MyApp/PiggyBank/Core.hs` and paste this in:

```haskell
module MyApp.PiggyBank.Core (
  PiggyBank (..),
  PiggyBankEvent (..),
  initialState,
) where

import Core
import Json qualified
import Service.Command.Core (Event (..))


-- | The entity — the thing whose state we care about.
data PiggyBank = PiggyBank
  { bankId :: Uuid
  , balance :: Int
  }
  deriving (Generic, Show)


instance Json.FromJSON PiggyBank
instance Json.ToJSON PiggyBank


-- | The starting point. Every PiggyBank begins empty.
initialState :: PiggyBank
initialState =
  PiggyBank
    { bankId = Uuid.nil
    , balance = 0
    }


-- | NameOf is how NeoHaskell knows what to call this entity on the wire.
type instance NameOf PiggyBank = "PiggyBank"


-- | Entity tells NeoHaskell "this is a stateful thing you can rebuild from events."
instance Entity PiggyBank where
  initialStateImpl = initialState
  updateImpl = update


-- | The event type — every fact that can happen to a piggy bank.
data PiggyBankEvent
  = MoneyDeposited
      { entityId :: Uuid
      , amount :: Int
      }
  deriving (Eq, Show, Generic)


instance Json.FromJSON PiggyBankEvent
instance Json.ToJSON PiggyBankEvent


-- | Every event has to know which entity it belongs to.
type instance EventOf PiggyBank = PiggyBankEvent
type instance EntityOf PiggyBankEvent = PiggyBank


instance Event PiggyBankEvent where
  getEventEntityIdImpl event = case event of
    MoneyDeposited { entityId } -> entityId


-- | The heart of event sourcing: given the current state and an event,
-- produce the new state. This is a pure function. You could call it from ghci.
update :: PiggyBankEvent -> PiggyBank -> PiggyBank
update event bank = case event of
  MoneyDeposited { amount } ->
    bank { balance = bank.balance + amount }
```

There is a lot on that page, but nearly all of it is boilerplate that NeoHaskell's type classes require. The **actual logic** is in two places:

- `initialState` — where the entity starts.
- `update` — how each event changes the state.

That is the whole thing. Your balance is no longer a column you write to — it is **the result of folding `update` over every event that has ever happened to this piggy bank**. Start at zero. Apply `MoneyDeposited { amount = 50 }`. You have 50. Apply another with 25. You have 75. The balance is derived, not stored.

<Aside type="tip" title="Why the record syntax?">
`bank { balance = bank.balance + amount }` is record-update syntax: "give me a new bank that is the same as the old one except with `balance` set to this value." It is not a mutation — `update` returns a *new* `PiggyBank`. The original is untouched. This is why events can be replayed safely.
</Aside>

## Step 3 — Write the command

Open `src/MyApp/PiggyBank/Commands/Deposit.hs` and paste this in:

```haskell
module MyApp.PiggyBank.Commands.Deposit (
  Deposit (..),
  getEntityId,
  decide,
) where

import Core
import Decider qualified
import Json qualified
import Service.Auth (RequestContext)
import Service.Command.Core (TransportsOf)
import Service.CommandExecutor.TH (command)
import Service.Transport.Web (WebTransport)
import MyApp.PiggyBank.Core


-- | The command — a request to deposit money.
data Deposit = Deposit
  { bankId :: Uuid
  , amount :: Int
  }
  deriving (Generic, Show)


instance Json.FromJSON Deposit
instance Json.ToJSON Deposit


-- | Which entity is this command about?
-- Returning Just means "this command targets an existing entity."
-- Returning Nothing would mean "this command creates a new entity."
getEntityId :: Deposit -> Maybe Uuid
getEntityId cmd = Just cmd.bankId


-- | The heart of a command: given the command and the current state,
-- either accept it (producing events) or reject it (with a reason).
-- This is a pure function — no I/O, no surprises.
decide :: Deposit -> Maybe PiggyBank -> RequestContext -> Decision PiggyBankEvent
decide cmd bank _ctx = case bank of
  Nothing ->
    Decider.reject "Piggy bank does not exist"
  Just _ ->
    if cmd.amount <= 0
      then Decider.reject "Deposit amount must be positive"
      else
        Decider.acceptExisting
          [ MoneyDeposited
              { entityId = cmd.bankId
              , amount = cmd.amount
              }
          ]


-- Wire up the command so NeoHaskell knows which entity it acts on,
-- which transports can carry it, and how to register it.
type instance EntityOf Deposit = PiggyBank
type instance TransportsOf Deposit = '[WebTransport]
command ''Deposit
```

Read that `decide` function top to bottom. Notice what it does *not* do:

- It does not open a database connection.
- It does not acquire a lock.
- It does not return the new balance.
- It does not mutate anything.

It takes the command, looks at the current entity, and returns a `Decision`. If the decision is **accept**, it lists the events to record. If the decision is **reject**, it gives a reason. That's it. Running `decide` a million times will produce identical results given identical inputs — it is a pure function. Tests become trivial.

## Step 4 — Wire up the service

Open `src/MyApp/PiggyBank/Service.hs`:

```haskell
module MyApp.PiggyBank.Service (service) where

import Core
import MyApp.PiggyBank.Commands.Deposit (Deposit)
import MyApp.PiggyBank.Core ()
import Service qualified


service :: Service _ _
service =
  Service.new
    |> Service.command @Deposit
```

`Service.new |> Service.command @Deposit` reads as "start a new service, then register the `Deposit` command." You can chain as many commands as you like — the compiler checks that they all share the same entity/event types.

## Step 5 — Plug it into the application

Open `src/App.hs` and replace its contents with this:

```haskell
module App (app) where

import Core
import Service.Application (Application)
import Service.Application qualified as Application
import Service.EventStore.InMemory qualified as InMemory
import Service.Transport.Web qualified as WebTransport
import MyApp.PiggyBank.Service qualified as PiggyBank


app :: Application
app =
  Application.new
    |> Application.withEventStore @() (\_ -> InMemory.new)
    |> Application.withTransport WebTransport.server
    |> Application.withService PiggyBank.service
```

We are using an **in-memory event store** here. It loses all data when the program restarts — perfect for a quickstart, useless for production. Later you will swap it for PostgreSQL with a one-line change.

## Step 6 — Run it

```bash
cabal build
cabal run
```

Your app starts an HTTP server on port 8080. In a second terminal, post a deposit command:

```bash
curl -X POST http://localhost:8080/commands/Deposit \
  -H 'Content-Type: application/json' \
  -d '{"bankId": "00000000-0000-0000-0000-000000000001", "amount": 50}'
```

The response will be a rejection — `"Piggy bank does not exist"`. That is correct. `decide` saw no entity, so it refused. The compiler made you think about that case, and you handled it.

To make this work end-to-end, you would add a `CreatePiggyBank` command (see [the tutorial](/tutorial/) for a full walkthrough). For now, focus on the shape of what just happened:

1. You defined an entity type and an event type.
2. You wrote a pure function (`update`) that folds events into state.
3. You wrote another pure function (`decide`) that converts commands into events.
4. You wired them into a service, plugged the service into an application, and launched an HTTP server.

That's it. No ORM. No repository pattern. No dispatcher to configure. No controller class. No validator library. Four files and a `cabal run`.

## The bug the compiler just caught

Here is the punchline. Change the `PiggyBankEvent` type in `Core.hs` to add a second event, but do not update the `update` function:

```haskell
data PiggyBankEvent
  = MoneyDeposited
      { entityId :: Uuid
      , amount :: Int
      }
  | MoneyWithdrawn
      { entityId :: Uuid
      , amount :: Int
      }
  deriving (Eq, Show, Generic)
```

Save the file and run `cabal build`. You get this:

```text
error: [GHC-18872]
    Non-exhaustive patterns in function update

    In an equation for 'update':
        Patterns of type 'PiggyBankEvent' not matched:
            MoneyWithdrawn{}
```

The compiler caught it. Not a test. Not a code reviewer. Not a production outage. The compiler, at the moment you introduced the mismatch, with the exact file and line number. This is the difference between "event sourcing as a library" and "event sourcing as a language." The library cannot enforce this. The language does.

Fix it by handling the new case in `update`:

```haskell
update :: PiggyBankEvent -> PiggyBank -> PiggyBank
update event bank = case event of
  MoneyDeposited { amount } ->
    bank { balance = bank.balance + amount }
  MoneyWithdrawn { amount } ->
    bank { balance = bank.balance - amount }
```

Rebuild. Green. The contract is satisfied, and you *cannot* forget a case — the compiler will not let you ship broken code.

## What you just learned

- **Events are immutable facts.** You record what happened, never what state "is now."
- **State is derived from events** by a pure `update` function.
- **Commands are intents.** A `decide` function reviews them and either accepts (producing events) or rejects (with a reason).
- **The compiler enforces the contract** between commands, events, entities, and the `update` function.
- **Everything is a pure function** until you hit the transport layer — which means everything is trivially testable.
