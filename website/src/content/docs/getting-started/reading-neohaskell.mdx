You opened a NeoHaskell file for the first time and the syntax felt alien. That's normal. In five minutes, it won't.

NeoHaskell's ancestor language (Haskell) has a reputation for unreadable symbols. NeoHaskell deliberately sheds most of that. What's left is a small, consistent set of building blocks. Here they are, with what each one means in languages you probably already know.

## The dictionary

| You see | In NeoHaskell this means | In JS/TS, think… |
|---|---|---|
| `data Foo = Foo { a :: Int, b :: Text }` | Define a record type with two fields | `type Foo = { a: number, b: string }` |
| `data Result = Ok | Err` | Define a sum type (either Ok or Err) | `type Result = { kind: "ok" } \| { kind: "err" }` |
| `case value of` | Pattern match — exhaustive switch | `switch (value.kind)` — but enforced |
| `let x = 5` | Bind a local value | `const x = 5` |
| `x |> f |> g` | Apply `f` to `x`, then `g` to the result | `g(f(x))` (or `x.then(f).then(g)` conceptually) |
| `func arg1 arg2` | Call `func(arg1, arg2)` | `func(arg1, arg2)` |
| `do { ... }` | Sequence of steps inside a `Task` | `async () => { ... }` |
| `Task.yield x` | Return `x` from an async-style block | `return x` |
| `Task.throw e` | Fail with error `e` | `throw e` |
| `-- comment` | Line comment | `// comment` |
| `{- ... -}` | Block comment | `/* ... */` |

If you memorize that table, you can read any NeoHaskell file. Seriously — that's almost the whole syntax.

## An annotated tour

Here is a real command file. We'll walk through it line by line.

```haskell
module MyApp.Cart.Commands.AddItem (
  AddItem (..),
  decide,
) where
```

**What you see:** A module declaration that exports `AddItem` (with all its fields — that's what `(..)` means) and the `decide` function.

**JS analogue:** `export { AddItem, decide } from './MyApp/Cart/Commands/AddItem';`

---

```haskell
import Core
import Decider qualified
import Service.Transport.Web (WebTransport)
```

**What you see:** Three imports.

- `import Core` imports everything from `Core` into scope, unqualified.
- `import Decider qualified` imports `Decider` but requires you to call things as `Decider.reject`, `Decider.acceptExisting`, etc. — namespace-prefixed.
- `import Service.Transport.Web (WebTransport)` imports only `WebTransport` from that module.

**JS analogue:**
```js
import * from 'core';
import * as Decider from 'decider';
import { WebTransport } from 'service/transport/web';
```

---

```haskell
data AddItem = AddItem
  { cartId :: Uuid
  , stockId :: Uuid
  , quantity :: Int
  }
  deriving (Generic, Show)
```

**What you see:** A record type with three fields. `deriving (Generic, Show)` tells the compiler to auto-generate some boilerplate — you don't have to know what it does yet, just that it's required.

**JS/TS analogue:**
```ts
type AddItem = {
  cartId: string;  // a UUID
  stockId: string;
  quantity: number;
};
```

The field types are on the right side of `::`. Read `cartId :: Uuid` as "cartId has type Uuid."

---

```haskell
decide :: AddItem -> Maybe CartEntity -> RequestContext -> Decision CartEvent
```

**What you see:** A type signature for a function called `decide`.

**How to read it:** "decide takes an `AddItem`, a `Maybe CartEntity`, a `RequestContext`, and returns a `Decision CartEvent`."

Each `->` separates an argument. The last one is the return type. So this function has three inputs and one output.

**TypeScript equivalent:**
```ts
function decide(
  cmd: AddItem,
  cart: CartEntity | null,
  ctx: RequestContext
): Decision<CartEvent>
```

<Aside type="tip" title="Why is 'Maybe CartEntity' different from 'CartEntity | null'?">
`Maybe` is the explicit "this could be missing" type. It forces you to check for `Nothing` before using the value — the compiler won't let you skip that step. It's the same idea as TypeScript's strict null checks, but always on, always enforced.
</Aside>

---

```haskell
decide cmd cart _ctx = case cart of
  Nothing ->
    Decider.reject "Cart not found"
  Just c ->
    if cmd.quantity <= 0
      then Decider.reject "Quantity must be positive"
      else
        Decider.acceptExisting
          [ ItemAdded
              { entityId = c.cartId
              , stockId = cmd.stockId
              , quantity = cmd.quantity
              }
          ]
```

**What you see:** The function body. Arguments are listed after the function name (`cmd`, `cart`, `_ctx` — the underscore means "we don't use this one"). Everything after `=` is the value the function returns.

**Read it like this:**
- `case cart of` — switch on whether `cart` is `Nothing` (null) or `Just c` (not null, with the value bound to `c`).
- `Nothing -> Decider.reject "Cart not found"` — if null, reject with a reason.
- `Just c -> if cmd.quantity <= 0 then ... else ...` — if non-null, check the quantity.
- `Decider.acceptExisting [...]` — accept the command and produce a list of events (one in this case).

**JS/TS equivalent:**
```ts
function decide(cmd, cart, _ctx) {
  if (cart === null) {
    return Decider.reject("Cart not found");
  }
  if (cmd.quantity <= 0) {
    return Decider.reject("Quantity must be positive");
  }
  return Decider.acceptExisting([
    {
      kind: "ItemAdded",
      entityId: cart.cartId,
      stockId: cmd.stockId,
      quantity: cmd.quantity,
    },
  ]);
}
```

The logic is identical. NeoHaskell uses `case`/`if`/records the same way TypeScript does. The difference is that the compiler forces you to handle *every* case of `cart`, every case of the event type downstream, every missing field, every type mismatch — before the program runs.

## The pipe operator

You will see `|>` everywhere. It is not magic. It is just function application, written left-to-right instead of inside-out.

```haskell
x |> f |> g |> h
```

Means exactly the same as:

```haskell
h (g (f x))
```

Which in TypeScript is:

```ts
h(g(f(x)))
```

Use whichever reads better. NeoHaskell prefers `|>` because it reads top-down and lines up cleanly in vertical chains:

```haskell
Application.new
  |> Application.withEventStore config
  |> Application.withTransport WebTransport.server
  |> Application.withService cartService
  |> Application.withQuery @CartSummary
```

Read that as: "Take a new Application. Add an event store. Add a web transport. Add a cart service. Add a query." Each step transforms the value and hands the result to the next step.

**JS equivalent:**
```js
withQuery(
  withService(
    withTransport(
      withEventStore(Application.new, config),
      WebTransport.server
    ),
    cartService
  ),
  CartSummary
);
```

…or with the proposed pipe operator:
```js
Application.new
  |> withEventStore(%, config)
  |> withTransport(%, WebTransport.server)
  |> withService(%, cartService)
  |> withQuery(%, CartSummary);
```

NeoHaskell has had pipes since day one. They are not an afterthought.

## `do` blocks

A `do` block is a sequence of steps inside a `Task`. Each step can be named with `<-` (bind) or just executed for its side effect.

```haskell
handleRequest :: Request -> Task Text Response
handleRequest req = do
  user <- fetchUser req.userId
  cart <- fetchCart user.id
  updatedCart <- addItem cart req.item
  Task.yield (renderResponse updatedCart)
```

**Read it as:**

1. Fetch a user. Bind the result to `user`.
2. Fetch that user's cart. Bind the result to `cart`.
3. Add an item. Bind the result to `updatedCart`.
4. Return a rendered response.

**TypeScript equivalent:**

```ts
async function handleRequest(req: Request): Promise<Response> {
  const user = await fetchUser(req.userId);
  const cart = await fetchCart(user.id);
  const updatedCart = await addItem(cart, req.item);
  return renderResponse(updatedCart);
}
```

`<-` is `await`. `Task.yield` is `return`. `Task.throw` is `throw`. If any step throws, the whole block short-circuits. That is it. If someone tells you a `do` block is a monad and starts drawing arrows, ignore them — for your purposes, it is async/await.

## Record access

NeoHaskell uses **dot notation** for records, the way nearly every mainstream language does:

```haskell
user.name           -- Get the name field
user.address.city   -- Nested access
```

If you had a variable `cart` of type `CartEntity` with a `cartId` field, you write `cart.cartId`. That is all there is to it.

## Record construction

Creating a record uses the field names explicitly:

```haskell
user = User
  { name = "Alice"
  , email = "alice@example.com"
  , age = 30
  }
```

Notice the commas go at the **start** of each line, not the end. That is a NeoHaskell style convention — it makes adding and removing lines cleaner in diffs.

## Record update

Updating some fields of an existing record looks like this:

```haskell
olderUser = user { age = user.age + 1 }
```

This does not mutate `user`. It creates a *new* record with `age` bumped by one, and leaves `user` alone. NeoHaskell has no mutation in your domain code — every "update" is really "build a new version based on the old one."

## Strings

Plain strings use double quotes:

```haskell
greeting = "Hello, world!"
```

For interpolation, use the `[fmt|...|]` quoter:

```haskell
message = [fmt|Hello, #{name}! You have #{count} unread messages.|]
```

`#{expr}` injects any expression into the string. Think of it as JavaScript's backtick template literals, except instead of `${x}` you write `#{x}`.

<Aside type="note" title="Never use ++ for strings">
In other Haskell code you might see `"Hello, " ++ name`. NeoHaskell style is to use `[fmt|Hello, #{name}|]` instead — cleaner, faster, more explicit.
</Aside>

## The vocabulary you need

You now have enough to read any NeoHaskell file. A few more terms to know — they appear so often that knowing them by name helps:

- **`Uuid`** — a universally unique identifier. Just a string on the wire.
- **`Text`** — a string. Use this, not the older `String`.
- **`Int`** — an integer.
- **`Array`** — an ordered collection. `Array.map`, `Array.filter`, etc.
- **`Map`** — a key-value dictionary. `Map.get`, `Map.set`.
- **`Maybe a`** — either `Just a` (a value) or `Nothing` (no value).
- **`Result err ok`** — either `Ok ok` (success) or `Err err` (failure).
- **`Task err ok`** — an async computation that can succeed with `ok` or fail with `err`.
- **`Decision event`** — what `decide` returns: a request to accept (producing events) or reject (with a reason).
