NeoHaskell is a language built around one idea: **commands, events, entities, queries, services, and integrations are first-class language constructs, not patterns you implement in a framework.** You do not wire up a CQRS library. You do not decide whether to use event sourcing. You write a `decide` function, return a `Decision`, and the compiler makes sure every piece of your domain is consistent before the program runs.

The result: a codebase that reads like pseudocode, a type checker that catches entire categories of bugs other languages leave for production, and an architecture that scales from a single-file prototype to a distributed system without rewriting.

## What it looks like

Here is a complete command — the kind of code you will write in production. Read it once, top to bottom. No setup, no preamble, just the thing:

```haskell
module Bank.Commands.DepositMoney (DepositMoney (..), decide) where

import Core
import Decider qualified
import Service.Auth (RequestContext)
import Service.CommandExecutor.TH (command)
import Service.Transport.Web (WebTransport)
import Bank.Account.Core (Account, AccountEvent (..))


data DepositMoney = DepositMoney
  { accountId :: Uuid
  , amount :: Int
  }
  deriving (Generic, Show)


getEntityId :: DepositMoney -> Maybe Uuid
getEntityId cmd = Just cmd.accountId


decide :: DepositMoney -> Maybe Account -> RequestContext -> Decision AccountEvent
decide cmd account _ctx = case account of
  Nothing ->
    Decider.reject "Account does not exist"
  Just _ ->
    if cmd.amount <= 0
      then Decider.reject "Deposit amount must be positive"
      else
        Decider.acceptExisting
          [ MoneyDeposited
              { entityId = cmd.accountId
              , amount = cmd.amount
              }
          ]


type instance EntityOf DepositMoney = Account
type instance TransportsOf DepositMoney = '[WebTransport]
command ''DepositMoney
```

That is the whole command. The compiler will reject this code if:

- `Account` does not have an event type.
- `AccountEvent` does not include `MoneyDeposited`.
- The `update` function for `Account` does not handle `MoneyDeposited`.
- You forgot to return a `Decision`.
- You pattern-matched on a case that does not exist.

You did not write a validator. You did not register this command with a dispatcher. You did not wire it into a test harness. The compiler did. The last line — `command ''DepositMoney` — generates the endpoint, the JSON schema, the CLI parser, and the handler registration. Your domain logic is the entire file.

## The idea in one picture

```text
   Command                     Event                     State
   ───────                    ───────                   ───────
     "I want to       accepted    "This     folded into    "current
      deposit   ────────────────►  happened"  ───────────►  balance
      $50"      rejected                                   is $50"
                     │
                     └──► "reason: account closed"
```

Commands are **requests** ("I want to deposit $50"). The `decide` function reviews them against the current state and either accepts them (producing events) or rejects them (with a reason). Events are **immutable facts** ("$50 was deposited"). State is **derived by folding events** — it is never stored directly.

This is how banks work. It is how accounting works. It is how git works. NeoHaskell just makes it how your code works, and lets the compiler keep everyone honest.

## Built for AI-assisted development

NeoHaskell was designed for a world where AI assistants write most of the code. The language's opinionated structure, small files, and compiler-enforced contracts turn a property other languages envy into a default: **if the assistant's output compiles, it is structurally correct.**

- **The compiler is the validator.** Wrong code does not compile — missing event handlers, mismatched types, broken service boundaries are all caught at build time.
- **Cheaper models are enough.** You do not need a frontier-tier model to produce correct NeoHaskell. A smaller, faster model writes plausible code; the compiler verifies it; you ship.
- **Fewer tokens per feature.** Small files, zero boilerplate, and predictable file layout mean whole modules fit in the assistant's context with room to spare.
- **MCP transport built in.** Expose every command as an [MCP](/reference/transports/) tool — the same command file, zero extra code.

See [**Using AI with NeoHaskell**](/getting-started/using-ai/) for the ready-to-paste system prompt and the workflow that reliably produces correct code on the first build.

## The promise

Every page in this documentation is written to hold up one promise: **you should never feel like you need permission to be here.** Strongly-typed functional languages have a reputation for being intimidating, academic, and closed off to anyone without a category theory textbook. This is not that. If you have ever written a Redux reducer, a REST handler, or a SQL query, you already know how to think in NeoHaskell — you just need the syntax.

Start with the [10-minute quickstart](/getting-started/installation/) if you want to feel the language in your hands. Start with [the tutorial](/tutorial/) if you want to build something real. Start with [Why NeoHaskell?](/concepts/why-neohaskell/) if you want to know what you are signing up for.
