## Every application has two parts

**The shell** — HTTP routing, database connections, file I/O, external APIs. Important, but interchangeable. You can swap your web framework, your database, your queue. The product still works.

**The core** — Your business logic. The rules that make your product yours. *"Premium customers get 15% off orders over $500."* *"If payment fails twice, escalate to support."* *"An account cannot go negative."* This is what you cannot swap out. This is your competitive advantage.

Most languages mix these together. Your pricing rule lives next to your HTTP handler, which lives next to your ORM call. When the rule changes, you are touching three layers and hoping you don't break the plumbing.

**NeoHaskell keeps them separate.** You write the core. The shell plugs in.

## What it looks like

You describe what happens in your domain:

```haskell
data DepositMoney = DepositMoney
  { accountId :: Uuid
  , amount :: Int
  }
```

You describe how to handle it:

```haskell
decide cmd account _ctx = case account of
  Nothing ->
    Decider.reject "Account does not exist"
  Just _ ->
    if cmd.amount <= 0
      then Decider.reject "Deposit amount must be positive"
      else
        Decider.acceptExisting
          [ MoneyDeposited
              { entityId = cmd.accountId
              , amount = cmd.amount
              }
          ]
```

The shell is handled for you:

- HTTP endpoint — generated.
- JSON schema and OpenAPI spec — generated.
- CLI parser — generated.
- Event persistence — automatic.
- State reconstruction — automatic.
- Audit trail — automatic.

You wrote the business rule. NeoHaskell handled the plumbing.

## Why your core deserves its own language

**If it compiles, the logic is structurally correct.** Add a new rule to your domain and forget to handle it somewhere? The compiler tells you exactly where before the program runs. Not a test. Not a code reviewer. The compiler.

**In an era of AI-assisted coding, your core is the last line of defense.** AI is great at writing shell code — routing, boilerplate, CRUD. But when it writes core logic without guardrails, mistakes hide in the part of your system that matters most. NeoHaskell is the guardrail. The compiler validates the assistant's output so you don't have to review every line by hand.

**Your core should be readable.** Business logic that reads like business language. When rules change, you see exactly what to update — because the rules are the code, not a layer buried inside framework abstractions.

**Your core should be auditable.** Every state change is recorded as an immutable fact. Every decision is traceable. Complete history — no audit logger required, no history table to maintain. Ready for debugging, compliance, or feeding into your analytics pipeline.

## The promise

Every page in this documentation is written to hold up one promise: **you should never feel like you need permission to be here.** If you have ever written a switch statement, a REST handler, or a form validation function, you already know how to think in NeoHaskell — you just need the syntax.

Start with the [quickstart](/getting-started/installation/) if you want to feel the language in your hands. Start with [the tutorial](/tutorial/) if you want to build something real. Start with [Why NeoHaskell?](/concepts/why-neohaskell/) if you want to understand the reasoning.
