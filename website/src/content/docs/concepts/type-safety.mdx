You've heard that NeoHaskell is "type safe." Most languages call themselves that. This page is a concrete list of the specific things the compiler catches in NeoHaskell that the compilers of other popular languages let through — and why those things matter.

## The goal of type safety

The goal of type safety is not to make you write type annotations. It is to make **certain classes of bugs impossible to write**. A program that type-checks is not a correct program — the compiler can't read your mind. But a program that type-checks is a program that has certain failure modes ruled out, and those failure modes are the ones that bite hardest in production.

NeoHaskell has a powerful, strongly-typed compiler that catches entire categories of bugs at build time. On top of that, its event-sourcing primitives layer additional type-level constraints, and those are what make the *specific* bugs in event-sourced systems impossible.

## What a strong type system gives you

Before the event-sourcing bits, here's what you get just from having a strong type system at all:

### 1. No null pointer errors

A variable of type `Account` has an `Account` in it. It is never `null`. If the value might be missing, the type is `Maybe Account`, and the compiler forces you to check for `Nothing` before using the value.

```haskell
-- Wrong: won't compile
greet :: Maybe User -> Text
greet user = [fmt|Hello, #{user.name}|]  -- ERROR: user is Maybe User, not User

-- Right
greet :: Maybe User -> Text
greet user = case user of
  Nothing -> "Hello, stranger"
  Just u -> [fmt|Hello, #{u.name}|]
```

### 2. Exhaustive pattern matching

When you `case` on a sum type, you have to handle every case. Forget one, and the compiler tells you:

```haskell
data Color = Red | Green | Blue

classify :: Color -> Text
classify c = case c of
  Red -> "warm"
  Blue -> "cool"
-- Error:
-- Non-exhaustive patterns in function classify
-- Patterns of type 'Color' not matched: Green
```

### 3. Type mismatches at call sites

If a function expects an `Int` and you pass it a `Text`, you don't find out at runtime. You find out at compile time:

```haskell
deposit :: Int -> Account -> Account
deposit amount account = account { balance = account.balance + amount }

-- Wrong: won't compile
deposit "50" myAccount
-- Error: Couldn't match expected type 'Int' with actual type 'String'
```

### 4. Implicit conversions are not a thing

There is no "truthy" value in NeoHaskell. There is no automatic conversion from `Int` to `Text`. If you want one, you spell it out: `toText myInt`. The compiler will not "help" you by converting silently and then producing wrong results later.

## What NeoHaskell's event-sourcing primitives add

These checks are specific to event-sourcing contracts — and they're the bugs that bite hardest in event-sourced systems written in other languages.

### 1. Missing event handlers

When you add a new event variant, the compiler tells you every place that needs to handle it:

```haskell
data AccountEvent
  = AccountOpened { ... }
  | MoneyDeposited { ... }
  | MoneyWithdrawn { ... }  -- new

-- Error:
-- Non-exhaustive patterns in function update
-- Patterns of type 'AccountEvent' not matched: MoneyWithdrawn{}
```

And for `getEventEntityIdImpl`:

```text
Non-exhaustive patterns in function getEventEntityIdImpl
Patterns of type 'AccountEvent' not matched: MoneyWithdrawn{}
```

In Java or C# with annotation-based event-sourcing frameworks, adding an event type is a silent change — the framework doesn't know you forgot to handle it until it encounters one at runtime. NeoHaskell tells you at build time.

### 2. Wrong entity type for a command

Every command declares its entity type via `EntityOf`. If you accidentally use the wrong type, the compiler catches it:

```haskell
-- Command for Account entity
decide :: Deposit -> Maybe Account -> RequestContext -> Decision AccountEvent
decide cmd account _ctx = ...

-- Wrong: emitting a CartEvent from an Account command
Decider.acceptExisting [ ItemAdded { ... } ]
-- Error: Couldn't match type 'CartEvent' with 'AccountEvent'
```

### 3. Mixed event types in a service

Every command in a service must share the same event type. If you mix them, the error is explicit and specific:

```text
Event type mismatch in service definition!

All commands in a service must use the same event type.

Expected event type: AccountEvent
But command 'ProcessPayment' uses event type: PaymentEvent

To fix: Ensure all commands in this service share a common event type.
This typically means all entities should use the same domain event sum type.
```

This is a type-level check implemented with custom error messages. You get a compiler error that reads like a maintainer wrote it.

### 4. Missing transport configuration

Commands declare their transports with `TransportsOf`. If you forget, the compiler notices:

```haskell
-- Missing type instance TransportsOf Deposit = '[...]
-- Error: No instance for (AllKnownSymbols (NamesOf (TransportsOf Deposit)))
```

The error is technical, but it points you at the exact problem.

### 5. Forbidden transport combinations

Commands can be either public (via transports like `WebTransport`) or internal (via `InternalTransport`), but not both:

```haskell
type instance TransportsOf Deposit = '[WebTransport, InternalTransport]

-- Error:
-- Invalid command transport declaration
-- A command cannot declare InternalTransport together with public transports.
-- If you need both public and internal behavior, define two separate commands.
```

### 6. Duplicate transport declarations

If you accidentally list the same public transport twice, the compiler catches it:

```text
Invalid command transport declaration
Public transport list contains repeated transport entries.
Each public transport must appear at most once in TransportsOf.
```

### 7. Query/entity mismatches

Queries declare which entities they depend on via `deriveQuery`. If a `QueryOf` instance claims an entity type that isn't listed, the compiler complains. If a listed entity has no `QueryOf` instance, the compiler complains.

## The errors read like English

NeoHaskell makes heavy use of GHC's custom error messages. Most event-sourcing errors come with:

- A plain-English explanation of what went wrong.
- The specific types involved.
- An instruction on how to fix it.

Compare this to the Java equivalent, which is typically:

```text
RuntimeException: No handler found for event type com.example.AccountEvent$MoneyWithdrawn
  at com.example.framework.dispatcher.ReflectiveEventDispatcher.dispatch(...)
  at ...
```

You get a stack trace at runtime, after the event has already been stored. The NeoHaskell error is at build time, before you even shipped.

## The bugs NeoHaskell does NOT catch

Be honest with yourself about this. The compiler cannot:

- **Read your mind.** If your spec says "minimum balance 10" and you wrote "> 0", the compiler won't know.
- **Verify business rules.** The *structure* of your `decide` function is type-checked, but the *content* is up to you.
- **Detect wrong events.** Recording `MoneyWithdrawn` when the user actually deposited money is a logic bug, not a type bug.
- **Guarantee performance.** Your `combine` function might run in linear time when it should be constant.

For those, you still need tests. The compiler catches the categories of bugs where "being told at build time" is worth more than "being told at runtime"; tests catch the rest.

<Aside type="note" title="The right mental model">
Type safety is a safety net for the mechanical parts of your code. It catches "you forgot a case," "you used the wrong type," "you wired the service incorrectly." Tests are the safety net for the semantic parts — "did I implement the rule correctly?" You need both.
</Aside>

## Type inference

You almost never have to write type annotations in NeoHaskell. The compiler infers them from the shapes of functions and values. When you do add annotations, they act as documentation for future readers (including you, next Tuesday).

The convention in NeoHaskell is to write type signatures for top-level definitions and leave local bindings alone. That gives you a clean API surface while keeping function bodies uncluttered.

```haskell
-- Top-level: explicit signature
decide :: Deposit -> Maybe Account -> RequestContext -> Decision AccountEvent
decide cmd account ctx = do
  let balanceCheck = ...  -- local: no annotation
  let amountCheck = ...   -- local: no annotation
  -- ...
```

## Generic vs. specific type errors

A type error in NeoHaskell can be generic (a fundamental mismatch) or specific (a custom error message from a typeclass with `TypeError`). The specific ones are almost always the easier to read, because they were written by humans for humans. When you hit a generic error, the trick is to read it top to bottom — the first line is the "expected vs. actual" summary, and everything else is context.

A good habit: **read the first line, then scroll back to your code and look at the exact line mentioned**. Don't try to parse the entire error at once. Read the summary, check your code, iterate.
