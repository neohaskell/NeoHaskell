## What You Might Expect

- Type systems in other languages feel like bureaucracy: write the code, then annotate it
- Compiler errors are cryptic: "expected `Foo<Bar<Baz>>`, got `Foo<Bar<Qux>>`"
- The type system catches trivial mistakes (wrong argument order) but misses real bugs (wrong business logic)
- You've learned to work around the type system, not with it
- "Just add a cast" is a common escape hatch

## Why That Breaks

- A cast silences the compiler — but the bug is still there
- Loosely-typed code lets AI-generated code compile and fail silently in production
- Missing a case in a switch statement is a runtime error, not a compile error
- The overdraft bug compiles. The wrong state transition compiles. The missing handler compiles.
- The type system that doesn't catch business logic errors isn't doing its job.

## Type Safety: The Mental Model Shift

- The compiler isn't checking your work — it's **preventing entire categories of bugs from existing**
- Invalid states become *unrepresentable*: if the type system can't express it, the code can't produce it
- Exhaustive pattern matching means no forgotten branches — every case is handled or the code doesn't compile
- The compiler catches the overdraft before the test does, and the test catches it before the customer does
- **The compiler is a collaborator, not an adversary**

### What "Invalid States Unrepresentable" Means

- In CRUD: an account can be `status = "open"` or `status = "closed"` — a string, anything goes
- In NeoHaskell: account status is a type with exactly two constructors: `Open` and `Closed`
- You can't accidentally set `status = "opne"` — that string doesn't exist in the type
- You can't withdraw from a closed account — the type system makes that branch impossible to write
- The type IS the documentation: if the type is right, the domain is right

### Exhaustive Pattern Matching

- When you handle an event or command, you use `case..of` to branch on the type
- If you add a new event type (`DailyLimitSet`), the compiler immediately flags every `case..of` that doesn't handle it
- No forgotten branches. No "I forgot to handle the new case." No silent failures.
- This is the most valuable property for teams: adding a feature forces you to handle it everywhere

<Aside type="note" title="Readable by any developer">
NeoHaskell's type system is powerful, but the code it produces is readable. The testbed's `AddItem.hs` decide function uses `case..of` (switch), `if/then/else`, and `Decider.reject` / `Decider.acceptExisting`. Zero type class constraints visible to the user. A JavaScript developer can read every line. The compiler does the heavy lifting invisibly.
</Aside>

## In NeoBank Terms

- NeoBank's account state is a type: `Open`, `Closed`, `Frozen` — not a string column
- A `Withdraw` command on a `Closed` account is a type error, not a runtime error
- The overdraft check is enforced by the type system: `withdraw` requires `balance >= amount`
- Adding a new account type (savings, checking) requires handling it in every command handler — the compiler tells you where
- The compiler caught the overdraft bug before the test ran. The test confirmed it. The customer never saw it.

## How NeoHaskell Enforces This

- **Missing event handlers**: add a new event type, the compiler flags every unhandled `case..of`
- **Incorrect state transitions**: entity state machines are type-checked — invalid transitions don't compile
- **Type mismatches**: wrong data shapes are compile errors, not runtime surprises
- **Unhandled cases**: exhaustive pattern matching — no forgotten branches
- **Event sourcing contract**: every state change must go through the event log — no silent mutations
