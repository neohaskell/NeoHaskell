## "Why not just [my language] + Claude Code?"

- Most code is AI-generated now (Copilot, Claude, Cursor) — this is the reality
- The discipline to review AI output line-by-line is enormous — and nobody actually does it
- In loosely-typed languages, AI generates code that compiles and fails silently in production
- The fix can't be discipline — it has to be structural
- This wasn't designed for AI. It's a consequence of the constraints:
  - AI-generated code either compiles correctly or doesn't compile at all
  - The compiler catches the classes of errors AI is worst at: missing handlers, wrong state transitions
  - Every state change is an auditable event — AI can't introduce silent side effects
- This works whether a human or an AI wrote the code — the compiler doesn't care
- Less code per feature means AI can reason about entire modules, not fragments — and fewer places for hallucinations to hide
- Less code also means smaller context windows and cheaper API calls

## What NeoHaskell is

- A programming language where event sourcing is a language primitive, not a library
- Commands, events, entities, queries, services, and integrations are first-class constructs
- Strong types + pure functions + compiler-enforced event sourcing contracts
- Core insight: most tools treat correctness as something you add. NeoHaskell treats incorrectness as something you'd have to actively introduce.
- Built on GHC (Haskell's compiler) — proven type system, new developer experience
- Accessible syntax: pipes, qualified imports, do-blocks everywhere — no PhD required. The testbed's `AddItem.hs` command handler reads like pseudocode: `case..of`, `if/then/else`, `Decider.reject`, `Decider.acceptExisting`. Any developer who can write a Redux reducer can write NeoHaskell.
- Start as a monolith, split into microservices when you actually need to — no rearchitecture required. Services are deployment boundaries, not code boundaries.
- **Transport-agnostic**: the same application code runs as a CLI tool or a web API — change one line of configuration. Start with CLI for instant feedback, swap to web when you're ready.

## What the compiler enforces

- **Type mismatches**: wrong data shapes are compile errors, not runtime surprises
- **Missing event handlers**: if you add an event, the compiler demands you handle it everywhere
- **Incorrect state transitions**: entity state machines are type-checked — invalid transitions don't compile
- **Unhandled cases**: exhaustive pattern matching — no forgotten branches
- **Event sourcing contract**: every state change must go through the event log — no silent mutations, no skipping the audit trail
- **Entity boundaries**: entities define consistency boundaries — cross-entity access is explicit, not accidental

## Domain modeling as code

- Services define your services — the language enforces their boundaries
- Events are the domain language — not strings in a database, but typed declarations
- Queries are pure functions from event streams — declarative, testable, composable
- Services and integrations coordinate cross-entity workflows — compiler-enforced
- The type system IS the domain model documentation — if the types are right, the domain is right
- vs. DDD in Java/C#: NeoHaskell makes entities, value objects, and domain events language-level constructs instead of naming conventions

## Configuration as code

- Configuration is defined in code, not YAML — type-checked, self-documenting, impossible to misconfigure at runtime
- A single `defineConfig` block auto-generates: CLI flags, environment variable parsing, documentation strings, defaults, and type validation
- No separate schema files, no runtime type errors, no "oops I set port to a string"
- Config-dependent wiring: event stores, transports, integrations, and auth can all read from the same config — resolved at startup, type-safe throughout

## Testing: event sourcing's secret weapon

- Event sourcing's Given-When-Then is a natural testing grammar
  - Given: a sequence of past events
  - When: a command is issued
  - Then: specific events are produced (or an error)
- No mocking, no database setup, no test fixtures — just events in, events out
- Query tests are pure function tests — trivial to write, fast to run
- Time-travel debugging: replay to any point in history, inspect state
- Property-based testing: "no sequence of valid commands should produce a negative balance"

## How NeoHaskell compares

- vs EventStoreDB: NeoHaskell makes event sourcing a language concern, not an infrastructure choice
- vs Kafka + framework: no boilerplate glue code — commands, events, queries are language primitives
- vs Axon (Java): same architectural ideas, but compiler-enforced instead of convention-enforced. Axon is the closest comparison — NeoHaskell takes Axon's entity + integration model and makes it type-safe
- vs "just use Haskell": accessible syntax, event sourcing built-in, no monad transformer stacks

## What the compiler doesn't catch — be honest

- Wrong business logic (charging $1000 instead of $10) — the compiler can't read your mind
- Incorrect requirements — garbage in, garbage out, even with types
- Bad domain modeling decisions — if you model the wrong entities, the compiler enforces the wrong thing
 Event schema changes — NeoHaskell doesn't do event versioning. Events are immutable facts; if your schema needs to change, you create a new event type. The compiler ensures all handlers are updated for the new type, but the decision of *what* the new event should look like is yours.
- But: because every state change is an explicit event, errors are auditable and replayable — you can find and fix them

## When to adopt — and when to wait

NeoHaskell is honest about where it is today. This isn't "when NeoHaskell is bad" — it's how to make the right timing decision for your context.

- **Simple CRUD with no audit trail** → use something simpler. Event sourcing adds complexity that pays off only when you need history, replay, or audit.
- **You need a mature ecosystem with thousands of packages** → NeoHaskell is pre-release. The core is solid, but the package ecosystem is young.
- **You need production war stories** → NeoHaskell doesn't have Fortune 500 case studies yet. If your decision requires proof from other companies, wait.
- **Hiring at scale** → even though the code is readable by any developer, "NeoHaskell" on a job posting is an unknown. For a team of 2-5, this is fine. For a team of 50, factor in recruiter friction.
- **You need it yesterday** → NeoHaskell is alpha. It's moving fast and will have breaking changes.

**What is NOT a barrier:**
- ~~"My team doesn't know FP"~~ → NeoHaskell code uses `case..of` (switch), `if/then/else`, records, and pipe operators. [See for yourself.](/getting-started/reading-neohaskell/) No monads, no functors, no category theory in your application code.
- ~~"We only build CLI tools"~~ → NeoHaskell has a CLI transport. The same service code runs as a CLI tool or a web API — change one line.
