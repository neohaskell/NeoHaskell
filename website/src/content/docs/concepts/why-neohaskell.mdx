## The core problem

Your application has two parts.

**The shell** is the infrastructure: HTTP routing, database connections, file I/O, queue consumers, external API calls. It is important. It is also interchangeable. You can swap Express for Fastify. PostgreSQL for MySQL. Kafka for RabbitMQ. The product still works.

**The core** is your business logic. The rules that make your product yours. *"Premium customers get 15% off orders over $500 on Fridays."* *"An account cannot go negative."* *"If payment fails twice, escalate to support."* This is what you cannot swap out. This is your competitive advantage.

In most systems, the core and the shell are mixed together. Your pricing rule lives inside an HTTP handler, which lives inside a service class, which calls an ORM, which runs a SQL query. When the rule changes, you are touching four layers and hoping you don't break the plumbing. When something goes wrong, you are debugging across four layers to figure out whether the bug is in the rule, the handler, the service, or the query.

**NeoHaskell separates them.** You write the core — the rules, the decisions, the domain — in a language that gives your business logic the guarantees it deserves. The shell (HTTP, databases, queues, AI transports) plugs in around it.

## Why the core needs guarantees

Your shell can tolerate a bug. A 500 error on a rarely-used endpoint is bad but survivable. A misconfigured route is a deploy away from fixed.

Your core cannot tolerate a bug. A pricing rule that silently gives the wrong discount loses revenue on every transaction. A permission check that misses a case is a security hole. A workflow that skips a step creates data that can never be reconciled.

General-purpose languages treat your core the same as your shell — the same type system, the same error handling, the same "hope the tests catch it" safety net. NeoHaskell gives your core stronger guarantees:

**If it compiles, every case is handled.** Add a new rule to your domain and forget to handle it somewhere? The compiler tells you exactly which files, which functions, which lines need to change. Not a test. Not a code reviewer. The compiler, before the program runs once.

**Every state change is recorded.** Not because you built an audit logger. Because the language is structured so that state can only change through recorded facts. When someone asks "why is this customer's balance $47.32?", the answer is in the event log — every deposit, withdrawal, fee, and adjustment, in order, since the account was opened.

**The rules are the code.** There is no service layer that "interprets" business rules from a config file. There is no framework that "infers" behavior from annotations. Your domain logic is written as pure functions that take input and produce output. You read the function, you read the rule. They are the same thing.

## What this means for AI-assisted development

AI is changing how code gets written. Copilots, coding agents, generated PRs — velocity is increasing. That is great for the shell. Boilerplate writes itself. CRUD endpoints, database migrations, routing glue — an AI assistant handles these in seconds.

But your core? The business rules that define your product? You need guarantees, not suggestions.

NeoHaskell is the guardrail that makes AI-assisted development safe for the part of your system that matters most. The compiler validates every piece of AI-generated core logic. If the assistant forgets a case, introduces a type mismatch, or wires something incorrectly — the build fails with a specific error. The assistant reads the error and fixes it in one round. You do not review lines of code by hand; the compiler reviewed them for you.

The practical consequences:

- **You use cheaper, smaller models.** The compiler does the verification. The model just needs to be plausible, not perfect.
- **You spend fewer tokens per feature.** Core files are small (50–80 lines). Whole modules fit in the assistant's context window.
- **Your prompts get better over time.** When you understand the compiler's feedback, you write prompts that produce correct code on the first build.

And every fact your core records — every event, every decision — is structured data ready for analytics, training, or feeding back into your AI systems. You get that for free because the architecture produces it by default.

## How it works in practice

### If you are building something new

NeoHaskell provides the core language and a pre-built shell. You write your business logic — commands, rules, events — and NeoHaskell provides the HTTP server, the event store, the API documentation, the CLI, and the MCP transport for AI agents. Full application, minimal setup.

```haskell
-- Your core: the business rule
decide cmd account _ctx = case account of
  Nothing ->
    Decider.reject "Account does not exist"
  Just acc ->
    if cmd.amount > acc.balance
      then Decider.reject "Insufficient funds"
      else
        Decider.acceptExisting
          [ MoneyWithdrawn
              { entityId = acc.accountId
              , amount = cmd.amount
              }
          ]
```

The shell is handled:

- HTTP endpoint at `POST /commands/Withdraw` — generated.
- JSON schema and OpenAPI spec — generated.
- Event persisted to PostgreSQL — automatic.
- Account state reconstructed from events — automatic.
- Audit trail — automatic. You did not write it. It was always there.

### If you have an existing system

Your existing system becomes the shell. NeoHaskell becomes the core. They talk via HTTP.

Your Python / Java / Go / JavaScript code sends a command:

```python
response = requests.post(
    "http://core-service/commands/Withdraw",
    json={"accountId": "abc-123", "amount": 50}
)
```

NeoHaskell processes it, applies the rules, records the facts, and returns the result. Your existing code reacts to the result. Clean separation. No rewrite. The complex logic lives in one readable place with compiler guarantees. Everything else stays where it is.

## What goes in the core?

Not everything. Just the parts that matter most:

- **Pricing logic** — "15% off for premium customers on orders over $500."
- **Permission rules** — "Managers can approve expenses under $10k."
- **Workflow decisions** — "If payment fails twice, escalate to support."
- **Domain validation** — "Orders must have at least one item."
- **State machines** — "An order goes from placed to paid to shipped to delivered — never backwards."
- **Financial calculations** — "Balance is the sum of all deposits minus all withdrawals."

These rules are your product. They change when the business changes. They break when the business breaks. They deserve a language that catches mistakes before they reach production.

The HTTP routing, the database connections, the queue consumers, the email sending — that is shell code. It is important, but it does not need the same level of compile-time protection. Use whatever tools you already have for the shell.

## What NeoHaskell does not do

<Aside type="caution" title="Honest limitations">
- **Wrong business logic.** If the requirement says $10 and you wrote $1000, the compiler cannot read your mind.
- **Bad modeling decisions.** If you model the wrong things, the compiler will faithfully enforce the wrong model.
- **Shell concerns.** NeoHaskell does not try to be the best HTTP framework or the best ORM. It provides good-enough shell integrations so you can focus on the core. If you need a specialized shell, bring your own.
</Aside>

## When NeoHaskell is the right call

- Your system has business rules worth protecting — pricing, permissions, workflows, financial logic, compliance.
- You are using AI to write code and you want structural guarantees on the output.
- You care about audit trails, historical debugging, or regulatory compliance.
- You are building something you expect to maintain for years.

## When it is not

- Your system is mostly shell — a thin API over a database with no interesting business rules.
- You need a massive package ecosystem with pre-built integrations for every SaaS product.
- You need production case studies from Fortune 500 companies before you can commit.

## What is NOT a reason to hesitate

- **"My team doesn't know NeoHaskell."** The core reads like business rules. Anyone who has seen a switch statement can review it. One person owns the core; everyone else calls the API.
- **"AI won't know how to write it."** AI writes NeoHaskell effectively because the compiler catches mistakes. The [Using AI](/getting-started/using-ai/) page has a ready-to-paste prompt.
- **"What if NeoHaskell disappears?"** You keep the model. Commands, events, rules — that is architecture, not syntax. Rewrite the core in anything. The design survives the tool.

## The adoption path

1. **Pick one context.** Something painful. Complex rules, frequent bugs. Pricing, permissions, validation.
2. **Model the core.** Commands, events, handlers. A few days with an AI assistant; a couple of weeks without.
3. **Deploy alongside.** Your system calls the new core service. Keep old logic as fallback.
4. **Compare and decide.** Works? Expand. Doesn't? A contained experiment, not a failed migration.

No big-bang rewrite. No commitment. Just an experiment with a clear exit.

## Current status

<Aside type="caution">
NeoHaskell is in active development. The core language — commands, events, entities, queries, services, integrations — is stable enough for production use. There are real applications running today. The custom `.nh` syntax is still in design. Until that ships, you write NeoHaskell in `.hs` files using its curated standard library and style guide.
</Aside>

**Works today:** commands, events, entities, queries, services, integrations, PostgreSQL event store, web transport, CLI transport, MCP transport, declarative configuration, file uploads, OAuth2, OpenAPI spec generation, multi-tenant support.

**In progress:** custom `.nh` syntax, expanded test framework, more transport backends, ecosystem growth.
