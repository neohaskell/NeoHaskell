## The problem

You describe a feature. Your AI assistant writes the code. It looks right. It compiles. It passes a few tests. You ship it.

Three weeks later, a customer reports that their account balance is wrong. You dig in and discover the assistant wrote a function that silently skips a case your domain should have handled. The code compiled because nothing in your language required it to handle that case. The tests passed because nobody thought to write a test for a scenario nobody knew existed. The bug has been in production for three weeks, and now you are explaining it to your VP.

This is the problem NeoHaskell solves. Not with more tests. Not with better prompts. Not with a smarter model. **With a compiler that refuses to build code where cases are missing, contracts are broken, or state changes go unrecorded.**

## The pitch

**NeoHaskell is the language where incorrect code does not compile — so you ship faster, because you stop spending time finding bugs that the compiler already found.**

That is the whole thing. Everything else on this page is a consequence.

## What that means in practice

### You stop reviewing AI output line by line

In most languages, the only way to trust AI-generated code is to read every line. The model is the sole enforcer of correctness. If you miss something, production catches it for you.

In NeoHaskell, the compiler is the enforcer. If the assistant forgets to handle a case, the build fails. If it uses the wrong type, the build fails. If it wires a feature incorrectly, the build fails — with a specific message that says what is wrong and where. You review architecture and intent. The compiler reviews structure and consistency.

The result: you move faster because you delegate the tedious verification to a tool that never gets tired and never skips a line.

### You use cheaper, smaller AI models

When the compiler validates structure, the model's job shrinks to "propose something plausible." It does not need to be brilliant — it needs to be close enough that the compiler can tell it where it's off. A smaller, faster, cheaper model writes NeoHaskell as effectively as a frontier model writes Python, because the expensive part (catching structural errors) is handled by the build, not the model.

The structural reason is straightforward: fewer files to read, fewer lines per file, and a build step that replaces the "did the model get it right?" review. Token spend scales with how much context the model needs and how many rounds it takes to converge. Both shrink when the compiler handles verification.

### Adding a feature cannot silently break an existing one

When you add a new capability to your system — a new kind of transaction, a new status, a new field — the compiler immediately tells you every place in the codebase that needs to account for it. Not "might need to change." Not "you should check." It tells you exactly which files, exactly which functions, exactly which lines do not handle the new case yet. You fix them, rebuild, and the guarantee holds again.

In a language without this property, adding a feature is an act of faith. You add the thing, you write some tests, you ship, and you hope you didn't forget a handler somewhere in a file you haven't opened in months. In NeoHaskell, the compiler opened every file for you.

### You get a complete audit trail without building one

Every state change in a NeoHaskell application is recorded as an immutable fact. Not because you wrote an audit logger. Not because you added a `history` table. Because the language is structured so that state can only change through recorded facts. The audit trail is the default, not a feature you layer on.

When a customer asks "why is my balance $47.32?", the answer is in the event log: a list of every deposit, withdrawal, transfer, and fee, in order, with timestamps, since the account was opened. You did not write a single line of audit code. The architecture gave it to you.

### Your domain is the code

In most systems, the architecture lives in a diagram, the business rules live in a service layer, the data lives in a database, and the wiring lives in configuration files. When any of these drift apart — and they always drift — you get bugs.

In NeoHaskell, you describe your domain once. The rules, the state transitions, the boundaries between systems, the way data flows — all of it is expressed in the same typed code. There is no diagram to keep in sync. There is no YAML to misconfigure. The code is the architecture, and the compiler ensures it is consistent.

### You start simple and scale without rewriting

A NeoHaskell application starts as a single process running against an in-memory store. When you need persistence, you change one line to use PostgreSQL. When you need to split a service out, you move its module to a separate deployment — same code, different wiring. When you need an HTTP API, a CLI tool, or an MCP integration for AI agents, you register a transport. Your domain code does not change.

There is no "we need to rewrite for scale" moment, because the architecture boundaries are in the types from day one. You just move the deployment line.

## Built for AI-assisted development

Everything above compounds when an AI assistant is doing the writing. Here is why.

**The feedback loop is immediate.** The assistant writes code. You run the build. If it fails, the error is specific enough that the assistant fixes it in one round — usually by adding a single missing case or import. No guessing, no stack-trace archaeology, no "run the tests and see what happens."

**Context windows go further.** NeoHaskell files are small. A command is 50–80 lines. A query is 30–60. Entire modules fit in the assistant's working context. When the model can see all the types it needs to satisfy, it hallucinates less.

**The file layout is predictable.** Every command, every query, every service, every integration has a canonical location. The assistant never has to guess where a file goes. It puts the file in the right place because there is only one right place.

**Your prompts become better over time.** When you understand the compiler's feedback, you start writing prompts that preempt errors: "add the event and make sure the update function handles it." That is a prompt that produces correct code on the first try. You earn that instinct by working with the compiler, and once you have it, the assistant becomes dramatically faster.

See [Using AI with NeoHaskell](/getting-started/using-ai/) for the ready-to-paste system prompt, the workflow, and the specific patterns to watch for.

## What it feels like

Here is a real handler from a production codebase. Read it top to bottom:

```haskell
decide cmd entity _ctx = case entity of
  Nothing ->
    Decider.reject "Cart not found"
  Just cart ->
    if cmd.quantity <= 0
      then Decider.reject "Quantity must be positive"
      else
        Decider.acceptExisting
          [ ItemAdded
              { entityId = cart.cartId
              , stockId = cmd.stockId
              , quantity = cmd.quantity
              }
          ]
```

That is a `case` statement, an `if`, a record, and a function call. No academic concepts. No framework magic. No annotations. The business rule ("quantity must be positive") is right there on line four. The state change ("item was added") is right there on line seven. An AI assistant can write this. A junior developer can read this. The compiler guarantees that every case is handled.

## What the compiler does not catch

<Aside type="caution" title="Honest limitations">
- **Wrong business logic.** If the requirement says $10 and you wrote $1000, the compiler cannot help.
- **Bad modeling decisions.** If you model the wrong things, the compiler will faithfully enforce the wrong model.
- **Event schema changes.** Events are immutable facts. If the shape needs to change, you create a new event type. The compiler ensures every handler is updated for the new type, but the decision of *what* the new event looks like is yours.
</Aside>

The compiler guarantees that your code is consistent with your types. If your types are wrong, your code will be consistently wrong. This is still a massive improvement over "your code might be wrong and nobody will notice until production," but it is not omniscience.

## When NeoHaskell is the right call

- Your system has meaningful state transitions — orders, accounts, bookings, workflows, user lifecycles.
- You are using AI to write code and you want structural guarantees on the output.
- You care about audit trails, historical debugging, or regulatory compliance.
- You are a small-to-medium team (2–15 engineers) building something you expect to maintain for years.

## When it is not

- Your system is a simple CRUD app with no interesting state transitions.
- You need a massive package ecosystem with pre-built integrations for every SaaS product.
- You are hiring 50+ engineers and need a universally recognized language on the job posting.
- You need production case studies from Fortune 500 companies before you can commit.

## What is NOT a reason to hesitate

- **"My team doesn't know functional programming."** NeoHaskell uses records, `if/then/else`, pattern matching, and pipes. Anyone who has written a Redux reducer or a switch statement can read and write this code.
- **"It must be slow."** NeoHaskell compiles to native code with throughput targets in the tens of thousands of requests per second.
- **"We only build CLI tools."** NeoHaskell has a CLI transport. Same domain code, different delivery.
- **"We need to integrate with existing systems."** Inbound and outbound integrations are first-class.
- **"AI won't know how to write it."** AI assistants write NeoHaskell effectively because the compiler catches their mistakes. The [Using AI](/getting-started/using-ai/) page has a ready-to-paste prompt.

## Current status

<Aside type="caution">
NeoHaskell is in active development. The service layer, event store, commands, queries, and integrations are stable enough for production use — there are real applications running today. The custom `.nh` syntax is still in design. Until that ships, you write NeoHaskell in `.hs` files using its curated standard library and style guide.
</Aside>

**Works today:** commands, events, entities, queries, services, integrations, PostgreSQL event store, web transport, CLI transport, MCP transport, declarative configuration, file uploads, OAuth2, OpenAPI spec generation, multi-tenant support.

**In progress:** custom `.nh` syntax, expanded test framework, more transport backends, ecosystem growth.
