Most software is built on the assumption that the interesting thing worth storing is the current state. The account has a balance. The order has a status. The user has a role. We write these current values to a row in a database, and we UPDATE them when things change.

This is a convenient fiction that has a real cost. This page is about the cost, and about what you get when you stop paying it.

## The fiction

"The balance is $1,247.32" is not a fact about the world. It is a *summary* of facts about the world. The actual facts are things like:

- On March 3rd, $500 was deposited.
- On March 5th, $100 was withdrawn.
- On March 10th, a $2 fee was charged.
- On March 15th, $850 was deposited.
- ... and so on, fifty more entries.

$1,247.32 is the result of summing all of that. It is derived. It is not primary.

When you store `balance = 1247.32` in a database, you are storing the summary and throwing away the facts. This is efficient but lossy. You saved some bytes and you lost every question whose answer requires the history.

## Questions that need the history

Here are questions a `balance` column cannot answer:

1. **What was the balance on March 7th?**
2. **How many deposits happened in March?**
3. **What was the largest single withdrawal?**
4. **Who initiated the $100 withdrawal on March 5th?**
5. **If we reverse the $2 fee (because it was charged in error), what should the balance have been on March 11th?**
6. **If our fee structure had been 1% instead of 2%, what would the ending balance be?**
7. **When did the balance first exceed $1000?**
8. **How many times did the balance briefly dip below $100?**

Every one of these questions is important at some point. Regulatory audits ask them. Customer support needs them. Product managers want them for analytics. Bug fixes depend on them. Incident reviews require them.

In CRUD, every question becomes a political decision: *should we log this? Should we add a history table? How far back?* You end up with audit tables, history tables, journals, logs, and a hundred little ad-hoc attempts to recover the information you threw away.

In event sourcing, **every question is answered by the same mechanism**: fold the events. The events are right there. Always. You don't need to decide what to log; the log is the primary store.

## The cost of storing state

People assume that storing state is cheaper than storing events. It's not, once you factor in the cost of losing information. Here is a rough accounting of what CRUD's "cheaper" storage actually costs you over time:

| Cost | Reason |
|---|---|
| **Audit tables** | You need to reconstruct history *somehow*, so you build a side channel. |
| **Migration scripts** | Changing the schema means rewriting existing rows, which means writing and testing migrations. |
| **Race conditions** | Multiple concurrent updates to the same row require locking or optimistic concurrency, implemented by hand. |
| **Lost reasons** | "Why did the balance change?" Nobody knows. |
| **Untestable edge cases** | Bugs that depend on specific state histories are impossible to reproduce. |
| **Limited analytics** | "How did usage change over time?" Requires you to have been recording something. |
| **Denormalized projections done by hand** | Any specialized view (dashboards, reports) becomes a separate table you maintain by hand, often with consistency bugs. |

In event sourcing, each of these becomes either free or a natural consequence of the architecture:

| Benefit | Why |
|---|---|
| **Audit for free** | Events *are* the audit log. |
| **Migrations are additive** | New event types, no table rewrites. |
| **Concurrency is per-stream** | Events are atomic at the stream level; optimistic concurrency is built in. |
| **Reasons are in the events** | Each event has context (who, when, why). |
| **Reproducible bugs** | Bugs that depend on history can be replayed from the event log. |
| **Analytics for free** | Fold events however you want, whenever you want. |
| **Projections are just code** | Define a `combine` function; the runtime maintains it. |

## "But isn't storing state simpler?"

It is simpler *to write a row*. It is not simpler *to own the system*. You pay the complexity cost later, in the form of audit tables, migrations, reconciliation scripts, and the endless explanations to stakeholders about why the data doesn't match reality.

Event sourcing moves the complexity *forward*. Instead of a simple write followed by a year of ad-hoc reconstruction, you write events that are slightly more structured, and from then on every question is answerable.

## "But isn't storing state faster?"

Reading state from a denormalized row is faster than folding a thousand events, yes. This is why event-sourced systems build **projections** (NeoHaskell calls them queries). A projection is a denormalized read model that is updated incrementally as events arrive. Reading a projection is a single-row lookup — exactly the same cost as CRUD.

The difference is that in CRUD, the denormalized row is the *only* store. In event sourcing, the denormalized row is a *cache* over the event log. If the cache gets stale or corrupted, you rebuild it by replaying the events. The source of truth is never in doubt.

NeoHaskell also caches entity state via snapshot caches — so even when you fetch an entity via the event store, you don't replay events from the beginning. You start from the most recent snapshot and replay only what's come since.

## "What about write performance?"

Appending an event is cheaper than `UPDATE` — it's a pure append, no row rewrite, no index update on the updated row. On PostgreSQL with the NeoHaskell event store, you can sustain tens of thousands of events per second per stream, and many more across streams.

The bottleneck is usually not the event store. It's the projections that update on every event. And projections are fixable (you batch, you denormalize, you cache). The event store itself is typically nowhere near its limits.

## The philosophical part

Here's the deeper idea. **State is not a thing. State is a view.** Different readers want different views. The CRUD mistake is to pretend there's one canonical "current state" that everyone agrees on. There isn't.

- The accountant's view is the P&L.
- The customer service rep's view is the transaction history.
- The fraud team's view is the list of suspicious patterns.
- The machine learning team's view is the feature vector.
- The customer's view is the current balance.

In CRUD you pick one view, call it "state", and everyone else has to synthesize their view from it — with inevitable drift and inconsistency. In event sourcing, every view is built from the same events, independently, and they cannot diverge from the source of truth.

**Events are the thing. State is whatever the reader needs.**

## The practical mechanics in NeoHaskell

In NeoHaskell, the separation is explicit:

- **Events** live in the event store. Append-only. Immutable.
- **Entities** are the in-memory representation, reconstructed from events via an `update` function. This is what commands see when they make decisions.
- **Queries** are read models derived from entity state, maintained incrementally via `combine`. This is what readers see.

Each layer does exactly one thing. Events are the truth. Entities answer "what is the current state of this one thing right now?" Queries answer "what does this view of the data look like?" You can add new queries without touching the existing ones. You can delete a query and rebuild it from the events. You can migrate a projection's shape without touching the event store.

## The short version

- Storing state means storing a summary and throwing away the facts.
- Events are the facts. State is the summary.
- CRUD systems pay a hidden tax in audit tables, migrations, and lost answers.
- Event-sourced systems get audit, replay, and analytics for free.
- Performance is not a problem — projections and entity caching give you the read speed of CRUD with the fidelity of event sourcing.
