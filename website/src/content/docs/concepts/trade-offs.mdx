Event sourcing is not a universal win. It's an architectural trade-off with real costs, and there are domains where those costs aren't worth the benefits. This page is about being honest about both sides of the ledger.

If you've read the rest of the concept pages, you've seen the case *for* event sourcing in detail. Now let's talk about the case *against* it — and how to tell when you're in a domain where event sourcing is a bad fit.

## The costs

### 1. Slightly more code per feature

A simple CRUD create-read-update-delete feature is four endpoints and a database table. In event sourcing, the same feature is four commands, a handful of events, an entity, a query, and possibly an integration. More files, more concepts, more ceremony.

For features where you truly only need CRUD — a blog post, a settings panel, a lookup table — this extra structure is overhead. It's the difference between "I wrote a SELECT and it worked" and "I wrote a query projection that combines two entity types and I have to think about `NoOp` vs `Update`." For simple cases, the CRUD version is genuinely simpler.

The payoff only kicks in when your domain has meaningful state transitions — orders that move through phases, accounts that accrue history, users that evolve over time. If your data is mostly static lookup tables, event sourcing is overkill.

### 2. Schema evolution is different, and sometimes harder

The [schema evolution](/concepts/schema-evolution/) page covered this in detail. In short: you don't modify old events, you add new variants. This is honest but sometimes frustrating — you end up with a `UserRegisteredV1` and a `UserRegisteredV2` in the same sum type, and you have to handle both in every handler.

For teams used to "alter table, ship migration, move on," this feels weird. For teams that have been burned by migrations that silently corrupt data, it feels liberating. Your mileage depends on your instincts about honesty vs. expedience.

### 3. Eventual consistency on the read side

When a command is accepted, the events are persisted immediately. But the *projections* (read models) are updated asynchronously — usually within milliseconds, but not instantly. If your UI tries to read the projection immediately after submitting a command, there's a small window where it might not see the change yet.

For most applications this is fine — you just re-fetch after a short delay, or you update the UI optimistically. For applications where "read your writes" consistency is absolutely critical at sub-millisecond latency (some trading systems, for instance), you need a pattern like command handlers returning the new entity state alongside the events.

### 4. A learning curve

Event sourcing is unfamiliar to most developers. Onboarding a new engineer to an event-sourced codebase takes longer than onboarding them to a CRUD codebase, at least in terms of pure time-to-first-PR. They have to learn commands, events, entities, queries, integrations, and how they all compose.

The counter-argument is that once they've learned it, they're *faster* than in CRUD — because the architecture gives them a clear place for every piece of logic and the compiler catches most mistakes. But the ramp-up is real, and you should account for it when deciding whether to introduce event sourcing to a team.

### 5. Storage grows monotonically

Events are append-only. You never delete them (except for redaction, which is a special case). Over time, your event store grows. For a typical business system this is negligible — events are small, disks are cheap. For a system that generates millions of events per minute (IoT, analytics pipelines), the cost becomes real.

Mitigations exist — snapshots, cold storage, compaction for events that no longer affect any projection — but they're additional complexity you'd rather not have. If your domain generates events at a rate where storage growth matters, you need to plan for it.

### 6. Some kinds of queries are harder

"Bulk update all invoices matching a condition" is trivial in SQL and unnatural in event sourcing. You don't bulk-update anything; you iterate and emit individual commands. This is usually the *right* thing — bulk updates in CRUD are the source of many bugs — but when you're trying to do a one-off data fix, it's more work.

Similarly, ad-hoc analytical queries that cross many entities are awkward through the query pipeline. You usually export events to a separate analytical store (ClickHouse, DuckDB, BigQuery) for that kind of work.

### 7. A young ecosystem

NeoHaskell is pre-release. The core is solid, but the ecosystem is small. If you want a pre-built library for every third-party service on earth, you won't find it. You'll write integrations yourself, or you'll use the underlying Haskell ecosystem, which is larger but more idiosyncratic.

Expect to write more things from scratch than you would in Node.js or Python. Also expect to ship more features per line of code, because the things you write are smaller.

## The benefits

To be fair to both sides, here's the list of benefits — most of which were covered throughout the concept pages.

- **Audit trail for free.** Every action is recorded, in order, immutably.
- **Replayability.** You can reconstruct state at any past point in time.
- **Testability.** Pure functions all the way down; no mocks.
- **Type-level safety.** The compiler enforces the event-sourcing contract.
- **Scalability.** Event stores scale horizontally; projections scale independently.
- **Decoupled read and write.** Queries can evolve without touching commands.
- **Analytics-ready.** Your data is already in the shape analytics systems want.
- **No lossy state.** The full history of every change is preserved.

## When to use event sourcing

Strong signals that event sourcing is a good fit:

- **Your domain has meaningful state transitions.** Orders, accounts, bookings, workflows, user lifecycles.
- **History matters.** Regulatory compliance, audit, disputes, user-facing activity logs.
- **Read patterns are diverse.** Different views for different users/teams/screens.
- **You expect the system to live a long time.** Event-sourced systems age better than CRUD.
- **Debuggability is valuable.** You want to reconstruct "what happened" when bugs surface in production.

## When NOT to use event sourcing

Strong signals that you should pick something else:

- **Your domain is pure CRUD.** Settings panels, lookup tables, simple forms with no rules.
- **You need hard "read your writes" consistency at microsecond latency.** Trading, real-time control systems.
- **Your data is inherently transient.** Caches, session stores, pub/sub brokers.
- **You're building a data processing pipeline.** ETL, machine learning feature engineering — different tools exist for this.
- **The team has no appetite for learning.** Introducing event sourcing to a team that wants to ship fast and isn't interested in the mental model is a recipe for failure.

## When to use event sourcing for *part* of a system

You don't have to commit all at once. A very common pattern is to event-source the *core domain* (the part where history matters most) and use CRUD for the periphery (settings, admin panels, content management).

NeoHaskell supports this: services are independent, and you can have some services that are event-sourced and others that are just HTTP handlers talking to a relational database. The compiler doesn't care. The event-sourced services get all the benefits; the CRUD services get simplicity.

## The honest answer

Event sourcing is the right tool for **domains where you would otherwise be fighting to reconstruct history from an audit log**. If you're already building an audit log by hand, if you're already writing migration scripts to backfill data, if you're already losing nights of sleep over "what did the balance look like last Tuesday?" — event sourcing pays off quickly.

If your system genuinely doesn't have interesting history and never will, the CRUD route is fine. Don't adopt event sourcing because it sounds cool. Adopt it because you have a problem it solves.
