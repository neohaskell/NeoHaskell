## Who This Page Is For

This page is for architects and principal engineers evaluating NeoHaskell for production use. It assumes familiarity with event sourcing concepts (EventStoreDB, Kafka, Axon) and focuses on the technical properties that matter for adoption decisions.

If you're a developer getting started, begin with [Why NeoHaskell?](/concepts/why-neohaskell) and the [Tutorial](/tutorial/).

## Transaction Guarantees and Consistency Model

### Within a Service

- **Strong consistency**: within a single entity, all operations are serialized — no concurrent writes to the same entity
- **Optimistic concurrency**: the event store uses version numbers to detect concurrent writes — conflicts are detected, not silently lost
- **Atomicity**: a command either produces all its events or none — partial writes don't happen
- **Durability**: events are written to the event store before the command returns — no in-memory-only state

### Across Services

- **Eventual consistency**: cross-context operations communicate via events — consistency is eventual, not immediate
- **At-least-once delivery**: events are delivered at least once — handlers must be idempotent
- **Ordering**: events within a single entity stream are ordered; across streams, ordering is not guaranteed
- **Compensation**: the integration pattern handles cross-service failures — compensation events undo partial operations

### Comparison with Other ES Tools

| Property | NeoHaskell | EventStoreDB | Kafka + manual ES | Axon |
|----------|-----------|-------------|------------------|------|
| Consistency model | Strong within entity, eventual across services | Strong within stream | Depends on implementation | Strong within entity |
| Concurrency control | Optimistic (version numbers) | Optimistic (expected version) | Manual | Optimistic |
| Atomicity | Per-command | Per-stream append | Per-partition | Per-entity |
| Schema enforcement | Compiler (types) | None (schema registry optional) | Schema registry | Convention |

<Aside type="caution" title="Honest comparison">
NeoHaskell's event store is not yet battle-tested at the scale of EventStoreDB or Kafka. The consistency model is sound, but production war stories don't exist yet. If your decision requires proof from other companies at scale, wait.
</Aside>

### Event Store Throughput

- **Write throughput**: append-only writes are fast — no locking, no index updates on write
- **Read throughput**: event stream reads are sequential — cache-friendly, predictable latency
- **Update rebuild**: depends on event stream size and update complexity — parallelizable
- **Current state**: derived from events — cached updates give CRUD-like read performance

### Memory Profile

- **Event store**: append-only — grows linearly with event count
- **Updates**: in-memory queries — size depends on update complexity
- **Entity state**: derived on demand — no persistent in-memory state required
- **Snapshots**: optional — reduce replay time for large event streams at the cost of storage

### Latency Characteristics

- **Command processing**: validate → append event → return — typically sub-millisecond for simple commands
- **Query processing**: fold over events → return query — depends on event stream size and caching
- **Cross-context events**: eventual — latency depends on the event delivery mechanism
- **Update rebuild**: batch operation — not on the critical path for user requests

## Scaling Patterns

### Single-Node Scaling

- NeoHaskell compiles to native code — single-node performance is competitive with Go and Java
- GHC's runtime provides lightweight green threads — millions of concurrent tasks on a single node
- Event store reads are fully concurrent — no locking for reads
- Event store writes are serialized per entity — no contention across entities

### Horizontal Scaling

- **Stateless command handlers**: command handlers are pure functions — trivially horizontally scalable
- **Shared event store**: multiple instances share the same event store — PostgreSQL handles concurrency
- **Update scaling**: updates can be rebuilt on any node — no shared state required
- **Service scaling**: each service can be deployed independently — scale what needs scaling

### Sharding Strategies

- **Entity-based sharding**: route commands to the node responsible for that entity — reduces cross-node coordination
- **Service-based sharding**: each service on its own node — natural isolation
- **Event stream partitioning**: partition the event store by entity ID — parallel reads across partitions

## Integration with Non-NeoHaskell Services

### Outbound Integration

- **HTTP**: NeoHaskell services expose HTTP endpoints — any client can call them
- **Events**: NeoHaskell services can publish events to external message brokers (Kafka, RabbitMQ)
- **Webhooks**: NeoHaskell services can call external webhooks on event publication
- **OpenAPI**: NeoHaskell auto-generates OpenAPI specs from command schemas — standard tooling works

### Inbound Integration

- **HTTP**: external services call NeoHaskell command endpoints — standard REST
- **Events**: NeoHaskell services can subscribe to external event streams
- **Polling**: NeoHaskell services can poll external APIs and translate responses to internal events
- **The `withOutbound` pattern**: integrations are explicit in the type — the compiler enforces that all integrations are declared

### Database Integration

- **PostgreSQL event store**: the default event store — production-ready, well-understood operationally
- **In-memory event store**: for testing and development — no PostgreSQL required
- **Custom event stores**: the event store interface is pluggable — implement your own if needed
- **Query databases**: updates can write to any database — PostgreSQL, Redis, Elasticsearch

## Migration Path

### One Service at a Time

- You don't have to migrate your entire system — start with one service
- Identify a domain that benefits most from event sourcing: audit trails, complex business rules, high concurrency
- Build the new service in NeoHaskell alongside the existing system
- Route traffic to the new context gradually — feature flags, percentage rollout
- Decommission the old system when the new context is stable

### Strangler Fig Pattern

- The existing system continues to run — NeoHaskell doesn't replace it overnight
- New features go into NeoHaskell services
- Old features are migrated one at a time — or left in the old system if migration isn't worth it
- The boundary between old and new is explicit: integration events cross the boundary

### Data Migration

- Historical data can be imported as events: `AccountMigratedFromLegacy { balance, timestamp }`
- The migration event is the starting point for the event stream — history before migration is captured
- Updates can be built from the migration event forward — no need to replay the entire legacy history
- **Caveat**: migrating data is a one-time operation — test thoroughly before running in production

## Operational Concerns

### Monitoring

- **Event store metrics**: event count per stream, append rate, read rate
- **Update metrics**: rebuild time, staleness (time since last event processed)
- **Command metrics**: command processing time, rejection rate, error rate
- **Cross-context metrics**: event delivery latency, handler error rate

### Alerting

- **Event store growth**: alert when the event store grows faster than expected
- **Update staleness**: alert when updates fall behind the event stream
- **Command rejection rate**: alert when rejection rate spikes (may indicate a bug or attack)
- **Cross-context delivery failures**: alert when events fail to deliver to subscribing contexts

### Backup and Restore

- **Event store backup**: the event store is the source of truth — back it up like a database
- **Update backup**: updates are derived — they can be rebuilt from the event store
- **Point-in-time restore**: restore the event store to any point in time — replay events to rebuild updates
- **Disaster recovery**: the event store is the only thing you need to restore — updates rebuild automatically

### Observability

- **Event replay for debugging**: replay events to any point in time to inspect state — no need for production debuggers
- **Audit trail**: the event store is the audit log — every state change is recorded with timestamp and actor
- **Distributed tracing**: commands and events can carry correlation IDs — trace a request across services
- **Structured logging**: NeoHaskell's configuration-as-code includes log level configuration — structured logs by default
