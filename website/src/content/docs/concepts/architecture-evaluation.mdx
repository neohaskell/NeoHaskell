This page is for architects and senior engineers deciding whether NeoHaskell belongs in their stack. It's not marketing. It's the technical trade-offs, laid out plainly, with specific comparisons to the systems you'd pick instead.

## What you're actually evaluating

NeoHaskell is three things wearing one label:

1. **A Haskell dialect** with a curated standard library and specific style conventions.
2. **An event-sourcing and CQRS framework** that bakes commands, events, entities, queries, services, and integrations into the language's type system.
3. **A service runtime** with HTTP, CLI, and MCP transports, a PostgreSQL event store, OAuth2 integration, and a configuration system.

Most evaluations focus on #2 because that's where the marketing claims live. But when you adopt NeoHaskell you adopt all three together. Be honest with yourself about whether all three fit your situation.

## Who is it for

- **Small-to-medium teams** (2–15 engineers) building systems with meaningful state transitions: fintech, scheduling, workflow automation, healthcare, supply chain, collaborative tools.
- **Teams that value correctness and type safety** more than language popularity.
- **Teams that have the autonomy to pick their stack.** "My CTO will let me try anything that compiles" is a good sign; "I need approval from an architecture review board" is a warning.
- **Teams with at least one engineer** comfortable enough with Haskell or a similar language to handle the inevitable weird library edge case.

## Who it's NOT for

- **Pure CRUD teams.** If you don't have interesting state transitions, event sourcing is overkill. Use Django or Rails.
- **Teams at Fortune 500 hiring scale.** Recruiting 50 NeoHaskell engineers will be hard in 2026. Recruiting 5 is fine.
- **Teams that need a proven ecosystem.** NeoHaskell is pre-release. Many things will be "write it yourself."
- **Teams that need strict single-region consistency at microsecond latency.** Event sourcing adds read-side lag — usually milliseconds, but that's more than zero.
- **Teams allergic to learning.** The mental model takes a few weeks to internalize. If your team doesn't have the appetite, stop here.

## How it compares

### vs. EventStoreDB

**EventStoreDB** is a purpose-built database for events. It stores events, handles stream-based consistency, and exposes a read/write API. It is transport-layer and storage-layer; it is not a framework for modeling your domain.

In an EventStoreDB system, you still write:

- Your own command handlers (in Java, C#, Go, whatever).
- Your own aggregates (reconstructing state from events).
- Your own projections (building read models).
- Your own validation layer.
- Your own retry logic.
- Your own schema.
- Your own testing harness.

EventStoreDB does the storage. You do everything else.

NeoHaskell *is* everything else. It includes the event store (PostgreSQL-backed by default, pluggable), the command executor, the entity machinery, the projection system, the integration runtime, the transports. You write domain logic. The framework handles the rest.

**When to pick EventStoreDB:** you already have a robust framework in your language of choice, you want only the storage layer, and you don't want to switch languages.

**When to pick NeoHaskell:** you want the whole story, and you're willing to learn a new language for it.

### vs. Kafka + a framework (e.g., Spring, Akka, Phoenix)

**Kafka** is a distributed log. It is NOT an event store in the event-sourcing sense — it's a streaming message broker optimized for throughput. You can build event sourcing on top of it, but the concurrency semantics (single-stream atomic appends, optimistic concurrency) aren't first-class.

You'd usually pair Kafka with a framework (Spring, Akka, Phoenix, whatever) that provides command handling, state rebuilding, and projections. The framework is where most of the integration work lives, and every framework has a different opinion.

NeoHaskell's "framework" is the language. The same machinery — commands, events, entities, queries — is woven into the type system, so the integration and correctness guarantees are tighter. You don't worry about serialization mismatch between the framework and the store; there's no framework layer to mismatch.

**When to pick Kafka + a framework:** you need Kafka for other reasons (streaming analytics, log aggregation, pub/sub) and want to build event sourcing on top of infrastructure you already have.

**When to pick NeoHaskell:** you want a purpose-built solution that minimizes the glue code between your domain and your storage.

### vs. Axon Framework (Java)

**Axon** is the closest comparison. It treats commands, events, and aggregates as first-class concepts, provides a Java annotation-based API, and can plug into various event stores. It's mature and well-tested.

The difference is:

- **Axon enforces its contracts at runtime.** An aggregate missing an `@EventSourcingHandler` for a new event type is a runtime exception. NeoHaskell catches it at compile time.
- **Axon uses annotations.** NeoHaskell uses types. Annotations are easier to introduce gradually; types are harder to circumvent once in place.
- **Axon is Java, with Java's verbosity and Java's ecosystem.** NeoHaskell is a curated Haskell, with different trade-offs.

**When to pick Axon:** you're on the JVM, you have a Java team, and you want mature tooling with proven scalability in enterprise settings.

**When to pick NeoHaskell:** you want compiler-level guarantees and you don't already have a JVM investment.

### vs. "just use Haskell"

Regular Haskell gives you the type system but not the event-sourcing primitives. You'd pick libraries like Eventful, eventstore-client, or roll-your-own. You'd spend your first week fighting monad transformer stacks. The outcome depends heavily on library choices.

NeoHaskell is an opinionated curation. The standard library is chosen to make domain code approachable. The event-sourcing primitives are baked in. The style guide is specific. You give up some flexibility; you get a much shorter ramp.

**When to pick regular Haskell:** you need bleeding-edge libraries, you want full control, and your team is already deeply Haskell-native.

**When to pick NeoHaskell:** you want event-sourcing-first, you care about accessibility, and you don't want to shop for libraries.

### vs. Elixir / Phoenix with CQRS libraries

Elixir with Commanded (or similar) is a real alternative. Both languages are good at concurrency. Elixir has a bigger ecosystem and a more mature BEAM runtime. NeoHaskell has stronger compile-time guarantees.

**When to pick Elixir/Phoenix:** you want BEAM-style fault tolerance (soft real-time, let-it-crash), you value runtime observability, and you can live with dynamic typing.

**When to pick NeoHaskell:** you want everything Elixir gives you plus the compile-time safety net.

## Performance

The NeoHaskell service layer targets 50k req/s per node as a design goal. Real-world throughput depends heavily on your `decide` function complexity, your event store latency, and your projection strategy.

Specifically:

- **Event append latency** on the PostgreSQL event store is typically 1–5ms per append under moderate load.
- **Command latency** (decide + append + response) is 2–10ms on a well-configured cluster.
- **Query latency** is single-row lookup time — sub-millisecond when projections fit in memory.
- **Entity reconstruction** uses snapshot caches; cached entities are instant, and replaying a handful of events since the snapshot adds microseconds.

The practical bottleneck is usually PostgreSQL write bandwidth, not NeoHaskell itself. If you outgrow one PostgreSQL instance, you either shard events by stream, split into independently-deployed services, or move to a purpose-built event store.

## Operability

NeoHaskell applications compile to a single static binary via GHC. You can deploy them anywhere a Linux binary runs: Docker, Kubernetes, bare metal, Fly.io, Render, Nomad. Startup time is a few seconds.

Observability:

- Structured logging is built-in (`Log.info`, `Log.warn`, `Log.error`).
- Request tracing fields propagate through the command executor.
- OpenAPI spec is generated from the service definition automatically.
- Health check endpoint is enabled by default at `/health`.
- Metrics export (Prometheus-compatible) is a planned first-class feature; for now you wire it in yourself.

What's missing (as of this writing):

- Distributed tracing hooks beyond basic request IDs.
- Built-in circuit breakers (but you can build them with `Task` combinators).
- Managed service offerings. You self-host.

## Testing story

NeoHaskell's testing story is one of its strongest selling points. Because `decide`, `update`, `combine`, and integration `reactTo` functions are all pure, unit tests are trivial. A typical unit test is 5–15 lines, runs in microseconds, and requires zero infrastructure.

Integration tests use an in-memory event store and dispatch commands through the real executor. They're slower (milliseconds to a few seconds) but still fast enough to run on every commit.

Property-based testing integrates cleanly — event-sourced systems have natural invariants to express ("balance is never negative", "total deposits minus total withdrawals equals current balance", "replaying the same events always produces the same state"), and property tests catch classes of bugs example tests miss.

## Adoption risk

### Technical risk

- The language is pre-release. Breaking changes happen, though the team pins versions on release and publishes migration notes.
- Some libraries you'd reach for in other ecosystems don't exist yet. You write integrations yourself — usually a few hours each.
- The PostgreSQL event store is well-tested but not yet battle-hardened at 50k req/s production scale.

### Team risk

- "NeoHaskell" is not a recognized résumé item in 2025. Your engineers may worry about career risk. Good mitigations: public GitHub contributions to NeoHaskell itself, blog posts about your experience, conference talks.
- Onboarding takes longer than CRUD Python. Expect a new engineer to take 2–4 weeks to be productive.
- If you lose the lead champion, the project can stall. Make sure at least two engineers can maintain it before betting heavily.

### Organizational risk

- If you're regulated and your compliance team is used to reviewing specific frameworks, you'll need to explain what NeoHaskell is. This is friction.
- If you have a policy of "only use stacks we've shipped to production", NeoHaskell is a first-time adoption by definition. This is a political conversation.

## The "should we pilot it?" checklist

Adopting NeoHaskell for your first project? Here's a quick sanity check. Answer yes to most of these:

- [ ] Our domain has interesting state transitions (not pure CRUD).
- [ ] We value audit trails, replayability, or historical debugging.
- [ ] We have at least one engineer who is excited to learn Haskell.
- [ ] We have autonomy over our tech stack — no architecture review board involvement.
- [ ] The system we're building is new (greenfield), not a rewrite of an existing database-driven system.
- [ ] We can tolerate a longer ramp in exchange for a better ongoing experience.
- [ ] We have an appetite to contribute back to the ecosystem when we find rough edges.

If you answered yes to six or more, pilot it on a small project. If you answered yes to fewer, start with a toy project first and see how the team reacts.

## The honest recommendation

NeoHaskell is ready for real-world use if:

- You are building a new service where event sourcing is the right architectural choice regardless of language.
- Your team has one or more people willing to learn it.
- You can tolerate occasional rough edges in exchange for long-term benefits.

NeoHaskell is not yet ready if:

- You need mature tooling for every conceivable integration.
- You cannot stomach any adoption risk.
- Your domain doesn't justify the structural overhead.

Pilot it on a single service first. Measure what you get. Decide from there.
