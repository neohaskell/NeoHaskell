Event sourcing is the most testable architecture you can use for a business system. This isn't marketing — it's a direct consequence of how the pieces are shaped. Commands are pure functions from state to events. Projections are pure functions from entity state to read models. Integrations are pure functions from events to actions. Test each one by passing in data and asserting on the output.

No mocks. No in-memory database setup. No fixtures. No "stand up a test container." Just pure function tests.

## The three kinds of tests

An event-sourced NeoHaskell application has three layers, each tested differently:

1. **Command tests** — did `decide` produce the right events for the right inputs?
2. **Query tests** — did `combine` produce the right read model for a given entity state?
3. **Integration tests** — did the whole system, wired together, behave correctly end-to-end?

The first two are pure function tests. The third uses a test harness that spins up an in-memory event store and dispatches commands through the real command executor. All three are fast and reliable.

## Command tests — Given-When-Then

Commands test as pure functions. The grammar is straightforward:

- **Given** past events (or an entity state built from them).
- **When** a command is dispatched.
- **Then** specific events are produced (or the command is rejected).

Here's a test for the `Withdraw` command from the tutorial:

```haskell
module NeoBank.Account.Commands.WithdrawSpec where

import Array qualified
import Core
import Decider (runDecision, DecisionContext (..), CommandResult (..))
import NeoBank.Account.Commands.Withdraw (Withdraw (..), decide)
import NeoBank.Account.Core
import Service.Auth (RequestContext (..))
import Service.Event (InsertionType (..))
import Task qualified
import Test
import Uuid qualified


-- A helper: run decide in a test context.
runDecide ::
  Withdraw ->
  Maybe Account ->
  Task Text (CommandResult AccountEvent)
runDecide cmd entity = do
  let ctx = DecisionContext { genUuid = Task.yield Uuid.nil }
  let reqCtx = RequestContext { user = Nothing }
  runDecision ctx (decide cmd entity reqCtx)


-- A helper: make an account at a specific balance.
anAccount :: Int -> Account
anAccount balance =
  Account
    { accountId = Uuid.nil
    , ownerName = "Test"
    , accountType = Checking
    , balance = balance
    , history = Array.empty
    }


spec :: Spec Unit
spec = describe "Withdraw" do
  it "rejects when the account does not exist" \_ -> do
    let cmd = Withdraw { accountId = Uuid.nil, amount = 50 }
    result <- runDecide cmd Nothing
    result |> shouldBe (RejectCommand "Account does not exist")

  it "rejects zero or negative amounts" \_ -> do
    let cmd = Withdraw { accountId = Uuid.nil, amount = 0 }
    result <- runDecide cmd (Just (anAccount 100))
    result |> shouldBe (RejectCommand "Withdrawal amount must be positive")

  it "rejects insufficient funds" \_ -> do
    let cmd = Withdraw { accountId = Uuid.nil, amount = 150 }
    result <- runDecide cmd (Just (anAccount 100))
    result |> shouldBe (RejectCommand "Insufficient funds")

  it "accepts valid withdrawals" \_ -> do
    let cmd = Withdraw { accountId = Uuid.nil, amount = 40 }
    result <- runDecide cmd (Just (anAccount 100))
    let expected =
          AcceptCommand
            ExistingStream
            (Array.fromLinkedList
              [ MoneyWithdrawn { entityId = Uuid.nil, amount = 40 } ])
    result |> shouldBe expected
```

Every test is three lines:

1. Build the command (the "When").
2. Call `runDecide` with a given entity state (the "Given").
3. Assert the result (the "Then").

No database. No fixtures. No mocks. The test runs in microseconds. You can write a hundred of these in an afternoon.

## Query tests — combine is a pure function

Testing queries is even simpler because `combine` takes two values and returns one value:

```haskell
spec :: Spec Unit
spec = describe "MonthlySummary" do
  it "counts only deposits under totalDeposits" \_ -> do
    let account = Account
          { accountId = Uuid.nil
          , ownerName = "Alice"
          , accountType = Checking
          , balance = 70
          , history = Array.fromLinkedList
              [ Transaction { kind = "deposit", amount = 100 }
              , Transaction { kind = "withdrawal", amount = 30 }
              ]
          }
    case combine account Nothing of
      Update summary -> do
        summary.totalDeposits |> shouldBe 100
        summary.totalWithdrawals |> shouldBe 30
        summary.endingBalance |> shouldBe 70
      _ -> Test.fail "Expected Update action"

  it "returns NoOp for empty history on incremental update" \_ -> do
    -- When combine is called with the same entity state twice,
    -- the second call should be a no-op-equivalent update.
    let account = ...
    let existing = Just MonthlySummary { ... }
    case combine account existing of
      Update _ -> Task.yield ()  -- expected
      _ -> Test.fail "Expected Update action"
```

Query tests are where you catch logic bugs that the type system can't. A query's `combine` function compiles whether you wrote `+` or `-` — the test is what tells you which is correct.

## Given-When-Then as a DSL

For more complex scenarios — where you want to replay multiple events before running the command — NeoHaskell provides the `AppSpec` testing helpers:

```haskell
spec :: Spec Unit
spec = describe "Full account lifecycle" do
  specificationFor testAccount do
    scenario "deposit then withdraw" do
      given AccountOpened { entityId = aliceId, ownerName = "Alice", accountType = Checking }
      given MoneyDeposited { entityId = aliceId, amount = 100 }
      when Withdraw { accountId = aliceId, amount = 40 }
      then_ emits MoneyWithdrawn { entityId = aliceId, amount = 40 }

    scenario "withdrawal exceeds balance" do
      given AccountOpened { entityId = aliceId, ownerName = "Alice", accountType = Checking }
      given MoneyDeposited { entityId = aliceId, amount = 100 }
      when Withdraw { accountId = aliceId, amount = 500 }
      then_ rejects "Insufficient funds"
```

The DSL reads like a spec. This is literally the test style you'd write with a pen on a whiteboard — "given these events, when this command runs, then these events are produced." You don't have to translate it to code; the code *is* the translation.

## Integration tests — the whole stack

For end-to-end tests, use the in-memory event store and the real command executor. This lets you test the full stack — decide, executor, event store, query subscribers, integrations — without touching a real database.

```haskell
it "builds a correct statement query after a sequence of commands" \_ -> do
  -- Set up an in-memory app
  store <- InMemory.new |> Task.mapError toText
  let app = Application.new
        |> Application.withEventStore @() (\_ -> Task.yield store)
        |> Application.withTransport MockTransport.new
        |> Application.withService NeoBank.Account.service
        |> Application.withQuery @Statement

  -- Run the commands through the executor
  executor <- createExecutor app
  _ <- dispatch executor (OpenAccount { ownerName = "Alice", accountType = Checking })
  _ <- dispatch executor (Deposit { accountId = aliceId, amount = 100 })
  _ <- dispatch executor (Withdraw { accountId = aliceId, amount = 30 })

  -- Assert on the query
  statement <- queryByIdOrFail @Statement aliceId
  statement.currentBalance |> shouldBe 70
  Array.length statement.lines |> shouldBe 2
```

Integration tests are slower than unit tests (hundreds of milliseconds vs. microseconds) but they verify the wiring. Run them in CI to catch regressions; run unit tests locally for fast iteration.

## Property-based testing

Event-sourced systems are uniquely suited to property-based testing. Instead of writing specific examples, you express properties that should hold for *any* sequence of commands:

- "No sequence of valid commands should produce a negative balance on a checking account."
- "The sum of all deposits minus the sum of all withdrawals should equal the current balance."
- "Replaying the event stream twice should produce identical entity state."
- "A query's `combine` function should be idempotent with respect to identical entity states."

NeoHaskell integrates with QuickCheck-style property testing. A property test might look like:

```haskell
it "preserves balance invariants" \_ -> do
  Test.forAll genValidCommandSequence \commands -> do
    finalState <- replayCommands commands
    finalState.balance |> shouldSatisfy (>= 0)
```

The generator `genValidCommandSequence` produces random sequences of valid commands; the framework runs the property against hundreds of them and reports any failing case (with an automatically minimized counterexample).

<Aside type="tip" title="When to use property tests">
Reach for property-based tests when your invariants are stronger than any single example can show — especially "never" and "always" properties. Keep example tests for the specific cases you want to document explicitly.
</Aside>

## Testing integrations

Integrations are pure functions from events to actions, so they test like any other pure function:

```haskell
it "emits a Withdraw when a transfer is requested" \_ -> do
  let transfer = initialState { transferId = Uuid.nil }
  let event = TransferRequested
        { entityId = Uuid.nil
        , sourceAccountId = sourceId
        , destinationAccountId = destId
        , amount = 50
        }
  let outbound = reactTo Orchestrator transfer event
  let actions = Integration.getActions outbound
  Array.length actions |> shouldBe 1
```

You construct an event, call `reactTo` on the integration, and assert on the actions it emits. For outbound actions that make HTTP calls, you can mock the `ActionContext` to assert what would have been sent without actually sending it.

## Why this works so well

Everything in this page comes down to one property: **the core of your application is pure functions**. Pure functions are the easiest thing to test, and NeoHaskell has structured your code so that the interesting logic is all pure. The impure parts — the event store, the transports, the external APIs — are at the edges, and they're small enough to test with in-memory stubs.

Compare this to traditional layered systems, where you need Docker Compose to run the database, a mock HTTP server for external APIs, and a fake user for auth. Setting up the test environment is half the work; writing the actual test is the easy part. In NeoHaskell, the ratio is flipped.
