Most languages handle side effects by letting every function do whatever it wants. A function might open a file, call a network service, read from the global clock, print to stderr, or start a thread — and none of that is visible from its signature. You only find out at runtime.

NeoHaskell takes the opposite approach. Every side effect is tracked in the type system via the `Task` type. If a function has side effects, its return type says so. If it doesn't, you can trust that it's pure and you can test it as a pure function.

This page explains what `Task` is, how to work with it, and how NeoHaskell handles concurrency without you writing a single callback.

## The Task type

```haskell
Task err value
```

Read this as: "an asynchronous computation that will eventually produce a `value`, or fail with an error of type `err`."

Compare it to familiar equivalents:

| Language | Equivalent |
|---|---|
| JavaScript | `Promise<value>` (but with a typed error) |
| Rust | `async fn() -> Result<value, err>` |
| Go | `func() (value, error)` (but composable) |
| Java | `CompletableFuture<value>` (but with a typed error) |

The critical feature is that `err` is typed. A `Task Text User` returns a `User` on success and a `Text` message on failure. You know, from the type, that this function can fail, and you know what shape the failure takes. There is no surprise exception hiding in the body.

## Working with Task

Inside a `do` block, `<-` unwraps a `Task`:

```haskell
loadUserAndPosts :: Uuid -> Task Text (User, Array Post)
loadUserAndPosts userId = do
  user <- fetchUser userId
  posts <- fetchPostsByUser userId
  Task.yield (user, posts)
```

If `fetchUser` fails, the whole block short-circuits — `posts` is never fetched, the block returns `Task.throw`ing the same error. This is exactly how `async/await` works in JavaScript, except the failure type is in the signature.

## Task combinators

Beyond `do` blocks, you can compose tasks with a few combinators:

```haskell
Task.yield value           -- Lift a pure value into a Task
Task.throw error           -- Fail with a typed error
Task.map f task            -- Transform the value
Task.mapError f task       -- Transform the error
Task.andThen f task        -- Monadic bind (fmapping, but for Tasks)
Task.forEach xs action     -- Run a task for each element of a list
Task.forever task          -- Run forever (use sparingly)
Task.recover handler task  -- If task fails, run handler instead
```

A typical pipeline:

```haskell
fetchUser userId
  |> Task.map formatUserName
  |> Task.mapError (\err -> [fmt|Fetch failed: #{err}|])
```

## Integrating with legacy IO

Sometimes you need to call a function that returns raw `IO a` (from `base` or a legacy library). Convert it with `Task.fromIO`:

```haskell
currentTime :: Task err DateTime
currentTime = Task.fromIO getCurrentTimeFromSystem
```

If the IO action can throw exceptions, use `Task.fromFailableIO` instead — it catches exceptions and converts them to a typed error:

```haskell
readConfig :: FilePath -> Task Text Config
readConfig path =
  Task.fromFailableIO readFileRaw path
    |> Task.map parseConfig
```

`Task.fromFailableIO` converts any thrown exception into the error channel. From then on, you handle it with `Task.recover` or `Task.mapError` — no try/catch needed.

## Failure is data

Unlike exceptions, a `Task` failure is a first-class value. You can inspect it, log it, convert it, wrap it, recover from it. You don't need `try`/`catch`/`finally`. You handle failure by:

- **Letting it propagate.** If you don't catch it, the error flows up through `<-` bindings and out of the function. Same mental model as rethrowing in JS/TS.
- **Catching with `Task.recover`.** If `task` fails, run the recovery function:
  ```haskell
  fetchUser userId
    |> Task.recover (\err -> do
        Log.warn [fmt|Primary failed: #{err}. Trying fallback.|]
        fetchUserFromReplica userId)
  ```
- **Converting to a `Result`.** `Task.asResult` turns a failing task into one that always succeeds with a `Result`:
  ```haskell
  result <- Task.asResult (fetchUser userId)
  case result of
    Ok user -> ...
    Err reason -> ...
  ```

<Aside type="note" title="Why not use exceptions?">
Exceptions are invisible in type signatures. A function that looks pure might throw on every third call, and you only discover this at runtime. Task forces the failure to be named in the type, which means the compiler helps you handle it (or at least remember it exists).
</Aside>

## Concurrency

NeoHaskell gives you primitives for concurrent tasks via the `AsyncTask` module (for parallelism) and the runtime's ability to spawn effectful workers.

### Running things in parallel

```haskell
import AsyncTask qualified

loadEverything :: Task Text Dashboard
loadEverything = do
  userHandle <- AsyncTask.start (fetchUser userId)
  postsHandle <- AsyncTask.start (fetchPosts userId)
  ordersHandle <- AsyncTask.start (fetchOrders userId)

  user <- AsyncTask.wait userHandle
  posts <- AsyncTask.wait postsHandle
  orders <- AsyncTask.wait ordersHandle

  Task.yield Dashboard { user = user, posts = posts, orders = orders }
```

Three fetches run concurrently. The dashboard isn't assembled until all three finish. If any one fails, the whole pipeline fails.

### Channels

For producer/consumer patterns, use `Channel`:

```haskell
import Channel qualified

pipelineExample = do
  ch <- Channel.new
  -- Producer
  AsyncTask.start (forEach items \item -> Channel.send ch item)
  -- Consumer
  forever do
    item <- Channel.receive ch
    process item
```

### Shared mutable state

Real concurrent systems need shared state. NeoHaskell gives you `ConcurrentVar` (a mutex-protected cell), `Lock`, and `ConcurrentMap` for this purpose. Shared state is visible and explicit — you have to opt into it, and when you do, the concurrency primitive is in the type.

```haskell
import ConcurrentVar qualified

counter = do
  var <- ConcurrentVar.new 0
  AsyncTask.start (incrementer var)
  AsyncTask.start (incrementer var)
  -- ...
```

## Why `decide` doesn't return a Task

You might have noticed that `decide` functions do *not* return `Task`:

```haskell
decide :: Deposit -> Maybe Account -> RequestContext -> Decision AccountEvent
```

That's not an oversight. `decide` is pure. It's not allowed to do I/O. It's not allowed to access the outside world. This is deliberate: a pure `decide` function is trivially testable, and more importantly, it can be *replayed*. If the command executor needs to retry after a concurrency conflict, it just calls `decide` again with the fresh state. There are no side effects to undo.

If your business logic really needs an external lookup (like "check the user's credit score from an external service"), that's not a job for `decide`. You do the lookup *before* dispatching the command, and pass the result into the command as a field. Or you use the inbound integration pattern to react to the result of the external call.

This separation — pure decisions, effectful integrations — is how NeoHaskell stays testable at the core while still being able to talk to the messy outside world.

## The runtime

NeoHaskell uses the GHC runtime under the hood, which means lightweight green threads, preemptive scheduling, and STM for shared state if you need it. The practical implications:

- **Spawning a task is cheap** — microseconds, kilobytes of memory.
- **Blocking a task does not block a thread** — the runtime moves other work onto the OS thread while the blocked task waits.
- **You can have thousands of concurrent tasks** without worrying about thread pool sizing.
- **IO-bound workloads scale linearly** with work, not with threads.

For most NeoHaskell applications, you never think about thread pools, work queues, or scheduling policies. You write `do` blocks, the runtime figures out the rest.

## The short version

- `Task err val` is the type of effectful computations. Think `Promise<val>` with a typed error.
- `do` blocks sequence tasks. `<-` is `await`. `Task.yield` is `return`. `Task.throw` is `throw`.
- Failure is data, not exceptions. You handle it with `Task.recover`, `Task.asResult`, or propagation.
- `AsyncTask.start` / `AsyncTask.wait` for parallelism. `Channel` for producer/consumer. `ConcurrentVar` / `Lock` for shared state.
- `decide` is pure by design — no tasks, no I/O.
