You've been writing CRUD code your whole career. You know the patterns. INSERT. SELECT. UPDATE. DELETE. A REST endpoint per operation. An ORM somewhere. Migrations in a folder. A validation layer. A controller. A service. A repository.

This page translates every piece of that world into its event-sourced equivalent. Not by reinventing your mental model all at once, but by taking each pattern you already know and showing what replaces it.

## The direct translations

### INSERT → Create event

**CRUD:**
```sql
INSERT INTO users (id, email) VALUES ('...', 'alice@example.com');
```

**NeoHaskell:**
```haskell
-- The command
data RegisterUser = RegisterUser { email :: Text }

-- The decision
decide cmd existing _ctx = case existing of
  Just _ -> Decider.reject "User already exists"
  Nothing -> do
    userId <- Decider.generateUuid
    Decider.acceptNew
      [ UserRegistered { entityId = userId, email = cmd.email } ]
```

The INSERT becomes a `UserRegistered` event. The event is immutable; once written, it's the record that "yes, on this day, at this time, we registered a user with this email."

### UPDATE → Domain event

**CRUD:**
```sql
UPDATE users SET email = 'new@example.com' WHERE id = '...';
```

**NeoHaskell:**
```haskell
decide cmd user _ctx = case user of
  Nothing -> Decider.reject "User does not exist"
  Just _ ->
    Decider.acceptExisting
      [ UserEmailChanged
          { entityId = cmd.userId
          , oldEmail = u.email  -- if you have the user `u` in scope
          , newEmail = cmd.newEmail
          }
      ]
```

Important detail: the event is named after what **happened in the domain**, not what the system did. `UserEmailChanged` is the domain fact. `UpdateUser` is a SQL-speak name — avoid it. The event often carries both the old and the new value so projections can reason about the change.

### DELETE → "Marked as..." event

**CRUD:**
```sql
DELETE FROM users WHERE id = '...';
```

**NeoHaskell:**
```haskell
Decider.acceptExisting
  [ UserDeactivated
      { entityId = userId
      , reason = "User requested account deletion"
      , deactivatedAt = now
      }
  ]
```

Events are append-only. You never actually delete anything. What you do is record a new fact — "this user was deactivated" — and your projections handle the consequences. For user-visible "deletion" you might also emit a `UserDataRedacted` event that strips PII from read models while leaving the event log structure intact.

<Aside type="note" title="GDPR and the right to be forgotten">
"Append-only" does not mean you have to store personal data forever. The standard pattern is to store PII in a separate side table keyed by event ID, and to respond to a redaction request by nulling those rows. The event log still has the event, but the sensitive payload is gone. See the [Schema evolution](/concepts/schema-evolution/) page for details.
</Aside>

### SELECT → Query

**CRUD:**
```sql
SELECT name, email, created_at FROM users WHERE id = '...';
```

**NeoHaskell:**

Define a query that projects what you need. The projection stays up to date automatically as events arrive:

```haskell
data UserProfile = UserProfile
  { userId :: Uuid
  , name :: Text
  , email :: Text
  , createdAt :: DateTime
  }

instance QueryOf User UserProfile where
  queryId user = user.id
  combine user _existing = Update UserProfile
    { userId = user.id
    , name = user.name
    , email = user.email
    , createdAt = user.createdAt
    }
```

Read it via `GET /queries/user-profile/{id}`. Performance is equivalent to a SQL `SELECT` on a denormalized table — because that's essentially what it is. The difference is that the denormalization is automatic, driven by events, and you can build as many different projections as you like from the same event stream.

### JOIN → Query across entities

**CRUD:**
```sql
SELECT u.name, COUNT(o.id)
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id;
```

**NeoHaskell:**

Define a query that listens to both `User` and `Order` entities:

```haskell
data UserOrders = UserOrders
  { userId :: Uuid
  , userName :: Text
  , orderCount :: Int
  }

deriveQuery ''UserOrders [''User, ''Order]

instance QueryOf User UserOrders where
  queryId user = user.id
  combine user existing = Update UserOrders
    { userId = user.id
    , userName = user.name
    , orderCount = existing |> Maybe.map (.orderCount) |> Maybe.withDefault 0
    }

instance QueryOf Order UserOrders where
  queryId order = order.userId
  combine order existing = case existing of
    Just uo -> Update uo { orderCount = uo.orderCount + 1 }
    Nothing -> NoOp  -- User query not built yet; skip this event
```

The query doesn't do an actual JOIN — it maintains a denormalized read model that gets updated whenever either entity changes. Reads are single-row lookups; updates are incremental.

### Migration → Add an event variant

**CRUD:**
```sql
ALTER TABLE users ADD COLUMN phone VARCHAR(20);
```

**NeoHaskell:**
```haskell
data UserEvent
  = UserRegistered { entityId :: Uuid, email :: Text }
  | UserEmailChanged { entityId :: Uuid, oldEmail :: Text, newEmail :: Text }
  | UserPhoneAdded { entityId :: Uuid, phone :: Text }  -- new
```

Add the variant. The compiler will tell you every place that needs to handle it. No database migration to write, no backfill script, no rollback plan. Existing users simply have no `UserPhoneAdded` event in their stream — that's fine, it means their phone is not set.

For more complex schema changes, see [Schema evolution](/concepts/schema-evolution/).

### Validation layer → Decide function

**CRUD:**
```python
def register_user(email):
    if not email:
        raise ValueError("Email is required")
    if "@" not in email:
        raise ValueError("Email is invalid")
    if db.query(User).filter_by(email=email).exists():
        raise ValueError("Email already in use")
    db.session.add(User(email=email))
```

**NeoHaskell:**
```haskell
decide cmd existing _ctx = case existing of
  Just _ ->
    Decider.reject "User already exists"
  Nothing ->
    if Text.isEmpty cmd.email
      then Decider.reject "Email is required"
      else if not (Text.contains "@" cmd.email)
        then Decider.reject "Email is invalid"
        else do
          userId <- Decider.generateUuid
          Decider.acceptNew [ UserRegistered { entityId = userId, email = cmd.email } ]
```

Validation lives in `decide` because that's the layer with the context to validate correctly: it sees the command, the current state, and the request context. Putting it anywhere else (middleware, ORM hooks, triggers) means that layer has an incomplete view of the domain.

### Transaction → Decider insertion mode

**CRUD:**
```sql
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 'A';
UPDATE accounts SET balance = balance + 50 WHERE id = 'B';
COMMIT;
```

**NeoHaskell:**

You don't need a transaction because each entity has its own event stream, and `acceptExisting` / `acceptAfter` provide optimistic concurrency per stream. For cross-entity operations like transfers, you use the saga pattern (an integration that coordinates the flow). See [Tutorial Part 5 — Transfers](/tutorial/05-transfers/) for the full pattern.

The `Decider` has three insertion modes that act like transactional guarantees per-stream:

- `acceptNew` — "only if the stream doesn't exist yet" (creation).
- `acceptExisting` — "only if the stream already exists" (update).
- `acceptAfter position` — "only if the stream is at exactly this position" (optimistic concurrency).

Two concurrent commands on the same stream can race; `acceptAfter` lets you ensure the second one sees the effects of the first (or fails with a conflict and retries).

### Audit log → You already have it

**CRUD:** A separate `audit_log` table, painstakingly maintained by triggers or application-level logging. Often incomplete. Often wrong. Often out of sync.

**NeoHaskell:** The event store *is* the audit log. Every action that has ever affected your system is recorded, immutable, in order, with a timestamp. You don't write one; you had one the whole time.

### Soft delete → Deactivation event

**CRUD:**
```sql
UPDATE users SET deleted_at = NOW() WHERE id = '...';
```

**NeoHaskell:**
```haskell
Decider.acceptExisting
  [ UserDeactivated { entityId = userId, reason = "..." } ]
```

Soft delete was always a kludge around the fact that CRUD can't record "what happened, exactly, when". Event sourcing gets this right by default.

### Optimistic locking → acceptAfter

**CRUD:**
```sql
UPDATE users SET email = 'new', version = version + 1
WHERE id = '...' AND version = :expected_version;
```

**NeoHaskell:**
```haskell
Decider.acceptAfter expectedPosition
  [ UserEmailChanged { ... } ]
```

The stream position is the version. If another command has advanced the stream since you read it, `acceptAfter` rejects with a conflict and the executor automatically retries with the fresh state.

### Nested resources → Entity + projection

**CRUD:** `/users/:id/orders` → JOIN against `orders` table.

**NeoHaskell:** A `UserOrders` query (as shown above) that maintains a denormalized view for each user. Read it with a single lookup.

## What CRUD doesn't translate

A few CRUD patterns have no good event-sourced equivalent, because they shouldn't exist in event-sourced systems:

### Hard UPDATE WHERE with a bulk condition

**CRUD:**
```sql
UPDATE invoices SET status = 'paid' WHERE due_date < '2024-01-01';
```

**NeoHaskell:** You wouldn't do this. Each invoice is its own entity with its own stream. "Mark all overdue invoices paid" would be a scheduled integration that iterates over invoices and issues individual `MarkPaid` commands. It's more code, but it's also more honest — bulk updates erase the granularity your domain actually has.

### Cascade delete

**CRUD:**
```sql
DELETE FROM orders WHERE user_id = '...';  -- cascades to order_items
```

**NeoHaskell:** You emit a `UserAccountClosed` event. Any projections that listed orders for that user filter out the closed user's data. The orders themselves are not deleted — they're still facts, they still happened. If you need them gone for regulatory reasons, see the PII redaction pattern.

### Silent ON UPDATE triggers

**CRUD:** A trigger that fires on every update and writes to some side table.

**NeoHaskell:** You'd write an explicit outbound integration. Triggers hide side effects; integrations make them visible in the code.

## The mental compression

Here's the short version of everything above:

- **INSERT** → `acceptNew` event.
- **UPDATE** → `acceptExisting` domain event.
- **DELETE** → deactivation/redaction event.
- **SELECT** → projection (query).
- **JOIN** → multi-entity projection.
- **MIGRATION** → add event variant.
- **VALIDATION** → `decide` function.
- **TRANSACTION** → `acceptAfter` or saga.
- **AUDIT LOG** → you already have it.

Once this translation becomes reflex, reading event-sourced code will feel as natural as reading CRUD code does now.
