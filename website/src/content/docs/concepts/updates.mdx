You know the write side: commands produce events. This page is about the read side: how the same events become any number of different views of your data, automatically maintained, queryable over HTTP, fast to read.

## Queries, projections, read models — what's the difference?

They all mean the same thing. NeoHaskell uses **query** as the primary term because it's the most familiar to people coming from SQL. The terms you might see elsewhere:

- **Projection** — the name used in most event-sourcing literature.
- **Read model** — the name used in CQRS literature.
- **Materialized view** — the closest SQL analogue.

All four refer to the same concept: a denormalized view of your data, derived from events, kept up to date as new events arrive, optimized for fast reads.

## Why queries are separate from entities

You already have entities. Entities are the thing your `decide` function reads to make decisions. So why have a separate concept for queries?

Three reasons:

1. **Different shapes for different readers.** Your `Account` entity has whatever shape is convenient for the decide function. But a customer dashboard needs a different shape: recent transactions, a monthly chart, an alert flag. A fraud team needs yet another shape. Putting all of that into the entity would bloat it and make it slower to reconstruct.
2. **Cross-entity reads.** A single query can combine data from multiple entity types. An entity can only see itself.
3. **Independent evolution.** You can add a new query, change one, or delete one without touching the entities or the commands. They're decoupled by design.

## The three moving parts

1. **The query type** — a record describing the shape you want to read.
2. **`QueryOf`** — an instance for each entity type that contributes to the query, with a `combine` function.
3. **The query subscriber** — the runtime machinery that listens for entity changes and calls `combine`.

## A worked example

Let's build a query called `UserActivity` that tracks, for each user, how many posts they've written and how many comments they've left. It draws from two entity types: `User` and `Post`.

### The query type

```haskell
{-# LANGUAGE TemplateHaskell #-}

module MyApp.Queries.UserActivity (
  UserActivity (..),
  canAccess,
  canView,
) where

import Core
import Json qualified
import Service.Query.Auth (QueryAuthError, UserClaims, authenticatedAccess, ownerOnly)
import Service.Query.TH (deriveQuery)
import MyApp.User.Core (User)
import MyApp.Post.Core (Post)


data UserActivity = UserActivity
  { userId :: Uuid
  , userName :: Text
  , postCount :: Int
  , commentCount :: Int
  }
  deriving (Eq, Show, Generic)


instance Json.ToJSON UserActivity
instance Json.FromJSON UserActivity
instance ToSchema UserActivity


canAccess :: Maybe UserClaims -> Maybe QueryAuthError
canAccess = authenticatedAccess


canView :: Maybe UserClaims -> UserActivity -> Maybe QueryAuthError
canView = ownerOnly (.userId)


deriveQuery ''UserActivity [''User, ''Post]
```

The `deriveQuery ''UserActivity [''User, ''Post]` line is the crucial one. It tells NeoHaskell:

- This is a query type.
- It depends on `User` and `Post` entities.
- Generate the glue: the `Query` instance, the `NameOf` type instance, the `EntitiesOf` type instance, the HTTP endpoint registration.

### The QueryOf instances

Now we need to define how each entity type contributes to the query. Add these to the same file:

```haskell
instance QueryOf User UserActivity where
  queryId user = user.userId

  combine user maybeExisting =
    Update UserActivity
      { userId = user.userId
      , userName = user.name
      , postCount = maybeExisting |> Maybe.map (.postCount) |> Maybe.withDefault 0
      , commentCount = maybeExisting |> Maybe.map (.commentCount) |> Maybe.withDefault 0
      }


instance QueryOf Post UserActivity where
  queryId post = post.authorId  -- Queries are keyed by user ID, not post ID!

  combine post maybeExisting = case maybeExisting of
    Nothing ->
      -- The user query hasn't been built yet. We'll catch up later.
      NoOp
    Just activity ->
      Update activity
        { postCount = Array.length post.comments + 1  -- this post itself, plus comments
        , commentCount = activity.commentCount + Array.length post.comments
        }
```

Read the two instances carefully. Both answer two questions:

- **`queryId`**: what's the key of the query instance this entity contributes to? For `User`, it's the user's own ID. For `Post`, it's the `authorId` — because we want post counts *by user*, not by post.
- **`combine`**: given the current entity state and the existing query (if any), what should the new query look like?

The return value is a `QueryAction`:

- **`Update newQuery`** — store this as the new value of the query instance.
- **`Delete`** — remove this query instance from storage.
- **`NoOp`** — no change; leave the existing value alone.

### Registering the query

In your `App.hs`, add the query to the application:

```haskell
app :: Application
app =
  Application.new
    |> Application.withEventStore ...
    |> Application.withTransport WebTransport.server
    |> Application.withService userService
    |> Application.withService postService
    |> Application.withQuery @UserActivity
```

That's it. The query is now live:

- The HTTP endpoint `GET /queries/user-activity/{id}` serves data from it.
- Every `User` and `Post` event triggers a `combine` call.
- The result is persisted to the query object store (in-memory by default, PostgreSQL if you configure it).

### Querying over HTTP

```bash
curl http://localhost:8080/queries/user-activity/alice-user-id
```

```json
{
  "userId": "alice-user-id",
  "userName": "Alice",
  "postCount": 12,
  "commentCount": 47
}
```

## The full lifecycle

When an event is appended to the event store, here's what happens on the read side:

1. **The query subscriber sees the event.** It asks: "which queries care about this entity type?"
2. **For each matching query**, it does the following:
   - Reconstructs the current entity state by folding events (usually from cache + recent events, not from scratch).
   - Looks up the current value of the query instance (if any) in the query object store.
   - Calls `combine(entityState, existingQuery)`.
   - Persists the result according to the `QueryAction`.
3. **Multiple queries can depend on the same entity** — they all run (usually in parallel).
4. **Multiple entities can depend on the same query** — each runs its own `combine` call, sharing the query state.

This all happens asynchronously, after the command has returned. Query updates are eventually consistent relative to the write — there's a small delay between "the command was accepted" and "the query reflects it."

<Aside type="note" title="How eventually is 'eventually'?">
In practice, milliseconds. The query subscriber processes events as they arrive, and the processing is local (no network hop). If you need strict read-your-writes consistency for a specific query, you can poll the query endpoint until it shows the expected version — or, for the rare cases where this matters, read from the entity directly in a custom handler.
</Aside>

## Rebuilding queries

Sometimes you change the shape of a query, or fix a bug in the `combine` function, and you want to rebuild it from scratch. NeoHaskell does this automatically on startup: it replays every event through every query, producing fresh read models. This "rebuild phase" runs before the application starts serving traffic, so users always see consistent data.

For large event stores this can take a while. Production systems typically store snapshots of query state and only rebuild when they detect a version mismatch. The details are in the [Guides → Postgres event store](/guides/postgresql-event-store/) page.

## When to use multiple queries

Don't be shy about creating new queries. They're cheap. A few guidelines:

- **One query per view.** If you have a dashboard that shows X, Y, and Z, that's one query returning a record with X, Y, and Z.
- **Separate queries for separate consumers.** The customer dashboard and the fraud dashboard get different queries — they have different shapes, different auth rules, different denormalization strategies.
- **Separate queries for different access patterns.** If you want "all orders by user" and "all orders by date", those are two queries with different keys. The same events power both.

There is no meaningful cost to adding queries beyond a bit of storage and CPU time on updates. Use this generously.

## Authorization on queries

Queries have two auth hooks:

- **`canAccess`** — runs before fetching data. "Can this user access this query type at all?"
- **`canView`** — runs after fetching data. "Can this user see this specific query instance?"

`canAccess` returns `Nothing` to allow, or `Just error` to deny. Common helpers:

- `publicAccess` — anyone can access.
- `authenticatedAccess` — any logged-in user can access.
- `adminOnlyAccess` — only users with the admin claim can access.

`canView` is the same idea but receives the specific query instance. It's where you check ownership:

- `publicView` — anyone can view any instance.
- `ownerOnly (.ownerId)` — only the owner can view this instance.

You write these at the top of the query module, and `deriveQuery` wires them to the `Query` typeclass automatically.

## Pagination

Query endpoints accept standard pagination parameters:

```bash
GET /queries/user-activity?limit=50&offset=100
```

Each query type has a `maxResultsImpl` that caps the page size (default 1000). Override it if you need a smaller cap for a specific query:

```haskell
deriveQuery ''UserActivity [''User, ''Post]

maxResults :: Int
maxResults = 100  -- cap at 100 per page
```
