[Effects and concurrency](/concepts/effects/) covered the `Task` type and the basic primitives. This page is about *when* to reach for each concurrency tool and the patterns that make them work together.

## The tools, in one table

| Tool | Use when… | Key operations |
|---|---|---|
| **`AsyncTask`** | You want to run tasks in parallel | `start`, `wait`, `cancel` |
| **`Channel`** | Producer and consumer need to communicate | `new`, `send`, `receive` |
| **`Lock`** | Multiple tasks need exclusive access to something | `new`, `with` |
| **`ConcurrentVar`** | You have shared mutable state | `new`, `read`, `write`, `modify` |
| **`ConcurrentMap`** | You need a shared key/value store | `new`, `get`, `set`, `delete` |

Don't reach for a primitive until you need it. Most code in a NeoHaskell application is single-threaded from the application's point of view — you write `decide` functions and `combine` functions that are pure, and the runtime handles request parallelism for you.

## Pattern 1 — Parallel fetches

The most common concurrency need is "run these three independent queries at the same time and collect the results."

```haskell
import AsyncTask qualified

buildDashboard :: Uuid -> Task Text Dashboard
buildDashboard userId = do
  userHandle <- AsyncTask.start (fetchUser userId)
  postsHandle <- AsyncTask.start (fetchRecentPosts userId)
  statsHandle <- AsyncTask.start (fetchActivityStats userId)

  user <- AsyncTask.wait userHandle
  posts <- AsyncTask.wait postsHandle
  stats <- AsyncTask.wait statsHandle

  Task.yield Dashboard { user = user, posts = posts, stats = stats }
```

All three `AsyncTask.start` calls launch immediately. The three `AsyncTask.wait` calls block in sequence, but because the tasks are already running, total time is max of the three durations — not the sum.

## Pattern 2 — Fan out, fan in

Running the same task on a list of inputs, in parallel:

```haskell
fetchAllUsers :: Array Uuid -> Task Text (Array User)
fetchAllUsers userIds = do
  handles <- Task.mapArray (\id -> AsyncTask.start (fetchUser id)) userIds
  Task.mapArray AsyncTask.wait handles
```

Each fetch runs independently. If one fails, `Task.mapArray` on the `wait` step propagates the failure.

## Pattern 3 — Timeouts

Wrap a task in a timeout:

```haskell
import Timer qualified

fetchWithTimeout :: Task Text User
fetchWithTimeout = do
  handle <- AsyncTask.start (fetchUser userId)
  result <- Task.recover
    (\_ -> Task.throw "Fetch timed out")
    (Timer.withTimeout (Timer.seconds 5) (AsyncTask.wait handle))
  Task.yield result
```

`Timer.withTimeout` races a timer against the task. Whichever finishes first wins.

## Pattern 4 — Shared counter with ConcurrentVar

When multiple tasks need to update a single value:

```haskell
import ConcurrentVar qualified

countingExample :: Array Int -> Task Text Int
countingExample items = do
  counter <- ConcurrentVar.new 0
  Task.forEach items \item -> do
    ConcurrentVar.modify counter (\n -> n + item)
  ConcurrentVar.read counter
```

`ConcurrentVar.modify` is atomic — two tasks updating the same var cannot lose writes.

<Aside type="caution" title="ConcurrentVar is not your database">
Use `ConcurrentVar` for *in-process* state — caches, counters, coordination flags. For anything that needs to survive a restart, use events and the event store. `ConcurrentVar` is a memory primitive, not a persistence layer.
</Aside>

## Pattern 5 — Producer/consumer with Channel

When one task produces work and another consumes it:

```haskell
import Channel qualified

processJobs :: Task Text Unit
processJobs = do
  jobChannel <- Channel.new

  -- Producer: push new jobs onto the channel
  AsyncTask.start do
    Task.forever do
      job <- fetchNextJobFromApi
      Channel.send jobChannel job

  -- Consumer: pop jobs and process them
  Task.forever do
    job <- Channel.receive jobChannel
    processJob job
```

`Channel.send` doesn't block unless the channel is full (if you created it with a bound). `Channel.receive` blocks until something is available.

Use `DurableChannel` if you need the same pattern but with on-disk persistence — jobs survive a restart.

## Pattern 6 — Mutex with Lock

When you need to serialize access to a resource:

```haskell
import Lock qualified

exclusiveAccess :: Task Text Unit
exclusiveAccess = do
  lock <- Lock.new
  Task.forEach items \item ->
    Lock.with lock do
      -- Only one task at a time can be inside this block
      updateSharedResource item
```

`Lock.with` runs the inner action while holding the lock. It's released automatically when the action finishes or fails.

## Pattern 7 — Per-key locking with ConcurrentMap

Sometimes you want to lock *per key* rather than one global lock. This is common in OAuth token refresh: you don't want two tasks refreshing the same user's token at once, but you're fine with different users refreshing concurrently.

```haskell
import ConcurrentMap qualified
import Lock qualified

refreshTokenFor :: ConcurrentMap Text Lock -> Text -> Task Text Token
refreshTokenFor locks userId = do
  lock <- ConcurrentMap.getOrCreate locks userId Lock.new
  Lock.with lock do
    currentToken <- readCurrentToken userId
    if Token.expired currentToken
      then doRefresh currentToken
      else Task.yield currentToken
```

## Integrations are implicitly concurrent

Most of the time, you don't write concurrency primitives yourself. NeoHaskell's integration runtime already runs integrations concurrently — outbound integrations react to events in parallel, inbound integrations run their own workers, and the command executor handles request-level concurrency.

If you find yourself reaching for `AsyncTask.start` inside a `decide` function, stop. `decide` is pure and can't do async work. What you want is an integration that fires off the work in response to an event. The integration runs concurrently with the rest of the system and can do whatever I/O it needs.

## Debugging concurrency bugs

Concurrency bugs are the worst kind. A few habits that help:

- **Prefer immutable data.** Passing a value between tasks is always safe.
- **Limit shared state.** Every `ConcurrentVar` or `ConcurrentMap` is a potential bug source.
- **Use events for cross-task coordination** when possible. Events are serialized by the event store — you get ordering for free.
- **Wrap tricky sections with `Log.withScope`** so logs from different tasks are distinguishable.
- **Tests are your friend.** Write a test that spawns a hundred concurrent commands and asserts the final state is consistent. If it passes, you're probably fine. If it fails, you've found the bug early.
