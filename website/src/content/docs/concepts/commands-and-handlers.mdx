Commands are where the outside world meets your domain. A user clicks a button, an integration fires, a CLI tool runs — and somewhere along the way, a command object shows up at your `decide` function. This page walks through the full lifecycle of a command in NeoHaskell, so you understand exactly what happens at each step and why.

## The four moving parts

Before the walkthrough, here are the characters in the story:

1. **The command** — a typed record describing what the caller wants.
2. **The `decide` function** — a pure function that turns the command into a `Decision` (accept or reject).
3. **The command executor** — the runtime machinery that runs `decide` safely and stores the resulting events.
4. **The event store** — the append-only log where events live forever.

Everything else (the HTTP transport, the JSON parser, the authorization check, the retries) is plumbing. The four above are the concepts that matter.

## The full lifecycle

Let's follow a single request end to end. Imagine the caller makes this HTTP request:

```http
POST /commands/Deposit HTTP/1.1
Content-Type: application/json

{ "accountId": "abc-123", "amount": 50 }
```

### Step 1 — Transport decodes the request

The HTTP transport receives the request, parses the JSON body into the `Deposit` type, and constructs a `RequestContext` (containing authentication info, user claims, trace IDs, etc.).

This is pure plumbing. You didn't write it. The line `command ''Deposit` at the bottom of your command file generated this for you.

### Step 2 — Executor fetches the current entity

The executor now needs the current state of the account. It calls the `EntityFetcher`, which:

1. Looks up the snapshot cache for this entity ID. If a cached snapshot exists, it uses that as the starting point.
2. Reads any events that have arrived since the snapshot was taken.
3. Folds those events into the snapshot using the entity's `update` function.
4. Returns the current `Account` state (or `Nothing` if no events exist for this ID yet).

This is how `decide` receives a fresh, correct view of the entity without you having to touch the database yourself.

### Step 3 — Decide is called

Your pure `decide` function runs:

```haskell
decide :: Deposit -> Maybe Account -> RequestContext -> Decision AccountEvent
decide cmd account _ctx = case account of
  Nothing ->
    Decider.reject "Account does not exist"
  Just _ ->
    if cmd.amount <= 0
      then Decider.reject "Deposit amount must be positive"
      else
        Decider.acceptExisting
          [ MoneyDeposited
              { entityId = cmd.accountId
              , amount = cmd.amount
              }
          ]
```

It receives the command, the current entity state, and the request context. It returns a `Decision`. That's it. This function is pure — no I/O, no database access, no side effects. You can test it in isolation by passing fake inputs.

The `Decision` is one of:

- **`Accept` + `InsertionType` + events** — "go ahead, persist these events under the given insertion mode."
- **`Reject` + reason** — "no, and here's why."

### Step 4 — Executor persists the events

If `decide` returned an `Accept`, the executor tries to append the events to the entity's stream in the event store. The `InsertionType` controls the concurrency rule:

- **`StreamCreation`** (`acceptNew`) — succeed only if the stream doesn't exist yet. Used for creation commands.
- **`ExistingStream`** (`acceptExisting`) — succeed only if the stream already exists. Used for mutation commands.
- **`InsertAfter position`** (`acceptAfter`) — succeed only if the stream is at exactly this position. Used for optimistic concurrency.
- **`AnyStreamState`** (`acceptAny`) — succeed regardless. Rarely used.

If the insertion succeeds, the events are now part of the event store. Future reads will see them.

### Step 5 — Executor handles concurrency conflicts

If the insertion fails because another command raced ahead and modified the stream, the executor does **not** immediately error. Instead, it retries:

1. Re-fetches the entity (so it gets the latest state including the other command's events).
2. Runs `decide` again with the new state.
3. Tries to append again.

This retry loop uses exponential backoff and caps at a maximum number of attempts (default 10). Because `decide` is pure, running it twice produces the same result for the same inputs — the retry is safe.

The practical effect: your `decide` function never has to think about concurrency. You always see the latest state. If two commands race, they're serialized automatically.

### Step 6 — Transport returns the response

Once the events are persisted (or the command is rejected), the executor returns a `CommandResponse`. The transport serializes it to JSON and sends it back to the caller:

```json
{
  "kind": "Accepted",
  "entityId": "abc-123",
  "events": [
    { "kind": "MoneyDeposited", "entityId": "abc-123", "amount": 50 }
  ]
}
```

Or, on rejection:

```json
{
  "kind": "Rejected",
  "reason": "Account does not exist"
}
```

### Step 7 — Projections and integrations react

This step happens asynchronously and does not block the response. The newly appended events are visible to:

- **Queries (projections).** Every registered query that depends on this entity type is notified. Their `combine` functions run, producing `Update` / `Delete` / `NoOp` actions. The query's read model is updated.
- **Outbound integrations.** Every integration that subscribes to this entity type is notified. Their `reactTo` functions run, potentially emitting more commands or triggering external side effects.

Both of these read from the same event stream and are replayable. You can rebuild a projection from scratch by replaying all events through `combine`. You can test an integration by passing in events and asserting on the actions it produces.

## Why the split matters

Notice how sharply the roles are separated:

- **The command** describes what the caller wants. It's a data record.
- **`decide`** is a pure function that makes a decision. It doesn't know about HTTP, storage, or concurrency.
- **The executor** handles all the impure stuff — fetching, persisting, retrying, serializing.
- **The event store** is append-only and content-agnostic — it just stores bytes.

This separation is what makes NeoHaskell commands testable, replayable, and type-safe. You test `decide` by calling it with mocks — it's a pure function, no framework needed. You test the executor via integration tests. You trust the event store because it's just an append log.

## What `decide` should do

`decide` should answer one question: **given the current state, what should happen next?**

It should:

- Read the command.
- Read the entity state (or notice it doesn't exist).
- Apply business rules (using the fields of both).
- Return either `Accept` with events or `Reject` with a reason.

It should **not**:

- Talk to external systems (that's what integrations are for).
- Access databases (the state has already been loaded for you).
- Do random I/O (it must be pure for the retry logic to work).
- Throw exceptions on business errors (use `Decider.reject` instead).

## The `RequestContext` parameter

The third argument to `decide` is a `RequestContext`. It contains information about the caller — most notably authentication claims. You use it to make authorization decisions:

```haskell
decide :: Withdraw -> Maybe Account -> RequestContext -> Decision AccountEvent
decide cmd account ctx = case ctx.user of
  Nothing ->
    Decider.reject "Authentication required"
  Just user ->
    case account of
      Nothing -> Decider.reject "Account does not exist"
      Just acc ->
        if Uuid.toText acc.ownerId /= user.sub
          then Decider.reject "You do not own this account"
          else
            -- ... proceed
```

Authorization is *just code in `decide`*. There is no middleware layer, no annotation, no magic. If `ctx.user.sub` doesn't match the account's owner, you reject.

## Multi-tenant commands

If your system has tenants, you can mark a command as multi-tenant:

```haskell
instance Command CreateCart where
  type IsMultiTenant CreateCart = 'True
  getEntityIdImpl tenantId cmd = ...
  decideImpl tenantId cmd entity ctx = ...
```

In multi-tenant commands, both `getEntityId` and `decide` receive an extra `Uuid` argument representing the tenant. The framework routes events to per-tenant streams automatically — tenant isolation is enforced at the event store layer, not by your code remembering to filter.

## Rejections are not errors

A `Decider.reject` is not an exception. It's a first-class outcome. The caller gets a `Rejected` response, but nothing about the system is broken. This is important:

- **You don't need try/catch.** Rejection is part of the type signature.
- **You don't pollute logs.** Rejections are expected outcomes, not errors.
- **You can reason about rejection reasons.** They're just strings; log them, show them to the user, use them for metrics.

Reserve actual errors (thrown exceptions, `Task.throw`) for situations where the system itself has a problem — database connection lost, serialization bug, etc.

## Testing commands

Because `decide` is pure, testing commands is trivial:

```haskell
spec :: Spec Unit
spec = describe "Deposit" do
  it "rejects when account does not exist" \_ -> do
    let cmd = Deposit { accountId = someId, amount = 50 }
    let result = runDecision defaultContext (decide cmd Nothing defaultAuth)
    result |> shouldBe (RejectCommand "Account does not exist")

  it "accepts valid deposits" \_ -> do
    let account = Just (Account { ..., balance = 0 })
    let cmd = Deposit { accountId = someId, amount = 50 }
    let result = runDecision defaultContext (decide cmd account defaultAuth)
    case result of
      AcceptCommand _ events ->
        events |> shouldBe (Array.fromLinkedList
          [MoneyDeposited { entityId = someId, amount = 50 }])
      _ -> Test.fail "Expected accept"
```

No database, no HTTP server, no mocks. You construct the inputs, you call `decide`, you assert on the outputs.
