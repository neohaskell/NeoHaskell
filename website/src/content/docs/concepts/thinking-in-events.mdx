There is one idea at the center of NeoHaskell. Get this idea and the rest of the documentation will feel obvious. Miss it, and everything will feel strange and arbitrary.

**The idea:** state is a lie we tell ourselves. What really exists is a sequence of things that happened. State is just the current snapshot we compute from those things.

## The bank you already know

You already believe this idea. You just haven't applied it to code yet.

When the bank shows you a balance of $1,247.32, you don't think the bank has a little box somewhere labeled "Alice's balance" that it increments and decrements. You know — because banks have been around for a thousand years — that what the bank actually has is a **ledger**. A list. Every deposit, every withdrawal, every transfer, every interest accrual, every fee. In order. With dates. Never changed, never deleted.

When you want your balance, the bank sums up the list. That's what "your balance" is — the result of a function over the ledger.

This is not a quirk of banking. It is how banking works because it is how *accounting works*. And accounting has this shape for a reason: if you ever need to answer a question like "wait, how did the balance get to $1,247.32 — which deposit landed first, and was the interest calculated correctly?", the ledger has the answer. A `balance` column in a database does not.

## The CRUD world lost this

Somewhere along the way, database-first software engineering talked itself out of the ledger. Instead of recording "$50 was deposited at 3:14pm on Tuesday", we wrote:

```sql
UPDATE accounts SET balance = balance + 50 WHERE id = 'abc';
```

The balance changed. The *reason* is gone. The moment is gone. The old value is gone. All that remains is the current number. Next week, when someone asks "what was the balance before the dispute?", the honest answer is: *we don't know.*

CRUD code is built on the idea that "the current state" is the thing worth keeping. Events are temporary, lost, overwritten. This is backwards. **Events are facts about the world. State is a derived cache of those facts.** CRUD treats the cache as the truth and throws away the truth.

## Flipping it

Event sourcing is the simple act of flipping the relationship. Events become the truth. State becomes the cache.

- **Events are facts.** Immutable. Append-only. Never modified. "`$50 was deposited`" happened — there's nothing to update.
- **State is derived.** You compute it by folding events. You can compute it at any point in time. You can throw the cache away and recompute it. The events are still there.

Everything in NeoHaskell follows from this one flip:

- **You don't update a field.** You append an event.
- **You don't read a field.** You compute one from events (through a projection or an entity fold).
- **You don't worry about history.** You have it by construction.
- **You don't write migrations.** Your schema is a fact type; to change it, add a new fact type.
- **You don't write audit logs.** The events *are* the audit log.

## What an event is

An event in NeoHaskell is a **statement about the world that is true in the past tense**. It describes something that *happened*.

Good event names:

- `MoneyDeposited`
- `AccountOpened`
- `OrderShipped`
- `UserLoggedIn`
- `InvoicePaid`
- `TemperatureExceeded`

Bad event names:

- `UpdateBalance` (imperative, not a fact)
- `ValidateUser` (not a fact, not past tense)
- `ProcessOrder` (vague, implementation-speak)
- `DoThing` (nothing in the real world was affected)

The heuristic: *if two humans could disagree about whether the event occurred, it's probably a bad event*. `MoneyDeposited { amount = 50 }` is objective — either $50 was deposited or it wasn't. `UserValidated` is not — what does "validated" mean? By whom? Under what criteria?

## Events are not messages

This is the single most common confusion. Events and messages look the same on the wire — both are typed records of data. But they play completely different roles:

| Property | Event | Message |
|---|---|---|
| **Purpose** | Record that something happened | Request that something happen |
| **Lifetime** | Permanent | Ephemeral |
| **Subject** | Past tense | Present or future tense |
| **Owner** | The domain | The infrastructure |
| **Mutability** | Never changes | Discarded after handling |

When you see NeoHaskell code emitting events, do not think "I'm sending a message to another service." Think "I'm writing a line in the ledger." The event goes into the event store. It is there forever. Any reader, now or ten years from now, can read it.

Integrations *might* react to events by sending messages to external services (HTTP calls, emails, webhooks). The event itself is not the message. The event is the *reason* a message might be sent.

## Events are not logs

The other common confusion is: "so events are basically log lines?"

No. Logs are for debugging. You can delete a log, filter it, drop it when disk fills up, and your system still works. If you delete an event, you delete part of the truth about what happened. Replay a year later, and the balance will be wrong — because the event that would have added $50 is gone.

Events are the primary store. Logs are a secondary aid. Never confuse them.

## The mental shift in practice

Here is how the shift shows up in practice. Take any piece of CRUD code and ask: "what happened?"

- A user clicks "add to cart." **What happened?** `ItemAddedToCart`. Record it.
- A user pays. **What happened?** `PaymentReceived`. Record it.
- The payment fails. **What happened?** `PaymentFailed { reason = ... }`. Record it.
- An admin refunds the order. **What happened?** `OrderRefunded`. Record it.

Every "what happened" becomes an event. The state of the cart, the payment, the order — *all of it* — is a function you can compute from that event list at any time.

The initial mental adjustment is asking "what happened?" instead of "what do I want the database to look like now?" Once that becomes reflex, you are thinking in events.

## A common objection

*"But I need to know the current state to validate things. Like, I can't just accept a withdrawal without knowing the current balance."*

Right. You need the current state *to decide*. That's what the `decide` function in a NeoHaskell command is for. It receives the current entity state (derived from events) and makes a decision.

The trick is: you **do not mutate** the state to express "the balance went down." You emit an event — `MoneyWithdrawn` — and the `update` function re-derives the new state from the old state plus the event. You read the current state, you decide based on it, you write events, state re-derives. You never pick up the state and modify it in place.

## A second common objection

*"This sounds like it will be slow. Folding a million events to compute a balance is expensive."*

Two answers:

1. **You don't do it every time.** Entities are cached after they're reconstructed. If the cache is there, you start from the cache and only apply the events since the cache was taken. NeoHaskell handles this automatically via snapshot caches.
2. **You build read models (queries) that are maintained incrementally.** A "current balance" read model updates in-place as events arrive, so reading the balance is a single-row lookup — same cost as CRUD.

The event log is the source of truth. Reads are answered by cached projections. You get both the fidelity of event sourcing and the speed of denormalized reads.

## A third common objection

*"What if the event schema needs to change?"*

Events are immutable facts; you don't modify them. If the shape needs to change, you introduce a *new* event type. Old events stay in the store. The `update` function handles both old and new types until all the old data has aged out (if ever). This is covered in full in [Schema evolution](/concepts/schema-evolution/).
