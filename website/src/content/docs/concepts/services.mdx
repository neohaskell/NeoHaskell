A **service** in NeoHaskell is a bundle of commands that all operate on the same entity and event type. It's the smallest unit of transactional consistency in your system. Understanding services is the bridge between "I have some commands" and "I have a system that can be deployed, split up, and composed with others."

## The mechanical definition

Syntactically, a service is built like this:

```haskell
service :: Service _ _
service =
  Service.new
    |> Service.command @CreateCart
    |> Service.command @AddItem
    |> Service.command @RemoveItem
    |> Service.command @Checkout
```

All of those commands must share the same entity type and the same event type. The compiler enforces this with a specific error message if you try to mix types:

```text
Event type mismatch in service definition!

All commands in a service must use the same event type.

Expected event type: CartEvent
But command 'ProcessPayment' uses event type: PaymentEvent
```

This is deliberate. A service is a **consistency boundary** — the set of commands that can be ordered and coordinated against a single state machine. Putting commands from different boundaries in the same service is a modeling mistake, and the compiler catches it.

## The conceptual definition

Services are the smallest unit of your system that can be **independently consistent**. That's a mouthful, so let's unpack it:

- **Consistent** means "you can reason about the order of events within it and guarantee invariants." If your business rule says "a cart can have at most 50 items", the cart service is the layer that enforces it.
- **Independently** means "you don't need to coordinate with any other service to enforce your invariants." The cart service can accept or reject `AddItem` without consulting the payment service or the inventory service.

Two different services might be *related* (the cart service produces events that the inventory service cares about) but they are not *linked* (neither one blocks on the other to make a decision). Coordination between services is the job of [integrations](/concepts/commands-and-handlers/).

## Why consistency boundaries matter

In traditional systems, the consistency boundary is "the whole database." A single transaction can update any rows it wants; ACID guarantees hold across all of them. This is simple but doesn't scale — as your system grows, you end up with giant transactions that lock huge swaths of data and serialize everything.

In event sourcing, the consistency boundary is the *entity stream*. A single stream is atomic and sequential; events within it are strictly ordered; invariants can be enforced cheaply. But events in *different* streams are independent — they can be appended in parallel, they don't need coordination.

A service groups commands that share a stream, which means they share a consistency boundary. It's the unit at which you can say "these things are consistent with each other."

## Service ≠ microservice

A common confusion: "NeoHaskell has services, does that mean I have to build microservices?"

No. A service in NeoHaskell is a *code boundary*, not a *deployment boundary*. You can have ten services in a single NeoHaskell application, all running in the same process, all sharing the same event store. They're still independent consistency boundaries — they just happen to be colocated.

When you grow to the point where one service needs to scale independently, you split it out into its own deployment. That's a deployment decision, not a rewrite. The code doesn't change — you take the same service module and host it in a separate application.

This is what "start as a monolith, split when you need to" actually means in NeoHaskell. The boundaries are already there; splitting is just a matter of wiring.

## A worked example

Let's build a tiny application with two services: `Cart` and `Inventory`. These are obviously related — when you add an item to a cart, you should decrement available inventory — but they are independent consistency boundaries.

```haskell
-- Cart service
module MyApp.Cart.Service (service) where

import MyApp.Cart.Commands.CreateCart (CreateCart)
import MyApp.Cart.Commands.AddItem (AddItem)
import MyApp.Cart.Commands.Checkout (Checkout)
import Service qualified

service :: Service _ _
service =
  Service.new
    |> Service.command @CreateCart
    |> Service.command @AddItem
    |> Service.command @Checkout
```

```haskell
-- Inventory service
module MyApp.Inventory.Service (service) where

import MyApp.Inventory.Commands.StockItem (StockItem)
import MyApp.Inventory.Commands.ReserveItem (ReserveItem)
import Service qualified

service :: Service _ _
service =
  Service.new
    |> Service.command @StockItem
    |> Service.command @ReserveItem
```

In the application wiring, both services run in the same process:

```haskell
app :: Application
app =
  Application.new
    |> Application.withEventStore ...
    |> Application.withTransport WebTransport.server
    |> Application.withService Cart.service
    |> Application.withService Inventory.service
    |> Application.withOutbound @ReserveOnAddItem  -- integration that bridges them
```

The `ReserveOnAddItem` integration listens for `ItemAdded` events on the cart service and emits `ReserveItem` commands on the inventory service. Neither service knows about the other directly — the integration is the only connection.

If you later need to split inventory into its own deployment, you take `MyApp.Inventory.Service` and put it in a separate application. The integration becomes an HTTP call instead of an in-process dispatch. The cart service doesn't change at all.

## Services don't have to be big

A service with a single command is fine. A service with twenty commands is fine. The right size is "however many commands share the same consistency boundary." If you find yourself wanting to add a command to a service but its event type doesn't match, that's a signal: you're conflating two boundaries, and the compiler will tell you.

## Common modeling mistakes

### Treating "feature" as "service"

A "shopping" service with commands for users, carts, products, and orders is almost certainly wrong. Those are four different consistency boundaries. Split them.

### Splitting by layer, not by domain

Don't create a `Validation` service or a `Notifications` service. Services are domain boundaries. Validation goes in `decide` functions; notifications are integrations. Those are not services.

### One entity type, many services

If you find yourself wanting two services that both operate on the same entity type, merge them. Consistency boundaries are defined by the entity, not by the organization of your source tree.

### One service, many entity types

The compiler will stop you, but the underlying mistake is trying to enforce invariants across multiple streams. That's not what services are for — that's what the saga pattern (via integrations) is for.

## Services and multi-tenancy

In multi-tenant systems, each tenant has its own stream per entity. A single service still serves all tenants, but the streams are isolated at the event store level. Commands declare `IsMultiTenant = 'True`, and the framework automatically scopes reads and writes to the right tenant.

Tenant isolation is a *property of the event store*, not of the service. All tenants share the same service code; only their data is separated.

## How services connect to transports

A service does not know or care which transport carries its commands. You can attach a `WebTransport` to expose commands over HTTP, a `Cli` transport to expose them as CLI subcommands, an `Mcp` transport to expose them to AI assistants — or all three at once.

```haskell
-- In the command file:
type instance TransportsOf CreateCart = '[WebTransport, Cli]
```

This command is now available over both HTTP and the CLI. Same `decide` function. Same events. Zero changes to the service definition.

## The short version

- A **service** is a bundle of commands sharing an entity and event type.
- It's the smallest unit of independent consistency.
- You can have many services in one application.
- Services are a code boundary, not a deployment boundary — splitting into microservices is a later, separate decision.
- Cross-service coordination happens through integrations, not direct calls.
