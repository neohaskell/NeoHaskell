If you already know GHC Haskell, NeoHaskell will feel like a curated, opinionated reinterpretation of what you already know. Most of the syntax is identical — it compiles with GHC, after all. The differences are in the standard library, the conventions, and the philosophy. This page is the complete diff.

## The core philosophy difference

Haskell's philosophy is "solve whatever problem you have; the type system will help." NeoHaskell's philosophy is "most business systems need the same patterns; let's bake those in and make the common case trivial."

This means NeoHaskell is **more opinionated** than Haskell. It has a specific style, a specific standard library, a specific way of doing effectful computation, and a specific set of domain primitives (commands, events, entities, queries, services, integrations) baked into the type system. You give up some flexibility; you get a much shorter ramp and a tighter feedback loop.

NeoHaskell is not Haskell with fewer features. It is Haskell with **different defaults** — defaults chosen to make newcomers productive fast and experienced developers write fewer bugs.

## Standard library replacements

The most visible change. NeoHaskell replaces or re-exports many `base` types and functions:

| Haskell (base) | NeoHaskell | Notes |
|---|---|---|
| `IO a` | `Task err val` | Typed error channel. |
| `Either a b` | `Result err ok` | Renamed with explicit parameters. |
| `Maybe a` | `Maybe a` | Same. |
| `String` | `Text` | Strict, UTF-8, no linked lists. |
| `[a]` (user-facing) | `Array a` | Efficient persistent sequence. |
| `Data.Map.Strict.Map` | `Map k v` | Re-exported and renamed. |
| `pure` / `return` | `Task.yield` / `Result.ok` | No confusion about which "pure" you mean. |
| `>>=` | `|>` with combinators (and `do` notation) | Same semantics, different syntax. |
| `<$>`, `fmap` | `Mappable.map` / type-specific `map` | Same idea, specific forms. |
| `$` | `|>` | Pipes over backwards application. |
| `throwIO` | `Task.throw` | Typed errors. |
| `bracket` | `Task.finally` | Scoped cleanup. |

You can still use `base` when necessary — the convention is to import it with a `Ghc` prefix:

```haskell
import Data.List qualified as GhcList
```

This visually flags "I'm reaching outside the NeoHaskell standard library." If you see `GhcList.something` in someone's code, it's a deliberate escape hatch.

## Syntax conventions

Same compiler, different style. NeoHaskell enforces these through the style guide and `hlint`:

### Pipes over `$`

```haskell
-- Haskell
print $ map toUpper $ filter isAlpha text

-- NeoHaskell
text
  |> Text.filter Char.isAlpha
  |> Text.toUpper
  |> Console.print
```

The pipe reads top-to-bottom; `$` reads inside-out. NeoHaskell prefers the former unconditionally.

### `do` blocks, not `let..in`

```haskell
-- Haskell
computeResult :: Int -> Int
computeResult x =
  let
    doubled = x * 2
    squared = x * x
  in
    doubled + squared

-- NeoHaskell
computeResult :: Int -> Int
computeResult x = do
  let doubled = x * 2
  let squared = x * x
  doubled + squared
```

Inside a `do` block, NeoHaskell uses `let` as a statement, not `let..in`. Outside a `do` block, you'd write small functions instead.

### No `where` clauses

```haskell
-- Haskell
go :: Int -> Int
go n = helper + 1
  where helper = n * 2

-- NeoHaskell
helper :: Int -> Int
helper n = n * 2

go :: Int -> Int
go n = helper n + 1
```

NeoHaskell lifts helper functions to the top level. This feels verbose at first, but it keeps them visible and reusable.

### Case expressions, not pattern-matched function clauses

```haskell
-- Haskell
handle :: Maybe User -> Text
handle Nothing = "none"
handle (Just u) = u.name

-- NeoHaskell
handle :: Maybe User -> Text
handle maybeUser = case maybeUser of
  Nothing -> "none"
  Just u -> u.name
```

One function, one body. Pattern matching lives inside the body via `case`.

### String interpolation with `fmt`

```haskell
-- Haskell
greet name = "Hello, " ++ name ++ "!"

-- NeoHaskell
greet name = [fmt|Hello, #{name}!|]
```

Never use `++` for strings in NeoHaskell. Use `[fmt|...|]` everywhere.

### Qualified imports by default

```haskell
-- Haskell
import qualified Data.Text as T
import qualified Data.Map.Strict as Map

-- NeoHaskell
import Text qualified
import Map qualified
```

The NeoHaskell style is `import <Module> qualified` without an alias — you refer to `Text.trim`, `Map.get`, etc. Modules are named to be ergonomic without aliasing.

### Descriptive type variable names

```haskell
-- Haskell
compose :: (b -> c) -> (a -> b) -> a -> c

-- NeoHaskell
compose :: (b -> c) -> (a -> b) -> a -> c
-- ...actually the same in this case, but for multi-parameter generic functions:

-- Haskell
transform :: m a -> (a -> m b) -> m b

-- NeoHaskell
transform :: monad input -> (input -> monad output) -> monad output
```

NeoHaskell uses descriptive type variable names in domain code (`input`, `output`, `entity`, `event`). Single-letter names are allowed in truly polymorphic helper functions where no domain name fits.

## Features NeoHaskell deliberately avoids

These are valid Haskell features that NeoHaskell does not use in application code. They may appear in the framework internals, but you should not use them in your own services.

### Monad transformers

Standard Haskell uses `StateT`, `ReaderT`, `ExceptT`, etc., stacked on top of `IO`. NeoHaskell uses `Task err val` as the sole effect type, with explicit record-passing for environment.

**Why:** Monad transformer stacks are hard to teach, have confusing error messages, and lock you into one library's conventions. `Task` is one type; it's easier to reason about.

### Template Haskell (except for framework helpers)

The only TH NeoHaskell uses in user code is:

- `command ''Foo` — generates command wiring.
- `deriveQuery ''Foo [''Bar, ''Baz]` — generates query wiring.
- `defineConfig "MyConfig" [...]` — generates configuration types.

Everything else — custom splices, lenses-via-TH, quasiquoters other than `[fmt|...|]` — is avoided.

**Why:** Template Haskell slows compilation, breaks tooling, and makes error messages opaque. We limit it to the few places where the benefit is overwhelming.

### Lens and optics

NeoHaskell uses **record dot notation** (`user.name`) and **record update** (`user { name = "Alice" }`) instead of the `lens` library.

**Why:** Lenses are powerful but introduce a steep learning curve and a new mental model. Record dot notation does 90% of what lens does with zero extra concepts.

### `where` clauses

Already covered above. Use top-level helpers or `let` inside `do`.

### Point-free style

```haskell
-- Haskell (point-free)
countEvens = length . filter even

-- NeoHaskell
countEvens :: Array Int -> Int
countEvens xs = xs |> Array.filter Int.isEven |> Array.length
```

Point-free is elegant until you have to debug it. NeoHaskell prefers explicit parameters.

### Custom operators

NeoHaskell sticks with `|>`, `<|`, standard arithmetic, and occasionally `<$>` for framework code. You are not encouraged to define custom operators in application code.

**Why:** Custom operators are a cognitive tax for every reader.

### `return` and `pure`

Both exist. Both are banned in NeoHaskell code. Use `Task.yield` instead.

**Why:** "Return" is confusing (it's not the C `return`). "Pure" is confusing (it's overloaded with "pure function"). `Task.yield` is unambiguous and reads correctly out loud.

## Event sourcing primitives

The big addition. NeoHaskell's service layer adds first-class types for event sourcing:

- `Command` — typeclass for commands.
- `Entity` — typeclass for entities (thing-with-state).
- `Event` — typeclass for events.
- `Query` / `QueryOf` — typeclasses for read models.
- `Decision` — free monad for command decision logic.
- `Task` — effectful computation with typed errors.
- `Service` / `Application` — builder patterns for assembling systems.
- `OutboundIntegration` / `Inbound` — cross-service coordination.

None of these exist in standard Haskell. You build them yourself with libraries, and every library has a different opinion. NeoHaskell bakes them into the standard library so everyone uses the same conventions.

## Compiler extensions enabled by default

NeoHaskell enables a consistent set of extensions in `fourmolu.yaml` / `cabal.project`. You don't manage them per-file:

- `OverloadedRecordDot` — for `user.name` syntax.
- `OverloadedStrings` — for `[fmt|...|]`.
- `GeneralizedNewtypeDeriving`
- `DeriveGeneric`
- `DuplicateRecordFields`
- `NoFieldSelectors`
- `DataKinds` (for the type-level symbol machinery)
- `TypeFamilies`

If you're used to adding a wall of `{-# LANGUAGE ... #-}` pragmas at the top of every file, you'll find NeoHaskell refreshingly quiet.

## What stays the same

Let's be clear about what you *don't* have to relearn:

- **The type system.** Same as GHC Haskell. Same generics, same kinds, same typeclass dispatch.
- **Pattern matching.** Same syntax.
- **Data types.** `data`, `newtype`, records — identical.
- **Type classes.** Same mechanism, slightly different style.
- **`do` notation.** Same.
- **Type signatures.** Same.

If you can write GHC Haskell, you can write NeoHaskell in an hour.

## When to use the escape hatch

NeoHaskell is opinionated, but it's not a prison. Sometimes you genuinely need something from the Haskell ecosystem that NeoHaskell doesn't wrap. In those cases:

1. Import it with a `Ghc` prefix: `import Data.Something qualified as GhcSomething`.
2. Wrap the parts you need in a small adapter module.
3. Don't let the escape hatch spread — keep it contained.

This pattern keeps the NeoHaskell code pure while letting you reach into the broader ecosystem when needed.

## Things you'll like

- **Compile times** are similar to regular Haskell (not faster, not slower).
- **Error messages** are customized for the event-sourcing primitives, which means better errors than stock GHC for those specific mistakes.
- **Testing** is trivially easy because the framework forces pure functions into the core of your logic.
- **Deployment** is just a binary. No runtime, no interpreter, no container orchestration required (though you can use those if you want).

## Things you'll miss

- **Some libraries** don't exist in the NeoHaskell ecosystem yet. You'll occasionally wrap a `base` library to use it.
- **Monad transformers** are hard to avoid in some advanced patterns. NeoHaskell doesn't give you clean alternatives for every StateT/ReaderT use case.
- **Advanced type-level programming** is possible but discouraged in user code.
- **The Haskell community's tutorials** don't always translate — a lot of them assume `IO`, `String`, monad transformers, and lens.
