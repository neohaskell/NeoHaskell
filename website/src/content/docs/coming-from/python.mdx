## In Python, you'd write...

- Django models as the source of truth: `class Account(Model): balance = IntegerField()`
- ORM updates that mutate state directly: `account.balance = 150; account.save()`
- Flask/Django views that handle HTTP requests and return responses
- Decorators for cross-cutting concerns: `@login_required`, `@cache_page`
- Type hints as optional annotations: `def get_user(id: int) -> User:`
- List comprehensions: `[x * 2 for x in items if x > 0]`
- `try/except` for error handling â€” exceptions propagate up the call stack
- `requirements.txt` / `pyproject.toml` for dependencies
- `None` as the absence of a value â€” `if user is None:`

## In NeoHaskell, the equivalent is...

- **Events instead of model updates**: instead of `account.balance = 150`, you record `MoneyDeposited { amount = 50 }`
  - The balance is never stored directly â€” it's computed by replaying events
  - `account.save()` doesn't exist â€” events are appended, never overwritten
- **Command handlers instead of views**: Flask routes become command handlers
  - `@app.route('/deposit', methods=['POST'])` â†’ a `Deposit` command type + a `decide` function
  - The handler doesn't touch the database â€” it validates the command and returns events
- **Queries instead of ORM queries**: `Account.objects.get(id=id)` â†’ a `Query` that folds over events
  - Queries are pure functions â€” no database calls, no side effects
  - They're fast because they read from in-memory updates, not live SQL
- **Full type system instead of type hints**: types aren't optional annotations â€” they're enforced
  - `def get_user(id: int) -> User:` can still return `None` at runtime
  - NeoHaskell's `getUser :: UserId -> Task NotFound User` means it either returns a `User` or a `NotFound` error â€” nothing else
- **`Array.map`/`Array.filter` with pipes instead of list comprehensions**:
  - `[x * 2 for x in items if x > 0]` â†’ `items |> Array.filter (\x -> x > 0) |> Array.map (\x -> x * 2)`
  - More verbose, but composable and type-checked at each step
- **Traits instead of decorators**: cross-cutting behavior is expressed through the type system, not decoration
  - `@login_required` â†’ the command type itself carries authentication requirements
- **`Maybe` instead of `None`**: absence of a value is explicit in the type
  - `if user is None:` â†’ `case user of { Nothing -> ...; Just u -> ... }`
  - The compiler forces you to handle the missing case

## But here's what's actually different...

- **The ORM is the thing to unlearn**: Django's ORM trains you to think of the database as the source of truth
  - In NeoHaskell, the event log is the source of truth â€” the database is a derived view
  - This isn't a performance optimization â€” it's a different model of what "data" means
- **State is computed, not stored**: `account.balance` doesn't exist as a field you read
  - Balance is the result of folding all `MoneyDeposited` and `MoneyWithdrawn` events
  - This means you can always ask "what was the balance at 3pm on Tuesday?" â€” for free
- **Errors are part of the function signature**: Python's `try/except` hides what can go wrong
  - A function that raises `ValueError` or `DatabaseError` doesn't say so in its signature
  - NeoHaskell's `Task DatabaseError Account` tells you exactly what can fail before you call it
- **No mutable global state**: Django's `settings.py`, thread-local storage, `g` in Flask â€” these patterns don't exist
  - Configuration is passed explicitly through the type system
  - This makes testing trivial: no setup/teardown, no mocking global state
- **Decorators vs types**: Python decorators are runtime behavior. NeoHaskell's type system is compile-time.
  - `@login_required` can be forgotten. A command type that requires authentication can't be called without it.

## Quick translation table

| Python / Django | NeoHaskell | Notes |
|----------------|-----------|-------|
| `class Account(Model)` | `data Account = Account { ... }` | Record type, not a database model |
| `account.balance = 150; account.save()` | `MoneyDeposited { amount = 50 }` | Record event, don't mutate state |
| `Account.objects.get(id=id)` | `Query` over event stream | Pure function, no DB call |
| `@app.route('/deposit', methods=['POST'])` | `Deposit` command + `decide` function | Handler returns events, not responses |
| `def f(x: int) -> User:` | `f :: Int -> Task Err User` | Error type is explicit |
| `[x for x in items if x > 0]` | `Array.filter (\x -> x > 0) items` | Pipe-composable |
| `try/except ValueError` | `Result ValueError val` | Errors are values |
| `None` | `Maybe val` | Compiler forces you to handle absence |
| `@login_required` | Type-level auth requirement | Compile-time, not runtime |
| `requirements.txt` | Nix + Hix | Reproducible builds |

<Aside type="tip" title="The Django ORM insight">
Django's ORM is the hardest thing to unlearn. It trains you to think "update the record." NeoHaskell trains you to think "record what happened." The balance isn't a field you set â€” it's the answer to "what do all the deposit and withdrawal events add up to?"
</Aside>
