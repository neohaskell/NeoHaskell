If you write Python (Django, Flask, FastAPI, or anything else), NeoHaskell will feel unfamiliar at the syntactic level but surprisingly clean at the conceptual level. Most of what you already know about clean code — dataclasses, type hints, separation of concerns — maps directly onto NeoHaskell patterns. This page draws the lines.

## The big-picture mapping

| Python | NeoHaskell | Notes |
|---|---|---|
| `async def f() -> T:` | `f :: Task err val` | Async with typed error channel. |
| `await x` | `x <- someTask` (inside `do`) | Same idea. |
| `raise ValueError("msg")` | `Task.throw "msg"` | Typed, not untyped. |
| `try/except` | `Task.recover` | Typed recovery. |
| `@dataclass` | `data Foo = Foo { a :: Int, ... }` | Product type. |
| `Enum` / `StrEnum` | `data Color = Red | Green | Blue` | Sum type. |
| `Union[A, B]` or `A | B` | `data Either' = L A | R B` | Discriminated union. |
| `Optional[T]` | `Maybe T` | No `None` hiding anywhere. |
| `list[T]` | `Array T` | Backed by efficient sequence. |
| `dict[K, V]` | `Map K V` | Immutable, persistent. |
| `with ctx:` | `Lock.with`, `Task.finally` | Resource scoping. |
| `for x in xs: ...` | `Array.forEach`, `Task.forEach` | Iteration. |
| Comprehension `[f(x) for x in xs]` | `xs |> Array.map f` | Same thing. |
| `if x: ...` | `if x then ... else ...` | Same. |
| Decorator `@require_login` | `RequestContext` check in `decide` | Different shape, same goal. |
| Django model | Entity (`data Foo = Foo { ... }`) | Domain object. |
| Django view | Command (`decide` function) | Handler. |
| Django serializer | Auto-generated from the command type | No need to write by hand. |
| Django URL | Auto-generated from the command name | No need to write by hand. |
| SQLAlchemy query | Query (`deriveQuery`) | Read model. |
| Celery task | Integration (inbound or outbound) | Async worker. |

The core insight: **Python gives you tools. NeoHaskell gives you types that guarantee the tools were used.** You still write business logic in familiar-looking code — pattern matching, records, if/else — but the compiler enforces that every piece is consistent.

## A side-by-side example

A typical Django view that creates a user:

```python
from django.shortcuts import get_object_or_404
from rest_framework.response import Response
from rest_framework.views import APIView

class CreateUserView(APIView):
    def post(self, request):
        email = request.data.get("email")
        if not email:
            return Response({"error": "Email is required"}, status=400)
        if "@" not in email:
            return Response({"error": "Invalid email"}, status=400)
        if User.objects.filter(email=email).exists():
            return Response({"error": "Email in use"}, status=409)
        user = User.objects.create(email=email)
        return Response({"id": user.id, "email": user.email})
```

The NeoHaskell equivalent:

```haskell
module MyApp.User.Commands.CreateUser (
  CreateUser (..),
  getEntityId,
  decide,
) where

import Core
import Decider qualified
import Json qualified
import Service.Auth (RequestContext)
import Service.Command.Core (TransportsOf)
import Service.CommandExecutor.TH (command)
import Service.Transport.Web (WebTransport)
import MyApp.User.Core
import Text qualified


data CreateUser = CreateUser { email :: Text }
  deriving (Generic, Show)

instance Json.FromJSON CreateUser


getEntityId :: CreateUser -> Maybe Uuid
getEntityId _ = Nothing


decide :: CreateUser -> Maybe User -> RequestContext -> Decision UserEvent
decide cmd existing _ctx = case existing of
  Just _ -> Decider.reject "Email in use"
  Nothing ->
    if Text.isEmpty cmd.email
      then Decider.reject "Email is required"
      else if not (Text.contains "@" cmd.email)
        then Decider.reject "Invalid email"
        else do
          userId <- Decider.generateUuid
          Decider.acceptNew
            [ UserRegistered
                { entityId = userId
                , email = cmd.email
                }
            ]


type instance EntityOf CreateUser = User
type instance TransportsOf CreateUser = '[WebTransport]

command ''CreateUser
```

The two files do the same thing. Differences:

- No URL routing — `command ''CreateUser` generates it automatically.
- No serializer — the type IS the serializer.
- Validation lives in one place (`decide`), not spread across multiple layers.
- Status codes are implicit — acceptance becomes 200, rejection becomes 400, system failures become 500.
- The event is the record of what happened — you can't forget to log it, because the event store always records it.

## Dataclasses vs. data types

Python's `@dataclass` and NeoHaskell's `data` declarations do the same job:

```python
from dataclasses import dataclass

@dataclass
class User:
    id: str
    name: str
    email: str
```

```haskell
data User = User
  { id :: Uuid
  , name :: Text
  , email :: Text
  }
  deriving (Generic, Show)
```

The NeoHaskell version is a little more verbose (you have to derive `Generic` for JSON support, for instance) but the concept is identical. A dataclass is a record type. That's it.

## Type hints

Python type hints are optional — the runtime doesn't enforce them. NeoHaskell types are mandatory and enforced at compile time. If you've been using `mypy` or `pyright` with strict settings, you already have the mental model — NeoHaskell just removes the option of ignoring it.

```python
# Python with type hints
def add(a: int, b: int) -> int:
    return a + b
```

```haskell
-- NeoHaskell
add :: Int -> Int -> Int
add a b = a + b
```

Same thing. NeoHaskell's type annotation is on its own line above the function; Python's is inline with the parameters.

## Async

Python `asyncio` and NeoHaskell `Task` are conceptually the same. Both represent computations that will eventually produce a value and might fail along the way.

```python
async def load_dashboard(user_id: str) -> Dashboard:
    user = await fetch_user(user_id)
    posts = await fetch_posts(user.id)
    stats = await fetch_stats(user.id)
    return Dashboard(user=user, posts=posts, stats=stats)
```

```haskell
loadDashboard :: Uuid -> Task Text Dashboard
loadDashboard userId = do
  user <- fetchUser userId
  posts <- fetchPosts user.id
  stats <- fetchStats user.id
  Task.yield Dashboard { user = user, posts = posts, stats = stats }
```

`await` becomes `<-`. `return` becomes `Task.yield`. The `do` block is the body of an async function.

NeoHaskell's error channel is typed. If the function returns `Task Text User`, you know the error is a `Text`. In Python, an async function can raise anything, and you find out at runtime.

## Parallelism

Python `asyncio.gather` is NeoHaskell `AsyncTask.start` + `AsyncTask.wait`:

```python
async def load():
    users, posts, stats = await asyncio.gather(
        fetch_users(),
        fetch_posts(),
        fetch_stats(),
    )
```

```haskell
load = do
  usersHandle <- AsyncTask.start fetchUsers
  postsHandle <- AsyncTask.start fetchPosts
  statsHandle <- AsyncTask.start fetchStats
  users <- AsyncTask.wait usersHandle
  posts <- AsyncTask.wait postsHandle
  stats <- AsyncTask.wait statsHandle
  Task.yield (users, posts, stats)
```

More lines, but cleaner separation between "start" and "wait" — which helps when you want to interleave other work.

## Error handling

Python's exceptions are untyped. NeoHaskell's errors are typed. This is the single most important difference in day-to-day coding.

```python
try:
    user = fetch_user(user_id)
except UserNotFound:
    return None
except DatabaseError as e:
    log.error(f"DB error: {e}")
    raise
```

```haskell
do
  result <- Task.asResult (fetchUser userId)
  case result of
    Ok user -> Task.yield (Just user)
    Err err -> do
      Log.error [fmt|Failed to fetch user: #{err}|]
      Task.throw err
```

The difference is that in NeoHaskell, `fetchUser`'s signature tells you exactly what errors it can produce. In Python, you have to read the source or the docs.

## SQLAlchemy vs. the event store

In Python, data access usually goes through SQLAlchemy (or a similar ORM):

```python
user = db.session.query(User).filter_by(id=user_id).one_or_none()
if user:
    user.email = new_email
    db.session.commit()
```

In NeoHaskell, you don't touch a database directly. You dispatch a command that produces an event:

```haskell
ChangeEmail { userId = userId, newEmail = newEmail }
-- ... the command executor handles everything
```

The event store records the change. Your projections update automatically. You never wrote an UPDATE statement.

## Django ORM vs. queries

Django ORM's `Meta.ordering` and `Manager` layers give you custom views of your data. NeoHaskell queries do the same thing, but they're kept fresh automatically as events arrive:

```python
# Django
class OrderManager(models.Manager):
    def recent_for_user(self, user_id):
        return self.filter(user_id=user_id).order_by("-created_at")[:20]
```

```haskell
-- NeoHaskell: define a query that maintains a per-user recent orders list
data RecentOrdersForUser = RecentOrdersForUser
  { userId :: Uuid
  , orders :: Array OrderSummary
  }

instance QueryOf Order RecentOrdersForUser where
  queryId order = order.userId
  combine order existing = case existing of
    Just ro ->
      Update ro
        { orders = ro.orders
            |> Array.push (toSummary order)
            |> Array.take 20
        }
    Nothing ->
      Update RecentOrdersForUser
        { userId = order.userId
        , orders = Array.fromLinkedList [toSummary order]
        }
```

The Django ORM version queries the database on every call. The NeoHaskell version maintains a denormalized read model that's updated incrementally — same read cost, no query latency.

## Celery vs. integrations

Celery tasks are Python's answer to background work. NeoHaskell uses integrations:

```python
# Celery
@app.task
def send_welcome_email(user_id):
    user = User.objects.get(id=user_id)
    sendgrid.send(to=user.email, template="welcome")
```

```haskell
-- NeoHaskell outbound integration
instance OutboundIntegration WelcomeEmail where
  reactTo _config user event = case event of
    UserRegistered { email } ->
      Integration.batch
        [ Integration.outbound
            Sendgrid.Email { to = email, template = "welcome" }
        ]
    _ -> Integration.none
```

The integration is a pure function of `event → action`. The runtime dispatches it. You don't worry about queue configuration, retries, or dead letters — the framework handles them.

## Decorators vs. typeclasses

Python decorators are mostly about cross-cutting concerns — auth, logging, validation, caching. NeoHaskell doesn't have decorators, because most decorator use cases are handled differently:

- **Auth decorators** (`@require_login`) → check `ctx.user` in the `decide` function.
- **Validation decorators** (`@validate_email`) → check fields in the `decide` function.
- **Logging decorators** → structured logging via `Log.info` at the relevant points.
- **Caching decorators** (`@lru_cache`) → use `ConcurrentMap` or let projections do the work.

You end up with more explicit code (you see where auth is checked, you see where validation happens), which trades a bit of sugar for a lot of clarity.

## Testing

pytest:

```python
def test_rejects_if_user_does_not_exist():
    result = handle_deposit(DepositCmd(amount=50), user=None)
    assert result.error == "User not found"
```

NeoHaskell:

```haskell
it "rejects if account does not exist" \_ -> do
  let cmd = Deposit { accountId = Uuid.nil, amount = 50 }
  result <- runDecide cmd Nothing
  result |> shouldBe (RejectCommand "Account does not exist")
```

Same shape. No fixtures. No mocks. Pure functions all the way down.

## The learning curve

For a Python developer, expect the ramp to feel like this:

- **Day 1:** Unfamiliar syntax. `<-` and `|>` look weird. Type signatures are confusing.
- **Day 3:** You can read code but writing it still requires the cheat sheet.
- **Week 1:** Writing feels OK. Compiler errors still intimidate you.
- **Week 2:** You start to trust the compiler — errors point at real bugs, and you start fixing them faster than you'd write tests for the equivalent Python code.
- **Week 3:** You realize the whole Django controller + serializer + model + manager stack collapses into one `decide` function, and you stop missing Python.
