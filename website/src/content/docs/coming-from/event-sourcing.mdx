If you've already built event-sourced systems in another language or framework, you're bringing useful instincts. This page maps the concepts you already know to NeoHaskell's vocabulary, and calls out the specific places NeoHaskell does things differently.

## Vocabulary translation

Event sourcing has a few competing vocabularies. Here's how NeoHaskell's terms map to the common ones.

| Standard ES / DDD | Axon | Commanded (Elixir) | EventStoreDB | NeoHaskell |
|---|---|---|---|---|
| Aggregate | Aggregate | Aggregate | Stream | **Entity** |
| Aggregate root | Aggregate root | — | — | **Entity** (same thing) |
| Event sourcing handler | `@EventSourcingHandler` | `apply/2` | — | **`update` function** |
| Command handler | `@CommandHandler` | `execute/2` | — | **`decide` function** |
| Saga | Saga / process manager | Process manager | — | **Integration** (outbound with lifecycle) |
| Projection / read model | `@EventHandler` / projection | Projector | Subscription | **Query** (`QueryOf`) |
| Bounded context | Bounded context | Application | — | **Service** |
| Event store | EventStorageEngine | — | EventStoreDB | **EventStore** (built-in, pluggable) |
| Event bus | Axon SimpleEventBus | — | — | Implicit — the framework handles it |

If you're reading about something in DDD or CQRS literature, translate "aggregate" to "entity" and "read model" to "query" — the rest is close enough to follow.

<Aside type="note" title="Why the terminology change?">
The DDD/CQRS vocabulary is precise but dense. "Aggregate" means multiple things in different books. "Projection" overlaps with database terminology. NeoHaskell picks words that are shorter, clearer to newcomers, and don't carry the academic baggage.
</Aside>

## What NeoHaskell does differently

### Language primitives, not framework annotations

Axon (Java), EventFlow (.NET), and Commanded (Elixir) all use **annotations** or **macros** to mark methods as command handlers, event handlers, and so on. The framework scans for them at startup and wires things up.

NeoHaskell uses **types**. A command is a type that implements the `Command` typeclass. An entity is a type that implements the `Entity` typeclass. The connections between them are expressed with type families:

```haskell
type instance EntityOf Deposit = Account
type instance EventOf Account = AccountEvent
```

The compiler verifies the connections. If you wire something up wrong, you get a compile error, not a runtime exception. This is the biggest structural advantage over annotation-based frameworks.

### Typed consistency boundaries

In most frameworks, you write an aggregate and hope everyone remembers it's a consistency boundary. Mix commands from two aggregates in the same handler? The framework can't tell. Runtime bug.

In NeoHaskell, a service is a type-checked consistency boundary. Two commands in one service must share the same entity and event type. If you try to mix, you get:

```text
Event type mismatch in service definition!

All commands in a service must use the same event type.

Expected event type: CartEvent
But command 'ProcessPayment' uses event type: PaymentEvent
```

The compiler catches it before you run a single test.

### No serialization framework glue

In most ES frameworks, you pick a serialization library (Jackson, Newtonsoft, Jackson again, Poison) and wire it up separately from the event store. You write upcaster classes, migration handlers, and "legacy" event wrappers.

NeoHaskell serializes events automatically via `aeson` (using the `Json.FromJSON` / `Json.ToJSON` instances you derive). Schema evolution is handled by **never modifying old event types** — you add new variants instead. There's no upcasting framework because there's nothing to upcast.

### Transport-agnostic commands

In Axon, a command is dispatched via the Axon command bus. In Commanded, it's dispatched via the Commanded router. In most frameworks, the command machinery is tied to one transport mechanism.

NeoHaskell commands declare their transports with `TransportsOf`:

```haskell
type instance TransportsOf CreateCart = '[WebTransport, Cli, Mcp]
```

The same command is available over HTTP, CLI, and MCP. You choose which transports to register with the application, and the compiler verifies consistency.

### Sagas as regular integrations

Axon has a specific `@Saga` abstraction with its own lifecycle rules. Commanded has process managers. EventFlow has sagas.

NeoHaskell uses **outbound integrations with lifecycle** for saga-like patterns. An integration is a pure function from events to actions. You model the saga's state machine as its own entity, and the integration reacts to its events to drive the next step. This is the pattern in [Tutorial Part 5](/tutorial/05-transfers/).

The result is more code to spell out, but simpler to reason about — a saga is "just" an entity plus an integration, not a special kind of thing.

### Snapshots are automatic

Most frameworks make you opt into snapshotting — you write a snapshot trigger, a snapshot store, and handlers to restore from snapshots.

NeoHaskell does it automatically via the `SnapshotCache`. Entities are cached after reconstruction, and subsequent loads start from the cache. You don't configure it; it's just on.

### Projections rebuild on startup

In EventStoreDB-based systems, projection rebuilds are manual or tooling-driven. In NeoHaskell, on startup the framework replays every event through every registered query, producing fresh read models. This takes care of "I changed the projection logic and need to rebuild" without you writing a migration script.

For large stores, this can take time on startup, but it means read models are always consistent with the current projection code.

### Testing is direct

In most frameworks, testing requires spinning up a test runner, mocking a command bus, fake event stores, and assertion helpers that check sequences of events. It works, but it's heavyweight.

In NeoHaskell, `decide`, `update`, and `combine` are all pure functions. You test them by calling them with constructed inputs and asserting on outputs. No test runner beyond Hspec, no mocks. See [Testing event-sourced systems](/concepts/testing-event-sourced-systems/).

## What NeoHaskell does similarly

- **Commands produce events.** Same as every other ES framework.
- **Events are immutable.** Standard.
- **Entities (aggregates) are reconstructed from events.** Standard.
- **Projections (read models) are denormalized views.** Standard.
- **CQRS separates the write side from the read side.** Standard.
- **Optimistic concurrency per stream.** Same model as EventStoreDB.

If you've done CQRS in Axon, Commanded, or EventFlow, you'll recognize the overall architecture immediately. The concepts are the same; NeoHaskell just pulls them into the type system instead of into runtime annotations.

## Migration from an existing system

If you have an existing system built on Kafka, EventStoreDB, or another framework, and you want to try NeoHaskell:

1. **Start with a new service.** Don't migrate existing data; add a new service that emits and consumes events.
2. **Bridge the two systems.** Write outbound integrations that publish NeoHaskell events to your existing bus (Kafka, etc.). Write inbound integrations that subscribe to relevant topics and emit NeoHaskell commands.
3. **Expand gradually.** Once the new service is stable, consider rewriting the next piece.
4. **Don't migrate all historical events.** Old events stay in the old store. New events go into the NeoHaskell event store. The two streams can coexist indefinitely.

The [Integrating existing systems](/guides/integrating-existing-systems/) guide has the full playbook for this.

## What NeoHaskell still lacks vs. established frameworks

Be honest about this:

- **No first-class schema registry.** You write event types in code; there's no external schema registry with versioning workflows.
- **No distributed deployment primitives out of the box.** Multi-region event routing, cross-datacenter replication, etc., are your responsibility (or via PostgreSQL's native replication).
- **Smaller ecosystem.** Axon has been around for years; NeoHaskell is pre-release. Expect to write more integrations from scratch.
- **Less battle-tested.** Axon runs at banking scale. NeoHaskell has real users but fewer big-name deployments.

If your priority is "proven at scale", you're still better off with Axon or a mature CQRS library in your current language. If your priority is "strongest possible compile-time guarantees and the smallest possible codebase", NeoHaskell is ahead.
